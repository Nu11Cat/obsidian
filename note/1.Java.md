## 特点
### 跨平台性
在JVM以前，程序通过源代码编译成机器码给CPU执行。但是不同平台的机器码不一样，所以同一个程序就要分别进行编译。
跨平台性就是同一套程序逻辑，在不同操作系统/硬件上运行结果一致，且无需重新编写代码。
JAVA的跨平台能力是通过把.java源代码编译成.class字节码文件，再由JVM解释成不同平台的机器码。
这也是我们为什么说java是编译与解释并存的语言（》）
同时，每个操作系统平台都有对应JVM实现，并且都遵循相同的行为准则，包括统一的内存模型、类加载机制、核心类库等。这也是java跨平台性的一大保障。
### 编译与运行并存
编译，是把源代码换成另一种更接近机器执行形式的过程，比如C
解释，是程序在运行时逐行或逐条读取代码并立即执行的过程，比如Py
Java程序先编译成字节码，这一步是编译，在运行时再由JVM对字节码解释执行，或者由JIT编译把热点代码直接编译成机器码执行。（》JIT）
这种设计，一方面通过字节码的中间层设计为Java赋予了跨平台能力，另一方面，解释执行让程序快速启动。同时JIT编译让热点代码运行时更高效。

#### 包装类型
1. 包装类本质上是对象，基本数据类型是原始数据。包装类具备面向对象的特征，继承Object，可以调用方法
2. 存储位置上，包装类对象存在堆内存，同时Java对部分包装类实现了缓存优化（》）；基本数据类型作为方法局部变量就存在栈帧、作为对象成员变量就存在堆内存、静态基本变量元数据存在元空间但是实际数据存在堆。
3. 包装类默认为null，基本数据类型有明确默认值，比如int是0
4. 需要高频计算场景使用基本数据类型，需要使用对象特征就用包装类。
**包装类的作用：**
1. 使得基本数据类型具备OOP特征
2. 允许基本数据类型以对象形式存储在集合
3. 提供null值支持，基本数据类型无法表示”无值“
4. 包装类提供了静态方法，允许进行复杂的数值处理
5. 提供反射机制的支持
6. 缓存机制，提高内存使用率和性能
**自动装箱和拆箱：**
自动装箱是将基本数据类型转换为其对应的包装类的过程，拆箱则是将包装类转化成其基本数据类型的过程。这两个过程是编译器自动完成的。装箱的时候JVM调用包装类的构造方法或valueOf()方法，拆箱的时候通过xxxValue()方法（如 `Integer.intValue()`）获取基本数据类型的值。
但是频繁的装箱可能会导致性能问题，因为在缓存机制范围之外的装箱操作会不断产生新对象，增加GC压力。StringBuffer 是线程安全的，实现方式是 方法前面都加了`synchronized`关键字。StringBuilder并没有增加同步锁或者对它使用的方法新增同步锁，所以是非线程安全的。

### 关键字
#### static
“静态的”，用于声明“属于类级别的成员”，使其不依赖于对象实例，而是属于类本身。
本质是static 成员进入方法区（或元空间）中的类信息结构，跟随类的生命周期。
可以修饰静态变量、静态方法、静态代码块、静态内部类等。
在类中，static方法可以直接通过类名调用，不能被重写，不参与多态。
在接口（JDK8+）中，允许定义static方法，可以通过接口名调用。
#### final
用于表示“不可变”，限制修改。
可以修饰变量，方法，类。
如果修饰变量，对于基本数据类型就表示值不能变了；对于引用数据类型就表示这个引用不能再指向别的对象，但是对象本身的内容可以修改。
如果修饰方法，那么它就不能被子类重写。比如lang.Object类的getClass方法。
如果修饰类，则类不能被继承，如字符串类String类。
扩展：final、finally、finalize 的区别？（》）
#### this
作用有三个
1. 引用当前对象，常见于实例方法或者构造器。
2. 调用当前类的其他构造器，这种情况只能放在构造器的第一行。
3. 把当前对象作为参数传递。
#### transient
简单来说就是在序列化（》序列化）的时候，被修饰的变量的序列化过程会被组织，在反序列化的时候这个值就不会被持久化或者恢复。
比如，transient a=1，b=2；在序列化的时候a的序列化被跳过，在反序列化后，b = 2，a = 默认值。如果是int就是0
1. 只能修饰变量，不能修饰类/方法
2. 被static修饰的变量不属于任何对象，所以不管有没有transient修饰都不会被序列化
### 其他
#### &和&&有什么区别
单个&：对于整数，按位与，用于计算；对于Boolean，逻辑与，即使左边是 false，右边仍然会执行。
两个&：逻辑与，专用于Boolean，如果左边为 false，右边就不会再执行，也就是短路，更安全。
#### == vs equals
双=是一个运算符，对于基本数据类型是对比值是否相等，对于引用数据类型是对比是否指向同一个对象，也就是在堆内存的内存地址是否相同
equals是Obejct类提供的方法，默认实现是`return this == obj`，也就是比较引用的地址。但是很多标准类都重写了equals方法，比如String类。
#### 深拷贝、浅拷贝、引用拷贝
深拷贝，递归地复制，创建出完整独立的对象树，两个对象互不影响。
浅拷贝，创建一个独立的外层对象，但是如果对象内部有引用数据类型，那么它们的指向是相同的，会互相影响。
引用拷贝，没有创建对象，只是把对象的引用地址进行了复制。对其中一个的修改都会影响另一个。
#### 语法糖
语法糖是为了方便开发人员，在语法层面的一种简化写法，让代码更简洁易读。但是在编译阶段就会被还原成基础语法结构。
常见的语法糖有：泛型、自动装箱拆箱、可变长参数、枚举、内部类、增强for循环、lambda 表达式等。
1. 泛型（》泛型）
2. 自动装拆箱（》）
3. 可变长参数，方法传任意多个参数，编译器会把它转换成一个数组参数来处理。
4. 枚举，编译器会将 `enum` 类转化为一个普通的类，并生成所有的常量、构造方法以及辅助方法（`values()`、`valueOf()` 等），这让我们可以方便地使用枚举而不需要手动编写大量的常量和方法。
5. 内部类，内部类、局部内部类、匿名内部类 都是语法糖。编译器会把它们拆成独立的类文件，并加上必要的外部类引用或构造函数参数。
6. 增强for循环，转换成了普通的Iterator迭代器或者下标访问。
7. lambda表达式，用来简化函数式接口的写法。底层要么编译成匿名内部类，要么在运行时动态生成函数对象，本质仍然是函数式接口的实现类实例。
#### 移位运算符
1. 高效，现代处理器有专门的硬件指令来执行移位操作。
2. 节省内存，移位操作可以使用一个整数存储多个布尔值/标志位。
常用于：
1. 乘除2的幂次方的运算。
2. 存储和操作多个Boolean值。
3. 哈希算法/加解密
补充：如果移位位数超过了数值所占有的位数会在实际执行的时候取余。

### 序列化
序列化就是把Java对象转化成字节流的过程，方便把对象保存到磁盘、通过网络传输、存储到内存、存储到数据库。反序列化就是把字节流恢复成原来的Java对象。
Java序列化方式：
1. Java原生序列化：只要一个类实现了`Serializable`接口就能通过 `ObjectOutputStream` 和`ObjectInputStream` 反序列化。优点是使用简单，缺点是性能差，生成的字节大，并且不能跨语言，同时也不安全。
2. 基于文本格式的序列化方式，如JSON或XML。比如FastJson和Gson等，优点是可读性强，支持跨语言，但是性能一般。
3. 高性能二进制序列化框架，如Hessian、Protobuf等。高性能，支持跨语言，体积小。
### 值传递/引用传递
Java是值传递。
方法调用的时候，传递的变量种存储的值的副本
如果是基本数据类型，传递的是值的拷贝。
如果是引用数据类型，传递的是引用地址的拷贝，可以通过这个地址修改对象的内容，但是不能改变引用指向。
## JVM
### 对象
#### 创建对象的方式
1. new关键字，通过类的构造方法
2. 反射机制（》）
3. 使用clone()方法，前提是类实现了Cloneable接口
4. 反序列化，前提是类实现了Serializable接口，常见于网络传输和持久化场景。
#### 创建对象的过程
1. 当JVM遇到new指令的时候，先去运行时常量池解析类的符号引用，检查这个类是否已经被加载、解析、初始化过。如果没有，则先执行类加载过程。
2. 类加载检查后JVM会为新对象分配内存（这里分配的内存大小在类加载完成后即可确定）。分配内存的方式有两种：指针碰撞和空闲列表。（》）（》内存分配并发问题）
3. 内存分配后，JVM需要把分配到的内存空间初始化为零值。这一步是为了保证对象的实例字段在代码中不被赋初始值就能直接使用。
4. 初始化零值完成后，JVM会设置对象头，比如对象属于哪个类、怎么找到类的元数据信息、GC分代年龄信息等
5. 到此，从JVM的视角上来看，对象已经创建完成。但是从Java程序的角度，对象创建刚开始，所以会紧接着执行init方法，把对象按照程序的意愿初始化。
#### 对象的内存布局
对象在内存的布局分为三个部分：
1. 对象头，也是对象创建过程JVM设置对象头的地方，这里包括两部分信息：
- 标记字段 ，用于存储自身运行时数据，如哈希码、GC分代年龄、锁状态标志等。
- 类型指针，对象指向它的类元数据的指针，JVM就是通过它判断这个对象是哪个类的实例的
2. 实际数据，是对象真正存储的有效信息，也就是在程序中定义的各种类型的字段内容。
3. 对齐填充部分，仅起占位作用。因为Hotspot虚拟机的自动内存管理系统要求对象起始地址必须是8字节的整数倍，也就是我们需要通过对齐填充部分去凑对象的大小是8字节的整数倍。又因为对象头一般是8字节的倍数，所以主要是为了补齐实际数据部分。（8字节：现代CPU以“字”为单位访问内存，通常64位系统一个字就是8字节，对齐8字节能提供CPU访问内存的效率）
#### 对象的引用类型
1.2之前，只有一种引用，就是reference类型的数据存储的数值代表另一块内存的起始地址，我们就称这块内存代表一个引用。
1.2之后，分为四种引用，引用强度逐渐减弱：
1. 强引用，是最常见的引用类型，就是java中的普通对象，只要强引用在，就永远不会被GC回收其对象。
2. 软引用，当如果内存充足，JVM不会回收对象；但当JVM判断内存不足的时候就会尽可能回收这些对象。适用于缓存场景。
3. 弱引用，一旦发生GC，不管内存是否充足，都会被回收。比如ThreadLocal的key等规范引用关系场景。
4. 虚引用，无法通过get()获取对象引用。常配合引用队列ReferenceQueue追踪对象回收的时机，提供了一种回收后的通知机制。常用于底层资源释放或监控。（常和finalize一起讨论，但是后者是GC前通知》）
#### 对象的生命周期
创建、使用、销毁三个阶段
1. 创建：（》对象的创建过程）
2. 使用：对象被引用并执行相应的操作，可以通过引用访问对象的属性和方法，在程序运行过程被使用。
3. 销毁：当对象不再被引用，被JVM GC。
#### JVM访问对象的方式
对象的访问方式由虚拟机实现而定，主要的方式有：使用句柄、直接指针
1. 句柄访问：JVM为每个对象分配一个句柄池区域。对象的访问是“引用 → 句柄 → 对象”。类似于加了一个中间层，好处就是当对象在内存中移动的时候，只需要更新句柄中的地址，不需要修改所有引用，适合频繁移动对象的GC策略。
2. 直接指针：是HotSpot虚拟机默认采用的方式。这种方式下，引用中直接保存的是对象在堆内的地址。访问速度快，但是对象一旦发生移动就需要更新所有指向它的引用。
#### 对象在JVM的位置
大多数对象都在堆上，但是也有例外。
1. 栈，如果对象只在方法内部使用，不会被外部引用，也就是不会“逃逸”。那么JVM会通过“逃逸分析”把对象分配到栈上。这里的对象随方法调用结束自动释放，无需GC，性能更高。
2. 直接内存/本地内存，通过 `ByteBuffer.allocateDirect()` 分配的对象数据在堆外内存，不受堆 GC 管理。常用于高性能 I/O 场景。
3. 方法区/元空间中的类元数据，存放类对象本身，在类加载时创建。
### 类
#### 类的生命周期
加载、验证、准备、解析、初始化、使用、卸载
#### 类加载过程
类加载过程是指JVM将.class文件加载到内存，并完成必要准备供程序使用的完整流程，就是类生命周期的前五个阶段。
1. 加载，JVM通过类加载器（》类加载器）将.class文件读入内存，转化为Class对象，存储在方法区。此时创建类对象。双亲委派模型也是这个阶段生效（》双亲委派模型）。
2. 验证，保证加载的字节码是符合JVM规范的，是安全可执行的。其中包括四种：文件格式验证、元数据验证、字节码验证、符号引用验证。
3. 准备，JVM为类变量分配内存并设置默认初始值。只设置默认初始值，不赋值，赋值在初始化。
4. 解析，JVM将常量池的符号引用解析为直接引用。
5. 初始化，执行初始化逻辑。执行static变量赋值和static代码块。
#### 类卸载过程
类卸载是指某个类对应的Class对象及其相关元数据从方法区移除。
类卸载需要满足三个要求：
1. 该类的所有示例对象都已经被GC
2. 该类没有在被引用
3. 该类的类加载器示例已被GC

## Hard
#### 如何将对象从一个JVM转移到另一个JVM
对象跨JVM转移的核心方法
1. **对象序列化与反序列化**这是最常用的跨JVM对象传输方案。通过实现Serializable接口，可以将对象转换为字节序列，然后通过任意传输机制（如网络、文件）发送到另一个JVM后重建。Java原生序列化虽然简单，但存在性能较低、安全性等问题。实际开发中更推荐使用JSON、Protocol Buffers等跨语言序列化方案。
2. **RPC框架调用**在分布式系统中，通过Dubbo、gRPC等RPC框架可以实现透明的对象传输。框架内部会处理序列化和网络通信细节，开发者只需像调用本地方法一样使用远程对象。这种方式适合生产环境，但需要引入额外依赖。
3. **分布式缓存/消息队列**使用Redis、Memcached等分布式缓存作为中转，或通过Kafka、RabbitMQ等消息队列传递序列化后的对象。这种方案具有松耦合特性，适合异步场景。
4. **Java远程方法调用(RMI)** Java原生提供的RMI技术允许直接跨JVM调用方法，底层自动处理对象序列化和网络传输。但由于其强耦合性和防火墙穿透问题，现代项目中已较少使用。

### final、finally、finalize 的区别？
1. final（》）
2. finally是Java中异常处理机制的一部分，用来创建try块后面的finally块。无论try块的代码是否抛出异常，finally块的代码总是执行。常用来释放资源，如关闭文件、数据库连接等。
3. finalize是Object提供的一种方法。最初用来做GC前的清理工作，但是因为不可控和性能问题被弃用。
### 指针碰撞和空闲列表
**内存分配的两种方式**：

1. 指针碰撞： 

- 适用场合：堆内存规整（即没有内存碎片）的情况下。
- 原理：用过的内存全部整合到一边，没有用过的内存放在另一边，中间有一个分界指针，只需要向着没用过的内存方向将该指针移动对象内存大小位置即可。
- 使用该分配方式的 GC 收集器：Serial, ParNew

2. 空闲列表： 

- 适用场合：堆内存不规整的情况下。
- 原理：虚拟机会维护一个列表，该列表中会记录哪些内存块是可用的，在分配的时候，找一块儿足够大的内存块儿来划分给对象实例，最后更新列表记录。
- 使用该分配方式的 GC 收集器：CMS

选择以上两种方式中的哪一种，取决于 Java 堆内存是否规整。而 Java 堆内存是否规整，取决于 GC 收集器的算法是"标记-清除"，还是"标记-整理"（也称作"标记-压缩"），值得注意的是，复制算法内存也是规整的。
### 内存分配并发问题
在创建对象的时候有一个很重要的问题，就是线程安全，因为在实际开发过程中，创建对象是很频繁的事情，作为虚拟机来说，必须要保证线程是安全的，通常来讲，虚拟机采用两种方式来保证线程安全：

- **CAS+失败重试：** CAS 是乐观锁的一种实现方式。所谓乐观锁就是，每次不加锁而是假设没有冲突而去完成某项操作，如果因为冲突失败就重试，直到成功为止。**虚拟机采用 CAS 配上失败重试的方式保证更新操作的原子性。**
- **TLAB：** 为每一个线程预先在 Eden 区分配一块儿内存，JVM 在给线程中的对象分配内存时，首先在 TLAB 分配，当对象大于 TLAB 中的剩余内存或 TLAB 的内存已用尽时，再采用上述的 CAS 进行内存分配
### 类加载器
主要有以下几种：
1. 启动类加载器，加载JDK核心类库，类路径为JVM启动的时候指定的目录下的类。
2. 扩展类加载器，加载JDK扩展目录下的类。
3. 应用类加载器，加载编写的代码，即类路径的类。是最常见的类加载器。
4. 自定义类加载器，开发者提供继承ClassLoader自定义逻辑，用于热部署、模块隔离、插件化等场景。
### Tomcat的类加载器
Tomcat 的类加载机制通过多级类加载器、可配置的委派策略和每个 Web 应用独立加载器，实现类隔离、共享和热部署，是普通 JVM 类加载的增强版。 
### 双亲委派模型
双亲委派模型是Java类加载机制的核心设计之一，规定：当类加载器在场上加载一个类的时候，会先把这个请求交给自己的“父加载器”处理；只有父加载器找不到的时候才会由自己加载。从而形成一种自顶向下的逐级委托关系，防止类的重复加载，保障核心类库的安全与一致性。
#### 打破双亲委派模型的场景
一些场景需要我们通过设计打破双亲委派模型：
1. SPI机制，最典型。比如JDBC，我不想用JDK的驱动类，而是应用自己的驱动类。我们就可以通过线程上下文类加载器，这种自定义类加载器。
2. Web容器，比如Tomcat。在面对多个Web应用可能都有User.class，就会冲突。所以要求优先子类加载器加载。
3. 插件需求，多个插件有同名类，打破类的唯一性。
4. 热部署，通过重建类加载器，用类加载器生命周期替换类生命周期，解决类只能被加载一次的问题。
#### 加载过程
以应用类加载器为例
1. 应用类加载器接到请求，先将请求交给 **扩展类加载器**；
2. 扩展类加载器再将请求交给 **启动类加载器**；
3. 启动类加载器尝试加载，比如加载 `java.lang.String`；如果有，则加载结束；如果没有，则由下层加载器尝试加载。
4. 如果父加载器都无法加载，则由当前类加载器尝试自己加载类字节码。
### 包装类的缓存机制
JVM对常见的重复使用对象进行复用，避免每次都创建新的对象，提示性能，减少内存使用。
比如Integer，JVM会缓存-128到127的整数。
Boolean，JVM只会缓存true和false两个值。
```java
//包装类的缓存机制
Integer i1 = 100; // 会从缓存池获取
Integer i2 = 100; // 同样会从缓存池获取，i1 和 i2 指向同一个对象
Integer i3 = 200; // 不会从缓存池获取，而是新建对象
Integer i4 = 200; // i3 和 i4 不是同一个对象
```
