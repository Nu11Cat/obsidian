很多 AI 应用上线后都会遇到同一类问题：回答读起来顺畅，但关键事实存在错误。这个问题会直接影响可用性和信任度，常见于客服问答、技术支持、文档总结、代码生成等场景。

这个现象称为 **AI 幻觉（AI Hallucination）**。本文给出清晰定义、常见类型、识别方法和治理策略，重点放在可落地的工程做法。



<!-- more -->



## **AI 幻觉简介**

### **什么是 AI 幻觉**

AI 幻觉指模型输出了形式完整的内容，但事实错误、来源缺失或无法验证。大语言模型（Large Language Model，LLM）按概率预测下一个 token（词元），输出是否连贯和“是否真实”并不等价。


### **高频触发原因**

幻觉常见于上下文不足、问题边界模糊、知识过期、检索失败等条件。模型在不确定状态下仍会继续生成文本，因此会出现事实偏差或来源缺失。

这类问题有明确工程特征：输入约束越弱、检索质量越低、结果校验越少，幻觉率通常越高。

------

## **幻觉有哪些常见类型**

### **按结果表现分类**

| **类型** | **表现** | **典型例子** |
| -------- | -------- | ------------ |
| **事实型幻觉** | 编造客观事实 | 虚构人物经历、错误年份、错误法规 |
| **引用型幻觉** | 编造来源或链接 | 不存在的论文、假 DOI、失效文献 |
| **代码型幻觉** | 伪造 API 或参数 | 推荐不存在的方法名、错误库版本用法 |
| **推理型幻觉** | 推理链条看似合理但结论错误 | 数学步骤顺畅但中间关键条件遗漏 |

### **按触发来源分类**

- **知识盲区**：训练语料里缺失某领域细节
- **时效问题**：模型知识截止导致“过期正确”
- **提示不充分**：问题边界模糊，模型只能“补全猜测”
- **检索链路失效**：RAG 未取到准确信息，模型仍继续作答
- **上下文污染**：历史对话中混入错误前提，被模型继承

-----------

## **为什么难以完全消除**

### **生成式模型的机制决定了风险下限**

当前 LLM 属于生成式系统，核心能力是语言建模与模式归纳，不负责外部事实的自动校验。模型可以给出结构完整的回答，但不会自动执行来源检索、证据匹配和事实一致性检查。

工程实践应以风险控制为目标，重点在于持续降低幻觉率并提高错误可发现性：

1. 降低高风险场景中的幻觉率
2. 让错误更容易被发现和拦截
3. 在不确定时优先返回“可验证状态”

### **工程系统会放大或缩小幻觉**

同一个模型在不同系统里的表现差异很大。输入模板、检索质量、工具权限、输出约束、审核策略都会影响最终幻觉率。

因此“模型能力评估”和“系统可靠性评估”需要分开进行。

-----------

## **如何判断一次回答是否可能有幻觉**

可以用一个轻量检查框架：来源是否可追溯、过程是否可复现、结论是否可交叉验证。

当回答出现以下信号时，应提高警惕：

- 细节非常具体，但不给来源
- 引用格式完整，却无法检索到原文
- 结论极其肯定，不说明适用边界
- 代码示例可读性高，但实际运行报错
- 多轮追问后前后说法不一致

-----------

## **降低幻觉的实用策略**

### **提示词层（Prompt）**

1. 明确任务边界：说明领域、时间范围、输出格式
2. 要求“不确定时显式说明不知道”
3. 要求结论附来源或证据字段
4. 对高风险任务加“先验证后回答”的步骤约束

示例指令：

```text
请基于可验证资料回答；若无法确认，请明确标注“信息不足”，不要猜测。
```

### **系统层（RAG / Tool Use）**

1. 接入检索增强生成（Retrieval-Augmented Generation，RAG）
2. 强制关键结论绑定证据片段
3. 对外部工具调用结果做结构化校验
4. 对高风险场景加入规则引擎或人工复核

最小可用输出格式示例：

```json
{
  "answer": "......",
  "confidence": "low|medium|high",
  "evidence": ["source_1", "source_2"],
  "uncertainty_note": "若有不确定项在此说明"
}
```

### **业务层（按风险分级治理）**

- **低风险**（内容润色、标题建议）：允许自动直出
- **中风险**（技术说明、方案建议）：增加自动校验
- **高风险**（医疗、金融、法律、合规）：必须人审

-----------

## **常见误区**

### **误区 1：把幻觉当作“模型太笨”**

幻觉在不同规模模型上都存在。实际差异通常体现在发生频率、错误形态和修复成本。

### **误区 2：以为 RAG 能一劳永逸**

RAG 只能降低“知识缺失型”幻觉，无法自动解决推理错误、检索召回偏差和上下文污染。

### **误区 3：只盯模型，不看流程**

提示词、检索、校验、审核链路存在缺口时，系统会持续产出不可靠结果。

-----------

## **一个可落地的最小实践**

如果你正在做 AI 问答或 AI 助手，可以先从这 4 步开始：

1. 给输出加 `confidence` 与 `evidence` 字段
2. 对“无证据结论”设置自动降级策略
3. 为高风险问题增加人工确认开关
4. 建立幻觉样本库，按周复盘并更新提示词/检索规则

这样做可以把错误纳入可观测、可治理、可持续优化的流程，并持续降低实际业务风险。

-----------

## **总结**

**AI 幻觉是生成式模型在不确定条件下产生的事实偏差问题。**

工程落地应优先建设提示约束、检索增强、证据绑定和风险分级机制，把幻觉率控制在业务可接受范围内。
