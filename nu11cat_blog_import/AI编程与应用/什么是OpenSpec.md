在 AI 参与开发后，团队写代码的速度明显提升，但需求理解偏差、接口语义不一致、验收口径不统一这几类问题并没有消失。很多返工并不来自实现能力不足，而是来自“大家以为自己理解的是同一件事”。

OpenSpec（Open Specification，开放规范）可以理解为一套以规范为中心的协作方式：先把需求、边界、验收标准和接口契约写成可评审的规格，再进入实现与联调。它的价值不在于替代代码，而在于给代码提供可追溯的交付基线。

## OpenSpec 是什么

### 定义

OpenSpec（Open Specification，开放规范）指的是一种公开、可版本化、可评审的规格管理方式。这里的“开放”强调两个点：

1. 规格文本对相关角色可见（产品、研发、测试、运维都能查看与评论）
2. 规格变更有版本记录，可追踪“为什么改、改了什么、影响哪些模块”

它通常以 Markdown、YAML、JSON Schema、OpenAPI 等可读且可管理的格式存在，放在代码仓库或与代码仓库联动的文档系统中。

------

### 为什么在 AI 开发阶段更重要

AI 可以加速编码与重构，但模型生成结果依赖输入上下文。若输入只包含模糊需求，输出很容易出现“实现完整但方向偏离”的情况。OpenSpec 的作用是先固定上下文，再放大执行效率。

常见收益包括：

- 减少需求解释分歧导致的返工
- 降低跨端、跨服务联调的语义偏差
- 让测试用例和验收标准有统一依据
- 让 AI 助手获得更稳定的任务输入

-----------

## OpenSpec 的核心组成

一个可落地的 OpenSpec 通常包含以下部分：

| 组成 | 作用 | 交付建议 |
|------|------|----------|
| 问题与目标 | 说明要解决什么业务问题 | 用可验证业务指标描述 |
| 范围与非范围 | 约束本期做什么、不做什么 | 明确排除项，防止范围膨胀 |
| 术语与领域模型 | 统一关键概念定义 | 首次出现给出中文+英文 |
| 接口与数据契约 | 固定输入输出与错误语义 | 优先机器可校验格式 |
| 验收标准 | 定义完成条件 | 与测试用例一一对应 |
| 变更记录 | 记录版本演进 | 标注影响面与迁移策略 |

这套结构的目标是把“口头共识”变成“可审计共识”。

-----------

## 从需求到上线的典型流程

### 1）草拟规格

产品与研发先形成最小规格草案，至少覆盖目标、范围、接口、验收四部分。草案阶段关注可执行性，不追求修辞完整。

### 2）跨角色评审

研发关注实现复杂度与边界，测试关注可测性与异常路径，运维关注发布影响与回滚条件。评审结论直接回写到规格文档，而不是只存在会议纪要。

### 3）实现与测试对齐

开发以规格中的契约为实现基线，测试以规格中的验收标准设计用例。若实现阶段发现规格缺口，先补规格再改代码，保持一致的变更顺序。

### 4）发布与回溯

上线后把实际行为与规格比对，补充偏差原因与后续修订计划。这样下一次迭代可以直接复用已有上下文。

## 一个最小可用的 OpenSpec 模板

下面是团队可以直接落库的最小模板：

```markdown
# Feature: 用户资料更新

## 目标
- 用户可以修改昵称与头像，提交后 2 秒内在个人主页可见。

## 范围
- 本期包含：昵称、头像更新接口与页面交互。
- 本期不包含：个性签名、背景图编辑。

## 术语
- 用户资料（User Profile）：展示给其他用户的公开信息集合。

## 接口契约
- PATCH /api/profile
- Request:
  - nickname: string, 1-20 字符
  - avatarUrl: string, URL
- Response:
  - 200: { id, nickname, avatarUrl, updatedAt }
  - 400: 参数校验失败
  - 409: 昵称冲突

## 验收标准
1. 合法参数返回 200，页面展示更新结果。
2. 昵称超长返回 400，并提示可修复信息。
3. 并发更新场景下，最终状态与最后一次成功写入一致。
```

如果团队已使用 OpenAPI 或 Schema Registry，可把接口契约部分拆成独立文件，在规格中仅保留引用路径与版本号。

## OpenSpec 与常见文档的关系

| 文档类型 | 关注点 | 与 OpenSpec 的关系 |
|----------|--------|--------------------|
| PRD | 业务价值、用户目标 | OpenSpec 细化为工程可执行约束 |
| API 文档 | 接口定义 | OpenSpec 负责把接口放入业务场景与验收闭环 |
| 测试用例 | 验证路径 | OpenSpec 提供用例来源与覆盖边界 |
| 变更记录 | 历史演进 | OpenSpec 记录“变更原因 + 技术影响” |

因此 OpenSpec 不是替代这些文档，而是把它们在交付链路中串成同一条主线。

-----------

## 在 AI 工作流中的实践建议

### 给 AI 明确输入边界

把规格中的目标、范围、契约、验收标准作为提示词核心输入，减少模型凭经验补全导致的偏差。

### 把“改代码”与“改规格”绑定

当接口行为或业务规则发生变更时，同一提交中同时更新规格与实现，避免后续读者依据过期规格做决策。

### 在 CI 做轻量校验

可在持续集成中增加两类检查：

1. 关键接口变更时是否同步更新规格文件
2. 规格中的接口示例是否通过基础语法校验

这样可以把流程约束前置到提交阶段。

## 常见误区

### 误区 1：规格写得越长越好

规格长度不是目标。可执行、可评审、可追踪比篇幅更重要。超长但缺少约束的文档，维护成本会持续上升。

### 误区 2：规格只给产品看

若测试、后端、前端、运维不参与评审，规格无法成为统一基线，后续依然会回到多套口径并行的状态。

### 误区 3：规格冻结后不能改

真实项目中需求会变化。关键在于变更是否有记录、是否说明影响范围、是否同步到实现与测试。

-----------

## 总结

OpenSpec 的本质是把需求、契约与验收放进同一条可版本化的交付链路，让团队在“同一上下文”里协作。  
落地时先从一个高频功能做最小模板试点，再把评审与 CI 校验纳入常规流程，团队会更快看到返工率下降。
