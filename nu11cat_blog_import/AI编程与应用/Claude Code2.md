Claude Code 是一个强大的命令行 AI 编程工具,但用得好不好,差距巨大。本文整理 Anthropic 官方推荐的最佳实践,从**配置定制、工作流、效率技巧到高级用法**四个维度,帮你把 Claude Code 从"能用"调到"好用"。

<!-- more -->

## **配置与定制**

### **CLAUDE.md：一切的起点**

Claude Code 每次进入项目时会**自动读取 `CLAUDE.md`**,它相当于一份"写给 AI 的项目说明书"。写好它,是提升使用效果**性价比最高**的一件事。

**应该写进 CLAUDE.md 的内容：**

- **常用命令**：构建、测试、lint 的具体命令(例如 `npm run build`、`pytest tests/`)
- **代码风格约定**：缩进、命名、import 习惯
- **项目特有的"坑"**：那些新人容易踩、文档里又没写的注意事项
- **测试与分支策略**：测试怎么跑、分支怎么命名、提交规范

**编写技巧：**

- 用 `/init` 命令自动生成初版,再手工精简
- 保持**简洁**。CLAUDE.md 会占用上下文窗口,内容越啰嗦,重点越被稀释。把它当成"高频提示词",而不是文档
- 会话中用 `#` 键开头输入一条规则,Claude 会直接把它追加进 CLAUDE.md,无需手动编辑
- **持续迭代**：发现 Claude 反复犯同一个错误,就把纠正写成一条规则放进去。CLAUDE.md 是"越用越准"的

**多层级 CLAUDE.md：**

| 文件 | 作用 | 是否提交 git |
| --- | --- | --- |
| `~/.claude/CLAUDE.md` | 全局个人偏好,所有项目生效 | 否 |
| `CLAUDE.md`(项目根) | 团队共享指令 | 是 |
| `CLAUDE.local.md` | 个人项目级覆盖 | 否(git 忽略) |
| 子目录 `CLAUDE.md` | 仅该目录相关的指令 | 视情况 |

> 用一句话强调或调整某条规则时,可以用 **IMPORTANT**、**YOU MUST** 这类措辞——实践表明,适当强调能显著提高 Claude 的遵守率。

### **工具与权限管理**

Claude Code 默认对每个有副作用的操作(执行命令、写文件)都会**弹出确认**。这是安全设计,但频繁打断会拖慢节奏。全部放开又有风险。关键在于**精细化授权**:把安全的操作放行,把危险的操作留住。

**三种管理手段：**

- **会话中即时授权**：弹出权限请求时选 `Always allow`,该类操作以后不再询问
- **`/permissions` 命令**：随时查看、编辑当前的允许 / 禁止清单
- **`settings.json` 配置**：把规则写进配置文件,团队共享、版本管理

**`--dangerously-skip-permissions`(YOLO 模式)**：完全跳过所有确认。**仅建议**在 Docker 容器、CI 等隔离/无网络环境中使用,绝不要在本机直连生产环境时开启。

**推荐策略**:安全的只读和高频操作放进 `allow`,危险的破坏性操作放进 `deny` 或保持需确认。

下面是一份开发者可直接参考的 `.claude/settings.json` 示例：

```json
{
  "permissions": {
    "allow": [
      "Bash(git status)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(git add:*)",
      "Bash(git commit:*)",
      "Bash(ls:*)",
      "Bash(cat:*)",
      "Bash(npm run test:*)",
      "Bash(npm run lint:*)",
      "Bash(npm run build:*)",
      "Read(src/**)",
      "Edit(src/**)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push --force:*)",
      "Bash(git reset --hard:*)",
      "Read(.env)",
      "Read(.env.*)",
      "Read(**/secrets/**)"
    ]
  }
}
```

**这份配置的设计思路：**

- **`allow` 放行的都是"安全且高频"**：查看状态、读代码、跑测试和 lint、构建——这些做一百遍也不会出事,放行后体验顺滑很多
- **`deny` 拦截"不可逆或敏感"**:`rm -rf`、强制推送、硬重置这类一旦执行难以挽回的命令;以及 `.env`、密钥目录等不该被读进上下文的敏感文件
- **`commit` 放行但 `push --force` 拦截**:让 Claude 能自由提交到本地,但推送(尤其强制推送)这种"对外可见"的动作仍需人把关
- 冒号后的 `*` 是参数通配符,例如 `Bash(git diff:*)` 表示允许 `git diff` 带任意参数

> 团队共享的规则放 `.claude/settings.json`(提交 git);个人临时放宽 / 收紧的规则放 `.claude/settings.local.json`(git 忽略),不会影响其他成员。

### **自定义斜杠命令**

把**重复性任务**存成 `.claude/commands/` 目录下的 `.md` 文件,文件名会自动变成一条 `/命令名`。这是团队把"标准化流程"沉淀下来的核心机制——写一次,人人复用。

例如新建 `.claude/commands/fix-issue.md`：

```markdown
请修复 GitHub issue #$ARGUMENTS。步骤：

1. 用 `gh issue view $ARGUMENTS` 查看 issue 详情
2. 定位相关代码文件
3. 实现修复
4. 编写并运行测试,确保通过
5. 用清晰的提交信息提交
```

之后在会话里输入 `/fix-issue 1234`,`$ARGUMENTS` 就会被替换成 `1234`。

**实用技巧：**

- `$ARGUMENTS` 是参数占位符,把命令变成可复用的"模板"
- 命令文件里可以直接写要执行的 Bash 命令、要读的文件、要遵守的步骤
- 适合沉淀的场景:代码审查、修 issue、生成变更日志、发版流程、性能分析等
- 命令支持目录分组,例如 `.claude/commands/git/commit.md` → `/git:commit`

### **MCP：给 Claude 装"外设"**

MCP(Model Context Protocol,模型上下文协议)让 Claude 能连接**外部工具和数据源**:数据库、GitHub、浏览器、文档系统、监控平台等。配置后,Claude 不再只能读写本地文件,而是能直接查数据库、读 issue、驱动浏览器截图。

**常见用法：**

- **GitHub MCP**:直接读写 issue、PR、review
- **数据库 MCP**:让 Claude 查询表结构、执行只读 SQL 来理解数据
- **Playwright / Puppeteer MCP**:控制浏览器,做端到端测试、截图反馈
- **文档 MCP**:接入 Notion、Confluence 等知识库

**配置方式**:用 `claude mcp add` 命令添加,或写进配置文件。MCP server 可配置在三个层级——项目级(团队共享)、用户级(个人全局)、本地级。

> 提示:MCP 工具越多,占用的上下文越多。**只接入当前任务真正需要的 server**,用不到的及时移除,保持上下文清爽。

### **rules 与 skills**

除了 `commands`,`.claude/` 下还有两类常被忽略但很有用的定制:

- **`.claude/rules/`**:把 CLAUDE.md 里的规则**拆成模块化文件**(如 `code-style.md`、`testing.md`、`api-conventions.md`)。它和 CLAUDE.md 本质相同——**被动、始终生效**,Claude 整个会话都遵守。好处是当约定越来越多时,避免 CLAUDE.md 变成一坨臃肿的长文。
- **`.claude/skills/`**:更高级的**复合工作流**。每个 skill 是一个含 `SKILL.md` 的子目录,可附带 `scripts/`(脚本)、`references/`(参考资料)、`assets/`(模板)。它的特点是**自动、按语义触发**——Claude 判断当前任务命中某个 skill 的 `description` 时会自动加载,无需你手动调用。

### **怎么选:rules / command / skill 一张表搞定**

这是最容易混淆的地方。核心区别在于**触发方式**和**生效时机**:

| 机制 | 触发方式 | 生效时机 | 适合什么 |
| --- | --- | --- | --- |
| **CLAUDE.md / rules** | 被动,自动加载 | **始终生效** | 代码风格、项目约定、长期不变的规则 |
| **command(斜杠命令)** | 主动,手动打 `/命令` | 你调用时 | 你**明确知道要做**的标准化流程(修 issue、发版、审查) |
| **skill** | 半自动,语义命中触发 | Claude 自己判断 | 希望 AI **不用提醒就自动执行**的复杂流程,常带脚本/模板 |

**一句话决策:**

- 想让 Claude **永远记住的约束** → 写进 **CLAUDE.md / rules**
- 想要一个**自己手动触发的流程模板** → 做成 **command**
- 想让 Claude **在合适时机自动识别并执行**的复杂能力 → 做成 **skill**

> 经验法则:规则用 rules,动作用 command,智能用 skill。三者不是竞争关系,成熟项目往往同时用——rules 兜底约束,command 跑高频流程,skill 处理需要判断的复合任务。

## **工作流实践**

工具配置好之后,真正拉开差距的是**怎么用**。下面是几个经过验证的高效工作流。

### **探索 → 计划 → 编码 → 提交**

这是最通用的四步法。新手最常见的错误是"一上来就让 Claude 写代码",结果方向跑偏、返工不断。正确的节奏是:

1. **探索(Explore)**:先让 Claude **读相关文件理解现状**,但明确告诉它"先别写代码"。例如:"阅读处理登录的相关文件,先不要写任何代码,告诉我现在的认证流程是怎样的。"
2. **计划(Plan)**:让它**给出实现方案**。复杂任务可以让它"想得深入一点"(ultrathink),并把方案写下来供你审阅。
3. **编码(Code)**:方案确认后再动手实现,过程中让它**随时验证**(跑测试、自查)。
4. **提交(Commit)**:实现完成后提交代码,并让它写清晰的提交信息、必要时更新文档。

> 关键在于**前两步**。让 Claude 先理解、先规划,再写代码,质量会有质的提升。跳过探索和计划,等于让它"盲写"。

### **测试驱动开发(TDD)**

AI 在**有明确、可验证目标**时表现最好,而测试正好提供了这种目标。TDD 工作流:

1. 让 Claude **先写测试**,基于预期的输入输出。明确告诉它"这是 TDD,先不要写实现"
2. **运行测试,确认它们失败**(此时还没有实现)
3. 让 Claude **编写实现代码,直到测试全部通过**——不准修改测试来迁就实现
4. 通过后再提交

这样做的好处:测试成了"防止 Claude 写错"的护栏,它会自己反复跑测试、自我修正,直到达标。

### **善用 Plan 模式**

对于**复杂、影响面大**的任务,不要直接让 Claude 改代码。按 `Shift+Tab` 切换到 **Plan 模式**:此时 Claude **只读不写**,先探索代码、给出完整方案,等你批准后才执行。

适合用 Plan 模式的场景:重构、跨多文件改动、架构调整、不确定方案的任务。这能避免"改了一半发现方向错了"的尴尬。

### **多 Claude 协作**

一个 Claude 不够用时,可以并行开多个:

- **写 + 审分离**:一个会话负责写代码,**另开一个会话专门审查**。"当局者迷",独立的审查会话往往能发现写代码那个会话忽略的问题
- **git worktree 并行**:用 `git worktree` 为不同任务创建独立工作目录,每个目录开一个 Claude,**并行推进互不干扰**的多个功能
- **生成 + 验证分离**:一个负责实现,一个负责写测试验证,形成制衡

> 多会话协作时,可以用文件(如 scratchpad、临时 markdown)在不同 Claude 之间传递信息和上下文。

## **效率技巧**

掌握下面这些细节,能让日常使用顺畅得多。

### **给精确的指令**

指令越具体,Claude 一次到位的概率越高。模糊的指令会让它猜,而猜往往是错的。

- ❌ "修复这个 bug"
- ✅ "用户在登录页输入错误密码时,页面会白屏。预期应该弹出红色错误提示。请定位 `LoginForm` 组件里的错误处理逻辑并修复。"

提供**预期行为、相关文件、约束条件**,而不是只丢一句话。

### **善用图片和截图**

Claude 能读图。这些场景贴图比打字描述高效得多:

- **UI 还原**:贴设计稿,让它实现对应界面
- **报错排查**:截图错误页面或控制台报错
- **复现问题**:贴出异常的界面状态

直接把图片拖进终端,或粘贴剪贴板里的截图即可。

### **及时纠偏**

不要等 Claude 跑偏很远才纠正。它走错方向时,**立刻打断**:

- **`Esc`**:打断当前操作
- **双击 `Esc`**:回退到历史某条消息,**编辑后重新发起**,相当于"换个说法重来"
- 发现方向不对,直接说"停,这个思路不对,应该……",及时把它拉回正轨

越早纠偏,浪费的上下文和时间越少。

### **管理上下文窗口**

上下文是有限资源,塞满了会变慢、变笨。

- **`/clear`**:任务切换时清空上下文,开始全新会话。**这是最常用、最该养成习惯的命令**——做完一个任务就 clear,别让无关历史拖累下一个任务
- **`/compact`**:压缩当前上下文,保留要点、丢弃冗余,适合长任务中途瘦身
- 不要在一个会话里连续干十个不相关的任务——该 clear 就 clear

### **交互小技巧**

- **`@` 引用文件**:输入 `@` 加文件名,把指定文件喂给 Claude,比让它自己找更准
- **Tab 补全**:输入文件路径时按 Tab 自动补全
- **拖拽文件**:把文件直接拖进终端窗口引用
- **`#` 快速记忆**:`#` 开头输入的内容会被追加进 CLAUDE.md
- **`/` 查看命令**:输入 `/` 浏览所有可用斜杠命令

## **高级用法**

当你想把 Claude Code 嵌入**脚本、流水线、自动化系统**时,这些能力派上用场。

### **无头模式(Headless)**

用 `claude -p "提示词"` 可以**非交互式**调用 Claude,执行完直接返回结果,适合脚本和自动化:

```bash
claude -p "总结这个项目的目录结构和主要模块"
```

常用参数:

- `-p / --print`:打印模式,执行后输出结果即退出
- `--output-format json`:以 JSON 格式输出,便于程序解析
- `--allowedTools`:限定本次允许使用的工具

### **管道与批量处理**

无头模式可以接入 Unix 管道,把 Claude 当成"智能文本处理器":

```bash
# 把日志喂给 Claude 分析
cat error.log | claude -p "找出这些日志里的根本错误原因"

# 批量处理:对每个文件执行同样的操作
for file in src/*.js; do
  claude -p "给 $file 添加 JSDoc 注释" --allowedTools "Edit"
done
```

这让 Claude 能融入既有的命令行工作流,处理数据、批量改文件、生成报告。

### **CI/CD 集成**

把无头模式放进流水线,实现自动化的代码审查、修复或生成:

- **自动代码审查**:PR 触发时,让 Claude 审查 diff 并留下评论
- **自动修复 lint / 测试失败**:CI 失败时让 Claude 尝试修复
- **生成变更日志、文档**:发版时自动生成

> 在 CI 环境中,因为是隔离沙箱,可以配合 `--dangerously-skip-permissions` 跳过确认,实现全自动。

### **子代理(Subagents)**

面对复杂任务,主 Claude 可以**派遣专业子代理**处理隔离的子任务。子代理在**独立的上下文**中运行,完成后只把结论返回给主代理。

优势:

- **节省主上下文**:子代理读了一堆文件、做了大量探索,但主会话只收到精炼结论,不会被中间过程塞满
- **专业化分工**:可在 `.claude/agents/` 定义专门角色(如 `code-reviewer`、`security-auditor`),各司其职
- **并行处理**:多个子代理可同时处理互不相关的子任务

适合"需要大量探索但只要结论"的场景,例如全代码库搜索、多文件审查、专项分析。

### **subagent 还是多 Claude 协作?**

容易混淆的还有一组:**子代理(subagent)** 和前面讲的**多 Claude 协作**(开多个会话、git worktree)。两者都是"多个 AI 干活",但定位完全不同:

| 维度 | 子代理(subagent) | 多 Claude / 代理团队 |
| --- | --- | --- |
| **谁来调度** | 主 Claude **自动派遣** | **你手动**开多个会话 |
| **上下文** | 子代理独立上下文,只回传结论 | 每个会话各自独立,靠文件/你来传递 |
| **协作方式** | 主从关系,子代理服务于主任务 | 平等关系,各做各的或互相制衡 |
| **典型场景** | 一个任务内的子步骤(搜索、审查、分析) | 多个独立功能并行 / 写审分离 |
| **成本** | 自动、轻量,无需你管 | 需要你切换、协调,管理成本高 |

**怎么选:**

- **同一个任务内**需要"分头探索再汇总"→ 用 **subagent**,让主 Claude 自动派。你只管提需求,它自己拆。
- **多个互不相关的功能**要同时推进 → 用 **git worktree + 多会话**,每个功能一个独立环境。
- **想要独立视角审查/制衡**(写代码的那个会"当局者迷")→ 开**第二个会话**专门审查或写测试。

> 简单说:**subagent 是 Claude 自己的"分身助手",由它调度;多会话是你手里的"多个工人",由你调度。** 任务内部的复杂度交给 subagent,任务之间的并行交给多会话。

### **两种角色,该配哪些子代理?**

现实里,**代码基本都是 AI 写的**——它质量和速度都够高,懂技术的人也不会去手敲。所以真正决定"该配哪些子代理"的,不是"谁写代码",而是**你能在哪个层面介入和把关**:你懂技术、能看懂代码并做技术决策,还是你只站在产品/业务层、靠规格和验收来控制结果。下面两种典型定位,需要的"班子"完全不同。

#### **情况一:全栈开发——你懂技术,拆业务 + 指挥 + 技术把关**

代码 AI 写,但**你能读懂、能在代码和架构层面做判断**。你的工作是把业务拆成清晰的技术任务、指挥 AI 实现、并在关键处技术把关。这时子代理的作用是**把技术信息喂到你面前供你判断**,以及分担"重活但只要结论"的探索。够用的就这几个:

| 子代理 | 职责 | 为什么需要 |
| --- | --- | --- |
| **code-explorer**(代码勘探) | 在大代码库里定位相关模块、理清调用链和现有实现 | 大项目不可能全读,让它去翻、只回传"这个功能涉及哪些文件、现状如何",供你做拆解决策 |
| **spec-reader**(文档解析) | 读需求文档 / 接口文档,提炼出明确的接口契约、字段、边界 | 把一份长 PRD/API doc 压成"要实现什么、输入输出是什么",避免主上下文被长文档塞满 |
| **test-writer**(测试编写) | 按预期行为写单测 / 集成测试 | 配合 TDD,给新功能上护栏 |
| **code-reviewer**(代码审查) | 审查改动:边界、异常、回归风险、风格 | 改大项目最怕"改 A 坏 B",独立视角把回归隐患摆到你面前,**最终由你判断** |

> 这套班子的特点是**轻**:因为**你自己能在代码层兜底**,子代理只需在"探索现状、读文档、写测试、查回归"这几个点上分担,把材料整理好交给你定夺。绝大多数模块级开发,有 `code-explorer` + `code-reviewer` 两个就够用了。

#### **情况二:产品经理——你不碰技术细节,只提需求和验收**

同样是 AI 写代码,但**你不在代码/架构层面介入**,只提需求、做产品决策、做验收。你看不到(或不想看)技术层的问题,要 Claude **从零交付一个完整可用的成熟软件**。这时你需要的不是几个辅助,而是**一支自带质量关卡的完整工程团队**——因为你无法在代码层兜底,团队必须自己把每道关都守住:

| 子代理 | 职责 | 对应环节 |
| --- | --- | --- |
| **product-analyst**(需求分析) | 把你的口头需求拆成清晰的功能规格、用户故事、验收标准 | 需求 → 规格 |
| **architect**(架构师) | 定技术栈、模块划分、数据模型、接口设计 | 规格 → 方案 |
| **feature-developer**(功能开发) | 按规格逐个实现功能模块(可并行多个) | 编码 |
| **test-engineer**(测试工程) | 写测试、跑测试、保证覆盖率 | 质量 |
| **code-reviewer**(代码审查) | 审查每个功能的实现质量与一致性 | 质量 |
| **security-auditor**(安全审计) | 查注入、鉴权、敏感信息泄露等安全问题 | 质量 |
| **devops**(部署运维) | 配置构建、CI/CD、打包、部署脚本 | 交付 |
| **doc-writer**(文档编写) | 写 README、使用文档、变更日志 | 交付 |

> 这套班子的特点是**全且需要编排**:你不在技术层把关,所以必须靠子代理 + 多会话把"需求 → 架构 → 开发 → 测试 → 审查 → 安全 → 部署"这条流水线**和它自带的质量关卡**一起补全。实践中常是**多会话(代理团队)+ 各角色子代理**配合:用 git worktree 让多个 `feature-developer` 并行做不同模块,每个模块完成后过一遍 `code-reviewer`、`test-engineer` 和 `security-auditor`。

**两种角色的本质区别——不在"谁写代码",而在"你在哪一层兜底":**

- **情况一**你在**代码/架构层兜底**:AI 写、AI 探索、AI 审查,但回归风险、技术方案这些关键判断**最终过你的眼**。子代理把材料整理好,决策权在你,所以班子可以精简。
- **情况二**你只在**产品/验收层兜底**:你不看代码,质量必须由团队**内部的关卡自己守住**(审查、测试、安全审计缺一不可)。你的精力全部放在**把需求规格写清楚、把验收标准定死**上。

> 一个递进的判断:**你介入的层面越浅,就越要把质量护栏堆厚。** 情况一你能在代码层随时拦截,所以护栏可以薄;情况二你失去了代码层的兜底,只能靠"清晰的规格 + 严格的测试/审查/安全关卡"来保证交付不出事——这也是为什么角色越往"甲方"靠,前置的规格和验收投入越关键。

## **实战:从零开发一个工具的完整链路**

前面讲的都是"零件",这一章把它们串成一条**完整流水线**。假设你要从零开发一个命令行小工具(比如一个"批量重命名文件"的 CLI),看看一个老手会怎么指挥 Claude,以及每一步该动用哪些特性。

### **第 0 步:搭好地基(只做一次)**

动手写代码之前,先把"环境"配好,这决定了后续所有交互的质量:

1. **初始化项目骨架**:让 Claude 跑 `/init`,或直接说"初始化一个 Python CLI 项目,用 click 做命令解析,pytest 做测试",生成基础结构。
2. **写 CLAUDE.md**:把关键约定固化进去——技术栈、目录结构、`运行测试用 pytest`、`提交信息用约定式 commit`、代码风格等。这一步花 5 分钟,后面省 1 小时。
3. **配 settings.json**:把 `pytest`、`ruff`、`git status/diff/commit` 等安全高频操作放进 `allow`,避免后面被反复打断。

> 这一步对应前两章的"配置与定制"。地基打好,Claude 才知道"在这个项目里该怎么做事"。

### **第 1 步:探索 + 计划(别急着写)**

需求到手,**先切 Plan 模式**(`Shift+Tab`)。告诉 Claude:

> "我要做一个批量重命名文件的 CLI,支持正则替换、预览模式(dry-run)、批量撤销。**先不要写代码**,深入想一下(ultrathink)整体设计:命令怎么分、核心模块怎么拆、有哪些边界情况,把方案写出来给我看。"

这一步用到:**Plan 模式**(只读不写)+ **精确指令**(说清功能和约束)+ **让它深度思考**。Claude 会产出一份方案,你审阅、纠偏、敲定。**方案错了改一句话,代码错了改半天**——前置规划的价值就在这。

### **第 2 步:TDD 打地基功能**

方案定了,核心逻辑用 **TDD** 推进。对"正则替换"这个核心功能:

> "用 TDD 做正则替换功能。**先写测试**,覆盖:普通替换、正则替换、无匹配、特殊字符转义这几种情况。**先别写实现**,写完跑一遍确认它们都失败。"

确认测试失败后:

> "现在写实现,直到所有测试通过。**不准改测试来迁就实现**。"

测试成了护栏,Claude 会自己反复跑 `pytest`、自我修正,直到全绿。**这是让 AI"自验证"最可靠的方式。**

### **第 3 步:派 subagent 处理探索性子任务**

做到"批量撤销"功能时,需要先搞清楚现有代码怎么记录操作历史。这种"要读一堆文件但只要个结论"的活,**交给 subagent**:

> "我要加撤销功能。先调研一下当前代码里文件操作是怎么执行和记录的,有没有现成的历史/日志机制可以复用,给我一个结论和建议方案。"

主 Claude 会派子代理去翻代码,**只把结论回传**,你的主会话上下文不会被一堆文件内容塞满。

### **第 4 步:截图驱动调错**

跑起来发现 dry-run 预览的输出排版乱了。**别用文字描述,直接截图**:把终端输出截图拖进去:

> "预览输出的对齐乱了,看截图。应该左边原文件名、右边新文件名对齐成两列。修一下。"

**贴图比打字描述精确得多**,尤其是排版、UI、报错这类视觉问题。

### **第 5 步:及时纠偏 + 管好上下文**

- 写"批量撤销"时,如果 Claude 开始往"加个数据库存历史"的方向跑偏(过度设计),**立刻 `Esc` 打断**:"停,不用数据库,一个 JSON 文件记录上一次操作就够了。"——越早拦越省事。
- 这个功能做完、要开始做下一个独立功能(比如"配置文件支持")前,**`/clear` 清空上下文**。重命名功能的一堆历史对新功能毫无用处,留着只会拖慢、干扰。

### **第 6 步:开第二会话做独立审查**

核心功能都完成后,**另开一个 Claude 会话**专门审查(写代码那个会话"当局者迷"):

> "审查 `src/` 下的代码,重点看:边界情况漏没漏、有没有未处理的异常、命名是否清晰、测试覆盖够不够。只指出问题,先别改。"

独立视角常能揪出原会话忽略的问题。确认后再回到主会话逐条修。

### **第 7 步:用 command 固化发版流程**

工具做好了,以后每次发版的流程都一样。把它做成 **command**——新建 `.claude/commands/release.md`:

```markdown
请执行发版流程:

1. 运行 `pytest` 确保全部通过
2. 运行 `ruff check` 确保无 lint 错误
3. 更新 CHANGELOG.md,汇总自上次 tag 以来的改动
4. 按语义化版本更新版本号
5. 提交并打 tag
```

以后发版只需打 `/release`,**把一次性的手工流程沉淀成可复用资产**。

### **第 8 步(可选):无头模式接入自动化**

如果想让这个流程**全自动**,用**无头模式**接进 CI:

```bash
# CI 里:PR 触发时自动审查 diff
git diff main | claude -p "审查这段 diff,指出潜在 bug 和风格问题" --output-format json
```

CI 是隔离沙箱,可配合 `--dangerously-skip-permissions` 实现全自动审查、修复或生成。

### **回顾这条链路**

把上面串起来,一个高效的"从零到完美"工作流是这样的:

> **配地基(CLAUDE.md + 权限)→ Plan 模式定方案 → TDD 写核心 → subagent 做探索 → 截图调错 → 及时纠偏 + 勤 clear → 二号会话审查 → command 固化流程 → 无头模式自动化**

不是每个项目都要用满所有特性,但理解"**什么阶段该用什么武器**",就是把 Claude Code 从"能用"调到"好用"的关键。

## **小结**

Claude Code 用得好的关键,可以浓缩成几条:

1. **写好 CLAUDE.md**——这是性价比最高的投入,持续迭代
2. **选对定制机制**——规则用 rules,动作用 command,智能用 skill
3. **精细化权限**——安全操作放行,危险操作留住
4. **先规划再编码**——探索、计划、TDD,别让它盲写
5. **分清两种"多 AI"**——任务内拆解用 subagent,任务间并行用多会话
6. **管好上下文**——勤用 `/clear`,只接必要的 MCP 和工具
7. **及时纠偏**——走错立刻打断,别等它跑远
8. **善用自动化**——无头模式、command、子代理,把 Claude 嵌入工作流

而真正把这些串起来的,是那条**从零到完美的链路**:配地基 → 定方案 → TDD 写核心 → subagent 探索 → 截图调错 → 纠偏与 clear → 独立审查 → command 固化 → 无头自动化。

工具会不断更新,但"清晰地表达意图、给出可验证的目标、保持上下文干净"这些原则不会过时。
