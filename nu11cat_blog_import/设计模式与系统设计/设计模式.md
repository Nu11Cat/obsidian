## 设计模式

#### 设计模式是什么，有哪些类

设计模式是软件工程中常用的解决特定问题的模版或者蓝图，可以帮助我们开发者以一种更加清晰、高效和可重用的方式来编写代码。通常分为三类：

1. 创建型模式：涉及对象实例化，用于创建对象的模式，可以增加程序的灵活性和可重用性。常见的创建型模式有工厂方法、抽象工厂、单例、建造者、原型等。
2. 结构型模式：涉及类和对象的组合，用于设计类和对象的结构，以便更好地实现程序的功能。常见的结构型模式有适配器、桥接、组合、装饰、外观、享元、代理等。
3. 行为型模式：关注对象之间的通信，包括责任链、命令、解释器、迭代器、中介者、备忘录、观察者、状态、策略、模板方法、访问者等。

----

#### 软件设计原则

软件设计原则中最核心的是 **SOLID 原则**：

**单一职责原则 (Single Responsibility Principle, SRP)**

- **核心思想**：一个类或模块应该有且只有一个引起它变化的原因。
- **通俗来说**：一个类只做一件相关的事情。如果一个类承担了多个职责，那么当其中一个职责发生变化时，可能会影响到其他职责的实现，这会增加代码的复杂性和耦合度。

**开闭原则 (Open-Closed Principle, OCP)**

- **核心思想**：软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。
- **通俗来说**：当需要增加新功能时，我们应该通过增加新的代码（例如新的子类或实现类）来实现，而不是去修改已有的、工作良好的代码。这是面向对象设计中最重要的一条原则，是实现系统可复用性和可维护性的基石。

**里氏替换原则 (Liskov Substitution Principle, LSP)**

- **核心思想**：所有引用基类的地方，都必须能够透明地使用其子类的对象。
- **通俗来说**：子类应当可以完全替换父类，并且程序的行为不会产生任何错误或异常。这意味着子类在继承父类时，不应该改变父类预期的行为。它是实现开闭原则的重要方式之一。

**接口隔离原则 (Interface Segregation Principle, ISP)**

- **核心思想**：客户端不应该被强迫依赖它不需要的接口。一个类对另一个类的依赖应该建立在最小的接口之上。
- **通俗来说**：接口的设计要“专”，不要设计大而全的“胖接口”，而应该根据功能设计多个细粒度的接口。这样可以避免一个类为了实现接口中的某一个功能，而不得不实现所有其他它并不需要的功能。

**依赖倒置原则 (Dependency Inversion Principle, DIP)**

- **核心思想**：高层模块不应该依赖于低层模块，两者都应该依赖于抽象；抽象不应该依赖于细节，细节应该依赖于抽象。
- **通俗来说**：我们要面向接口编程，而不是面向实现编程。代码的依赖关系应该通过抽象（如接口或抽象类）来建立，而不是具体的实现类。这样可以大大降低模块间的耦合度，方便系统的升级和扩展。

## 创建型模式

### **单例模式**

#### 是什么？应用场景？优缺点？

1. 是什么
   单例模式是一种 创建型设计模式，它保证一个类在整个应用中 只有一个实例，并提供一个 全局访问点 来获取该实例。
2. 应用场景

需要 全局唯一对象 的场景，例如配置管理器、日志管理器、缓存管理器。

频繁创建对象开销大 的场景，如数据库连接池、线程池。

控制资源访问或协调系统中某些全局操作的场景。

1. 优缺点

优点：

保证全局唯一，避免重复创建资源消耗。

提供统一访问点，方便管理和维护。

缺点：

单例对象生命周期长，可能造成内存占用。

并发访问需要考虑线程安全，增加实现复杂度。

在某些场景下不易扩展，过度使用会导致系统耦合度高。

---

#### 有哪些使用实例

- 一个是Spring Framework 中的单例Bean
- 然后是 Java.lang.Runtime 类，Runtime封装了Java应用运行时的环境。每个Java应用只有一个运行时环境，所以必须用单例来保证唯一性。
- 还有日志框架中，日志记录器（Logger）通常也是以单例或类似单例的模式（如缓存池）来管理的。是为了避免为每个需要日志的类都创建一个新的Logger对象，造成资源浪费，并保证日志输出的统一管理。

----

#### 常见的单例模式实现？优缺点？如何保证线程安全？

单例模式的实现关键在于：1）构造方法私有化；2）自行创建实例；3）提供获取实例的静态方法。 难点在于如何处理多线程环境。

1. 饿汉式

```java
public class EagerSingleton {
    // 1. 静态常量，在类加载时就完成实例化
    private static final EagerSingleton INSTANCE = new EagerSingleton();

    // 2. 私有构造方法，堵死外部通过new创建实例的可能
    private EagerSingleton() {}

    // 3. 提供全局访问点
    public static EagerSingleton getInstance() {
        return INSTANCE;
    }
}
```

- 优点：实现简单，无线程安全问题。
- 缺点：不管用不用，类加载时就创建实例，如果实例很耗资源，或者始终没被用到，会造成内存浪费。不符合“懒加载”思想。

2. 懒汉式（Lazy Initialization）- 线程不安全版

```java
public class UnsafeLazySingleton {
    private static UnsafeLazySingleton instance;

    private UnsafeLazySingleton() {}

    public static UnsafeLazySingleton getInstance() {
        // 如果instance为null，则创建
        if (instance == null) {
            instance = new UnsafeLazySingleton(); // ！！！多线程下会创建多个实例
        }
        return instance;
    }
}
```

- 缺点：线程不安全。如果两个线程同时进入 `if (instance == null)`判断，就会创建两个实例。

3. 懒汉式 - 线程安全版（同步方法）

```java
public class SynchronizedLazySingleton {
    private static SynchronizedLazySingleton instance;

    private SynchronizedLazySingleton() {}

    // 使用synchronized关键字修饰方法，保证线程安全
    public static synchronized SynchronizedLazySingleton getInstance() {
        if (instance == null) {
            instance = new SynchronizedLazySingleton();
        }
        return instance;
    }
}
```

- 优点：线程安全。
- 缺点：效率太低。`synchronized`修饰整个方法，每次调用`getInstance()`都要同步，而实际上只需要在第一次创建实例时同步。

4. 双重检查锁（Double-Checked Locking - DCL）

```java
public class DCLSingleton {
    // 使用volatile关键字禁止指令重排序，这是关键！
    private static volatile DCLSingleton instance;

    private DCLSingleton() {}

    public static DCLSingleton getInstance() {
        // 第一次检查：如果instance不为null，直接返回，避免不必要的同步
        if (instance == null) {
            // 同步代码块
            synchronized (DCLSingleton.class) {
                // 第二次检查：进入同步块后再次检查，确保只有一个线程创建实例
                if (instance == null) {
                    instance = new DCLSingleton(); // 1. 分配内存 2. 初始化对象 3. 将instance指向内存地址
                }
            }
        }
        return instance;
    }
}
```

- 为什么用`volatile`？

- `instance = new DCLSingleton();`这行代码并非原子操作，它可能被JVM指令重排序为 1->3->2。
- 如果线程A执行了1和3，但还没执行2，此时instance已不为null。线程B在第一次检查时发现instance不为null，便直接返回一个尚未初始化完成的对象，导致错误。
- `volatile`关键字可以防止这种指令重排序，保证写操作之前的代码不会重排序到写操作之后。

- 优点：线程安全，延迟加载，效率较高。
- 缺点：实现稍复杂，但已是经典写法。

5. 静态内部类（Static Inner Class） - 推荐使用

```java
public class InnerClassSingleton {
    // 1. 私有构造
    private InnerClassSingleton() {}

    // 2. 静态内部类
    private static class SingletonHolder {
        private static final InnerClassSingleton INSTANCE = new InnerClassSingleton();
    }

    // 3. 获取实例
    public static InnerClassSingleton getInstance() {
        return SingletonHolder.INSTANCE;
    }
}
```

- 原理：

- JVM在加载外部类时，不会加载其内部类。`SingletonHolder`是一个静态内部类，不依赖于外部类的实例。
- 只有当调用`getInstance()`方法时，才会触发JVM加载`SingletonHolder`类，从而初始化`INSTANCE`。
- 类的初始化阶段是线程安全的，JVM会保证一个类的`<clinit>()`方法（类构造器）被正确加锁同步。

- 优点：实现了懒加载，由JVM保证线程安全，实现简单。这是一种优雅的实现方式。

6. 枚举（Enum） - 《Effective Java》作者强力推荐

```java
public enum EnumSingleton {
    INSTANCE; // 这就是单例的实例

    // 可以添加业务方法
    public void doSomething() {
        System.out.println("Doing something...");
    }
}

// 使用方式：
// EnumSingleton.INSTANCE.doSomething();
```

- 优点：

1. 绝对防止多次实例化：即使面对复杂的序列化或反射攻击，枚举也能保证只有一个实例。JVM从底层确保枚举值的唯一性。
2. 线程安全：枚举的初始化本身就是线程安全的。
3. 实现简单。

- 缺点：不够灵活（无法继承其他类，是`final`的），但大多数场景下这不算缺点。

-----

**为什么枚举实现是最佳实践？**

**枚举实现**是最佳实践，因为它 **天然线程安全、单例唯一、可防止反射和序列化漏洞**，同时代码最简洁。

### 工厂模式

(简单工厂/工厂方法/抽象工厂)

#### 是什么？应用场景？优缺点？

**1. 是什么**
工厂模式是一种 **创建型设计模式**，它将对象的创建逻辑 **封装在工厂类中**，客户端通过工厂而不是直接 new 来获取对象，从而 **解耦对象的创建和使用**。

------

**2. 核心分类**

1. **简单工厂模式（Simple Factory）**

- 通过一个工厂类，根据参数返回不同的产品实例。
- **优点**：客户端无需关心对象具体创建逻辑。
- **缺点**：工厂类职责单一但集中过重，违反开闭原则（每增加新产品都需修改工厂）。

```java
public class CarFactory {
    public static Car createCar(String type) {
        if ("Benz".equals(type)) return new Benz();
        else if ("BMW".equals(type)) return new BMW();
        return null;
    }
}
```

1. **工厂方法模式（Factory Method）**

- 为每个产品定义一个具体工厂类，工厂负责创建对应产品。
- **优点**：符合开闭原则，增加新产品无需修改原有工厂。
- **缺点**：工厂类数量多，系统复杂度增加。

```java
public interface CarFactory {
    Car createCar();
}
public class BenzFactory implements CarFactory {
    public Car createCar() { return new Benz(); }
}
```

1. **抽象工厂模式（Abstract Factory）**

- 提供接口创建一组相关或依赖的产品族，而无需指定具体类。
- **优点**：一次创建一整套产品，保证产品之间的兼容性。
- **缺点**：增加产品族时需要修改接口或增加工厂实现。

```java
public interface GUIFactory {
    Button createButton();
    TextField createTextField();
}
public class MacFactory implements GUIFactory { ... }
public class WindowsFactory implements GUIFactory { ... }
```

------

**3. 应用场景**

- 对象的创建逻辑复杂，或者客户端无需关心具体类。
- 系统需要灵活扩展产品种类时。
- 保证同一产品族中对象的一致性或兼容性。

----

#### 有哪些使用实例

- 一个是 Spring Framework 的 `BeanFactory`，BeanFactory 是最基本的工厂接口，ApplicationContext 是 BeanFactory 的高级实现，实际项目中我们使用的就是它。
- 然后是JDK - `Collection.iterator()`是工厂方法，`Collection`接口定义了 `iterator()`这个工厂方法。`ArrayList`和 `HashSet`作为具体工厂，分别重写 `iterator()`方法，返回它们各自专用的迭代器 `Itr`和 `KeyIterator`作为具体产品。
- 应用实践中的一个典型例子就是支付系统
- 当你看到代码中出现大量的 `if-else`或 `switch-case`来创建不同类型的对象时，就应该立刻想到——这里可以用工厂模式来优化

### 建造者模式

#### 是什么？应用场景？优缺点？

建造者模式是为了解决复杂对象的创建问题。建造者模式通过一个内部静态类（Builder），使用链式方法调用的方式，一步步设置复杂对象的参数，最后通过`build()`方法生成一个稳定、完整的产品对象。它完美解决了多参数对象创建时的可读性、安全性和灵活性问题。

------

应用场景：

- 当创建一个对象需要很多步骤，并且这些步骤的顺序可以是灵活的。
- 当同一个创建过程需要产生不同的产品对象时。
- 当要创建的产品对象有复杂的内部结构，并且内部属性本身也可能是复杂的对象。
- 最主要、最常见的场景：当一个类的构造方法参数过多（比如超过4个），且很多参数都是可选时，使用建造者模式可以极大地提升代码的可读性和可维护性。

------

优点

- 创建逻辑清晰，步骤稳定
- 可构造不同配置信息的对象，灵活扩展
- 参数可读性强，不会出现构造器爆炸
- 便于维护，链式调用友好

缺点

- 类数量增多，多一个 Builder 类
- 构建步骤复杂时可能需要大量代码

----

#### 有哪些使用实例

Lombok的 `@Builder`注解

即：

```java
import lombok.Builder;
import lombok.ToString;

@Builder
@ToString
public class User {
    private final String name;
    private final int age;
    private final String phone;
    private final String email;
}

// 使用方式一模一样
User user = User.builder()
                .name("李四")
                .age(30)
                .phone("13900139000")
                .build();
```

## 结构型模式

### 适配器模式

#### 是什么？应用场景？优缺点？

适配器模式（Adapter Pattern）是一种结构型设计模式，它的核心作用是将一个类的接口转换成客户端所期望的另一个接口，从而使原本因接口不兼容而不能在一起工作的类可以协同工作。

可以把它想象成一个我们日常生活中常用的电源适配器或转换头。

------

包含三个核心角色：

- Target (目标接口)：客户端（Client）期望和它直接交互的接口。在上面的例子里，就是那个三孔插座。
- Adaptee (被适配者)：已存在的、但接口与 `Target` 不兼容的类。也就是那个两脚插头。
- Adapter (适配器)：模式的核心。它实现了 `Target` 接口，同时内部包装了一个 `Adaptee` 类的实例，负责将对 `Target` 接口的调用转换为对 `Adaptee` 接口的调用。它就是那个转换头。

------

优点：

1. 解耦：将目标类和适配者类解耦，提高了类的透明性和复用性。
2. 符合开闭原则：可以在不修改现有代码的情况下增加新的适配器。
3. 灵活性：可以适配多个不同的类。

缺点：

1. 增加复杂性：会增加类的数量，有时可能使代码变得复杂。
2. 不宜滥用：如果接口可以统一修改，直接修改接口可能比使用适配器更简单。

----

#### 两种实现方式

**类适配器**：

- **原理**：通过**类继承**来实现。`Adapter` 类同时继承 `Adaptee` 类并实现 `Target` 接口。
- **特点**：由于 Java 是单继承，这意味着 `Adapter` 只能适配一个 `Adaptee` 类。它的耦合度相对较高。

**对象适配器**：

- **原理**：通过**对象组合/关联**来实现。`Adapter` 类实现 `Target` 接口，并在内部持有一个 `Adaptee` 类的实例。
- **特点**：这种方式更加灵活，因为 `Adapter` 可以适配 `Adaptee` 的任何子类。它遵循了“合成/聚合复用原则”，是更推荐、更常用

----

#### 有哪些使用实例

Spring框架大量使用适配器模式，最典型的就是Spring MVC中的HandlerAdapter。

在Spring MVC中，处理器（Controller）可以有多种实现方式：

- 实现`Controller`接口的简单控制器
- 使用`@Controller`注解的基于注解的控制器
- 其他类型的处理器（如HTTP Invoker等）

### 代理模式

代理模式简单来说就是 **我们使用代理对象来代替对真实对象(real object)的访问，这样就可以在不修改原目标对象的前提下，提供额外的功能操作，扩展目标对象的功能。**

**代理模式的主要作用是扩展目标对象的功能，比如说在目标对象的某个方法执行前后你可以增加一些自定义的操作。**

代理模式有静态代理和动态代理两种实现方式：

-----

#### 静态代理

静态代理中，我们对目标对象的每个方法的增强都是手动完成的，**非常不灵活（比如接口一旦新增加方法，目标对象和代理对象都要进行修改）且麻烦(需要对每个目标类都单独写一个代理类）**

从 JVM 层面来说， **静态代理在编译时就将接口、实现类、代理类这些都变成了一个个实际的 class 文件。**

**实现步骤**：

1，定义一个接口及其实现类

2，创建一个代理类同样实现这个接口

3，将目标对象注入进代理类，然后在代理类的对应方法调用目标类中的对应方法。这样的话，我们就可以通过代理类屏蔽对目标对象的访问，并且可以在目标方法执行前后做一些自己想做的事情

静态代理就是写一个代理类，这个类实现和目标类一样的接口，然后在方法里，**手动在调用目标方法的前后加上你想要的逻辑**，比如日志、权限校验、时间统计等等。

----

#### 动态代理

相比于静态代理来说，动态代理更加灵活。我们不需要针对每个目标类都单独创建一个代理类，并且也不需要我们必须实现接口。Spring AOP、RPC 框架的实现都依赖了动态代理。

**从 JVM 角度来说，动态代理是在运行时动态生成类字节码，并加载到 JVM 中的**

**JDK的动态代理**

`**InvocationHandler**` **接口和** `**Proxy**` **类是核心。**

`Proxy`中使用频率最高的方法是：`newProxyInstance()` ，这个方法主要用来生成一个代理对象。

这个方法一共有 3 个参数：**loader** :类加载器，用于加载代理对象；**interfaces** : 被代理类实现的一些接口；**h** : 实现了 `InvocationHandler` 接口的对象。

要实现动态代理的话，还必须需要实现`InvocationHandler` 来自定义处理逻辑。 当我们的动态代理对象调用一个方法时，这个方法的调用就会被转发到实现`InvocationHandler` 接口类的 `invoke` 方法来调用。

这个方法一共有 3 个参数：**proxy** :动态生成的代理类；**method** : 与代理类对象调用的方法相对应；**args** : 当前 method 方法的参数。

**实现步骤：**

1. 定义一个接口及其实现类；
2. 自定义 `InvocationHandler` 并重写`invoke`方法，在 `invoke` 方法中我们会调用原生方法（被代理类的方法）并自定义一些处理逻辑；
3. 通过 `Proxy.newProxyInstance(ClassLoader loader,Class<?>[] interfaces,InvocationHandler h)` 方法创建代理对象；

**CGLIB 动态代理机制**

**JDK 动态代理有一个最致命的问题是其只能代理实现了接口的类**。**我们可以用 CGLIB 动态代理机制来避免**。

**在 CGLIB 动态代理机制中** `**MethodInterceptor**` **接口和** `**Enhancer**` **类是核心。**

你需要自定义 `MethodInterceptor` 并重写 `intercept` 方法，`intercept` 用于拦截增强被代理类的方法。

这个方法一共有 4 个参数：**obj** : 被代理的对象（需要增强的对象）；**method** : 被拦截的方法（需要增强的方法）；**args** : 方法入参；**proxy** : 用于调用原始方法。

你可以通过 `Enhancer`类来动态获取被代理类，当代理类调用方法的时候，实际调用的是 `MethodInterceptor` 中的 `intercept` 方法。

**实现步骤：**

1. 定义一个类；
2. 自定义 `MethodInterceptor` 并重写 `intercept` 方法，`intercept` 用于拦截增强被代理类的方法，和 JDK 动态代理中的 `invoke` 方法类似；
3. 通过 `Enhancer` 类的 `create()`创建代理类；

**对比**

**灵活性**：动态代理更加灵活，不需要必须实现接口，可以直接代理实现类，并且可以不需要针对每个目标类都创建一个代理类。另外，静态代理中，接口一旦新增加方法，目标对象和代理对象都要进行修改，这是非常麻烦的！

**JVM 层面**：静态代理在编译时就将接口、实现类、代理类这些都变成了一个个实际的 class 文件。而动态代理是在运行时动态生成类字节码，并加载到 JVM 中的。

----

#### 代理模式和适配器模式有什么区别？

**目的不同**：代理模式主要关注控制对对象的访问，而适配器模式则用于接口转换，使不兼容的类能够一起工作。

**结构不同**：代理模式一般包含抽象主题、真实主题和代理三个角色，适配器模式包含目标接口、适配器和被适配者三个角色。

**应用场景不同**：代理模式常用于添加额外功能或控制对对象的访问，适配器模式常用于让不兼容的接口协同工作。

----

### 装饰器模式

它的核心思想是在不改变原有对象结构和代码的前提下，动态地为该对象添加额外的功能。装饰器模式是开闭原则的典范应用，它让我们在不修改既有代码的情况下，为对象赋予了无限的扩展可能。

------

#### 什么时候应该使用装饰器模式？

- 需要动态地给一个对象添加功能，这些功能也可以动态地撤销。
- 需要扩展一个类的功能，但使用继承会导致子类数量爆炸。
- 需要为对象添加的功能种类很多，使用继承关系无法实现（如Java I/O需要同时具备缓冲、数据类型处理等多种功能）。

------

优点

- 功能扩展灵活，可按需叠加
- 避免继承导致的类膨胀
- 原始对象与装饰功能解耦

缺点

- 多层装饰嵌套过深，阅读和调试困难
- 系统复杂度上升

----

#### 有哪些使用实例

装饰器模式在Java I/O库中得到了极致的体现：我们可以用 `BufferedInputStream` 来装饰它，为其增加缓冲功能以提高性能；还可以再用 `DataInputStream` 来装饰，进一步为其增加读取基本数据类型的能力。

-----

## 行为型模式

### 策略模式

#### 是什么？应用场景？优缺点？

策略模式将一组**可互换的算法/行为**封装成独立的策略类，并允许在运行时动态替换，从而让算法的变化独立于使用算法的代码。

应用场景

- 含多种业务分支，并可能新增（避免 if-else）
- 多个算法可以互相替换
- 希望按运行时条件动态选择算法
- 处理不同规则、不同计费方式、不同校验逻辑等场景

------

优点

- 消除大量 if-else
- 新增策略无需修改原代码，开闭原则
- 不同策略间独立、可复用
- 可运行时动态替换策略

缺点

- 策略数量可能变多
- 使用者需知道各策略的差别（否则不知道选哪一个）

---

#### 有哪些使用实例

- `java.util.Comparator`接口：这是策略模式最经典的例子。`Collections.sort(List list, Comparator c)`方法。它接收一个比较策略，然后根据这个策略对列表进行排序。你可以轻松地替换排序策略，而无需修改`sort`方法的内部实现。
- Spring框架中的`ResourceLoader`：Spring需要从不同地方加载资源（类路径、文件系统、网络等）。它使用`Resource`接口作为策略，`ClassPathResource`、`FileSystemResource`等是具体策略。
- 业务中的折扣/优惠券计算：不同的优惠券类型（满减、折扣、立减）就是不同的策略。

----

### 状态模式

当一个对象拥有多种状态，并且在不同状态下同一个方法有不同的行为时，状态模式通过将每个状态封装成独立的类，来消除复杂的条件判断语句

------

什么时候应该使用状态模式？

- 一个对象的行为取决于它的状态，并且它必须在运行时根据状态改变它的行为。
- 一个操作中含有庞大的多分支条件语句，且这些分支依赖于该对象的状态。

------

**优点**

- 将状态和行为封装在独立类中，结构清晰
- 易于扩展新的状态
- 减少大量条件判断（if/else 或 switch）

**缺点**

- 状态类增多，系统复杂度上升
- 状态切换逻辑分散在各状态类中，调试稍复杂

----

#### 有哪些使用实例

1. 订单系统：订单有`待支付`、`已支付`、`已发货`、`已完成`、`已取消`等状态。在不同状态下，订单的`支付`、`发货`、`确认收货`等操作的行为完全不同。状态模式是订单系统设计的经典选择。
2. 工作流引擎：一个审批流程中的节点（如`草稿`、`待审批`、`审批中`、`已通过`、`已驳回`），每个节点下的操作（提交、审核、撤回）行为不同。
3. 游戏开发：游戏角色有`空闲`、`移动`、`攻击`、`死亡`等状态。同一操作（如“受到攻击”）在不同状态下效果不同（移动中受攻击可能被打断，死亡状态下不受影响）。
4. TCP连接：TCP连接有`已建立`、`监听`、`已关闭`等状态，`打开`、`关闭`、`发送`、`接收`等操作在不同状态下行为截然不同。

-----

#### 状态模式 和 策略模式 的区别

状态模式和策略模式在结构上非常相似（都有一个上下文类、一个抽象接口和多个具体实现类），但它们的**意图**完全不同：

- **状态模式**：关注的是**状态的自动转换**。状态的变更通常由状态内部逻辑决定，对客户端是透明的。它旨在解决对象在不同状态下行为不同的问题，强调的是“下一个状态是什么”。
- **策略模式**：关注的是**策略的灵活选择**。具体使用哪种策略由客户端（或上层逻辑）决定并主动设置，策略之间通常没有直接关联。它旨在解决同一行为有不同实现算法的问题，强调的是“如何做”

-----

### 模板模式

模板模式是 定义算法的固定流程（模板），把其中可变的步骤延迟给子类去实现。  

------

适合“流程固定，但步骤可变”的场景：

- 各种业务流程：如下单 → 扣库存 → 生成流水
- 钩子方法场景：统一执行某些步骤，但允许子类按需覆盖
- 代码复用：共性逻辑在父类实现，不同逻辑由子类定制

------

优点

- 固定流程，确保算法结构稳定
- 复用父类逻辑，减少重复代码
- 扩展性强，子类只需实现不同步骤

缺点

- 类数量增多
- 继承结构导致子类受父类约束较多
- 父类改动可能影响所有子类

----

#### 有哪些使用实例

1. Java Servlet 中的 `HttpServlet`：

- `HttpServlet`的 `service()`方法就是一个模板方法。它规定了处理HTTP请求的流程：先解析请求，然后根据方法类型（GET/POST）调用对应的`doGet()`, `doPost()`方法。
- `doGet()`, `doPost()`就是需要子类实现的抽象方法。你写的Servlet只需要重写这些方法，而不用关心整个请求响应的流程。

1. Spring Framework 中的 `JdbcTemplate`：

- `JdbcTemplate`的 `execute()`, `query()`等方法都是模板方法。它们定义了操作JDBC的标准流程：获取连接、创建语句、执行SQL、处理结果集、关闭资源。
- 其中“处理结果集”这一步是变化的，通过`RowMapper`回调接口（类似于抽象方法）交给使用者实现。

1. 工作流审批：一个审批流程可能是固定的（提交 -> 经理审批 -> 总监审批 -> 结束），但每个审批环节的具体操作（如发送邮件、通知消息）可以有不同的实现。

----

### 观察者模式

 核心思想在于定义了一种一对多的依赖关系：当一个对象（我们称之为“被观察者”或“主题”）的状态发生改变时，所有依赖于它的对象（即“观察者”）都会自动收到通知并进行相应的更新。

这种模式的本质是解耦，它将被观察者和观察者分离开来，使得它们可以独立地变化和复用，而不需要知道对方的具体实现细节。

------

#### 核心角色

1. Subject（主题/被观察者）：它维护了一个观察者列表，可以动态地添加（注册）或删除（取消注册）观察者。当自身状态变化时，会通知所有注册的观察者。
2. Observer（观察者）：为一个接口，定义了一个更新方法（如 `update()`），当主题状态改变时，该方法会被调用。
3. ConcreteSubject（具体主题）：实现主题接口，存储自身状态。当状态改变时，给所有注册的观察者发送通知。
4. ConcreteObserver（具体观察者）：实现观察者接口，维护一个对具体主题的引用。当它的 `update()`方法被调用时，可以从主题那里获取最新状态并执行相应操作。

------

适合“一个变化，多个响应”的场景：

- 消息通知：如用户下单 → 通知库存、物流、营销
- UI 事件监听：按钮点击 → 多个监听器响应
- 配置中心刷新：Nacos / Apollo 配置变更 → 客户端立即感知
- 发布订阅模型：消息队列（MQ）本质也是观察者思想

------

优点：

- 解耦：主题和观察者之间是抽象耦合，主题不知道观察者的任何细节。
- 支持广播通信：主题一次通知，所有观察者都会收到。
- 符合开闭原则：可以轻松增加新的观察者，而无需修改主题代码。

缺点：

- 通知顺序：观察者收到通知的顺序是不确定的，如果观察者有依赖关系，可能会出问题。
- 循环调用：如果观察者之间或观察者与主题之间有复杂的依赖，可能导致循环调用，系统崩溃。
- 性能考虑：如果观察者数量非常多，或者观察者的更新操作非常耗时，会严重影响主题的性能。

------

#### 有哪些使用实例

- Java 有`java.util.Observer` 和 `Observable`（已过时，但概念保留，设计有缺陷）
-  Spring ApplicationEvent 事件机制

### 责任链模式

责任链模式的核心思想：**把多个处理逻辑按顺序串成一条链，请求从链头往链尾传递，直到某个处理者处理它。**

请求发送者不知道最终由谁处理，实现完全解耦。

------

应用场景

- 有多个对象可以处理同一个请求，但具体由哪个对象处理需要在运行时动态决定。
- 你想在不明确指定接收者的情况下，向多个对象中的一个提交请求。
- 可动态指定一组对象处理请求

------

优点

- 解耦：请求方不需要关心由谁处理
- 灵活扩展：随时添加、删除、调整处理者顺序
- 链式结构清晰

缺点

- 调试困难：请求可能被很多节点处理
- 可能出现无人处理的情况
- 链过长可能影响性能

------

#### 有哪些使用实例

1. Java Servlet 中的 Filter：这是责任链模式最经典的例子。一个HTTP请求需要经过多个Filter（如字符编码Filter、安全验证Filter、日志Filter等）的处理，每个Filter都可以决定是否将请求传递给下一个Filter。
2. Spring Security 的认证链：请求需要经过一系列的安全过滤器链（`SecurityFilterChain`），每个过滤器负责不同的安全任务（如检查JWT、验证权限等）。
3. 日志框架的日志级别处理：如Logback，一条日志消息会根据其级别（DEBUG、INFO、WARN、ERROR）在责任链上传递，直到找到能处理该级别的Appender（如ConsoleAppender、FileAppender）。
4. 工作流引擎：如请假审批、订单审核等流程，每个节点都是一个处理者。
