## 连接串与客户端角色

生产环境通常使用 **副本集（Replica Set）** 或 **分片集群（Sharded Cluster）** 的连接 URI，其中包含多个主机、副本集名称 `replicaSet=...` 等字段。客户端驱动会据此发现拓扑、处理主从切换与读路由。

单机开发环境与副本集在 **故障切换、读偏好、默认行为** 上存在差异；讨论「读是否一定最新」「写是否已持久到多数节点」时，需要同时说明拓扑与读写关注配置，避免脱离部署形态下结论。

-----------

## 读关注与写关注（readConcern / writeConcern）

**写关注（writeConcern）** 描述写操作在返回成功前需要满足的复制或持久化条件，例如是否等待复制到多数节点、是否刷盘等。不同级别在 **延迟、可用性、数据丢失窗口** 之间取舍不同。

**读关注（readConcern）** 描述读操作可见的数据版本范围，例如是否只读已提交到多数节点的数据。与拓扑、选举、从节点延迟共同影响「应用读到的数据与最近一次写」的时间关系。

默认配置随版本与部署可能调整；升级大版本或更换云厂商托管形态时，应在变更清单中核对这两项与驱动的兼容性说明。

-----------

## 副本集与选举（概念）

副本集由多个承载相同数据的节点组成，其中 **主节点（Primary）** 处理写请求，**从节点（Secondary）** 异步复制 oplog。主节点不可用时，集群会触发 **选举**，选出新的主节点。

选举期间可能出现短暂不可写；读请求若路由到从节点，可能读到尚未追上的数据，具体取决于读偏好与读关注设置。

规划副本集时通常同时考虑：成员数量与跨机架分布、仲裁节点（Arbiter）的使用场景与限制、备份与恢复策略与 oplog 容量。

-----------

## 分片集群与路由（概念）

当单集合数据量或写入吞吐超出单机承载能力时，可采用 **分片**：数据按 **片键（Shard Key）** 分布到多个分片，**mongos** 作为路由层转发请求，**配置服务器**保存元数据。

片键决定数据分布是否均衡。**单调递增**类片键容易形成热点分片；与查询模式不匹配的片键可能导致大量 **scatter-gather** 查询。片键在集合初次分片后难以无损调整，选型阶段需要结合访问模式与增长预期评审。

-----------

## 多文档事务与 ACID 边界

MongoDB 在 **副本集** 等部署形态下支持多文档事务（API 与关系型「BEGIN / COMMIT」类似）。事务有 **时间限制与资源占用** 约束，长事务、大批量修改可能触发超时或性能问题。

事务适用于需要跨文档一致性的场景；若业务可接受以单文档原子性建模（单文档内字段一起更新），往往更简单。是否启用事务应结合 **数据模型、访问路径、延迟预算** 评估，而不是仅凭功能列表决定。

-----------

## Change Streams（点到为止）

**Change Streams** 允许应用订阅集合或库上的数据变更流，用于缓存失效、下游同步、审计等。依赖 **副本集协议与 oplog**；部署需满足相应前提，消费端需处理重连与断点续订策略。

-----------

## 常见工程坑

### 大文档

BSON 文档有大小上限（以官方当前版本说明为准）。单文档过大时，更新与网络传输成本上升，且容易触碰数组字段无限增长的模型问题。超大字段可考虑对象存储与元数据分离。

### 热分片键

片键取值分布极度倾斜时，读写集中在少数分片，失去水平扩展意义。设计阶段可用数据分布抽样与压测验证。

### 无索引全表扫描

与关系型数据库类似，过滤字段无索引或索引无法被查询使用会导致 COLLSCAN。上线前对核心查询路径做 **explain** 检查是常规动作。

### 类型不一致

同一字段在不同文档中类型混用（例如部分为字符串、部分为数字），可能导致查询条件匹配不到预期文档或索引选择异常。应用层与校验规则宜统一类型。

-----------

## 与 MySQL、Redis 的选型边界（按条件描述）

- **需要复杂多表连接、强 schema 与成熟 BI 工具链**：关系型（如 MySQL）在生态与 SQL 分析链路上往往更直接。
- **需要灵活嵌套结构、模式演进快、以文档为自然边界读写**：MongoDB 的文档模型可减少联表拆合成本，前提是接受 BSON 与查询语言的学习与运维投入。
- **需要极低延迟键值缓存、计数器、会话、限流**：Redis 的内存模型与数据结构更合适；MongoDB 不承担该类职责时，可与 Redis 组合使用。

同一系统内常见组合是：**MySQL 或 MongoDB 作主存**，**Redis 作缓存或短期状态**；具体以一致性要求、团队技能与托管成本为准。

-----------

## 总结

MongoDB 的工程实践核心是：**先明确部署形态（单机、副本集、分片），再为读写配置合适的关注级别与读路由，片键与索引在上线前评审，多文档事务在时间与模型上设边界，Change Streams 作补充能力而非默认架构中心**。
