`CopyOnWriteArrayList`（写时复制数组列表）是 JUC 里非常典型的“读写分离”容器：读操作直接基于当前数组快照，无需加锁；写操作持锁并复制新数组，再一次性发布。

这类实现适合读远多于写、且读线程不希望被写线程阻塞的场景。本文固定锚定 **OpenJDK 8u**，从字段、写路径、快照迭代和 `addIfAbsent` 几条主线展开。



<!-- more -->



## 简介

`CopyOnWriteArrayList` 实现了 `List`、`RandomAccess`、`Serializable`。它内部不是链表，也不是分段结构，而是一个 `volatile Object[] array` 加一把 `ReentrantLock`。

核心约束很明确：

- 读路径只读快照数组，不修改结构
- 写路径统一走加锁、复制、替换数组
- 迭代器拿到的是创建时刻快照，不反映后续写入

## 源码分节

### 1）核心字段：一把锁 + 一个 volatile 数组

```java
public class CopyOnWriteArrayList<E>
        implements List<E>, RandomAccess, Cloneable, java.io.Serializable {

    final transient ReentrantLock lock = new ReentrantLock();

    private transient volatile Object[] array;

    final Object[] getArray() {
        return array;
    }

    final void setArray(Object[] a) {
        array = a;
    }
}
```

`array` 是 `volatile`，意味着写线程发布新数组后，读线程读取 `getArray()` 可以看到最新引用。

### 2）读路径：直接读快照数组

`get` 不加锁，直接读取当前数组并按下标返回：

```java
public E get(int index) {
    return get(getArray(), index);
}
```

读取代价接近普通数组访问，因此在读密集场景下吞吐稳定。

### 3）写路径：复制后替换

`set`、`add`、`remove` 都遵循相同模板：加锁、基于旧数组复制、修改新数组、`setArray` 原子发布。

```java
public E set(int index, E element) {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        Object[] elements = getArray();
        E oldValue = get(elements, index);
        if (oldValue != element) {
            int len = elements.length;
            Object[] newElements = Arrays.copyOf(elements, len);
            newElements[index] = element;
            setArray(newElements);
        } else {
            setArray(elements);
        }
        return oldValue;
    } finally {
        lock.unlock();
    }
}
```

```java
public boolean add(E e) {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        Object[] elements = getArray();
        int len = elements.length;
        Object[] newElements = Arrays.copyOf(elements, len + 1);
        newElements[len] = e;
        setArray(newElements);
        return true;
    } finally {
        lock.unlock();
    }
}
```

```java
public E remove(int index) {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        Object[] elements = getArray();
        int len = elements.length;
        E oldValue = get(elements, index);
        int numMoved = len - index - 1;
        if (numMoved == 0)
            setArray(Arrays.copyOf(elements, len - 1));
        else {
            Object[] newElements = new Object[len - 1];
            System.arraycopy(elements, 0, newElements, 0, index);
            System.arraycopy(elements, index + 1, newElements, index, numMoved);
            setArray(newElements);
        }
        return oldValue;
    } finally {
        lock.unlock();
    }
}
```

这也是它“写放大”成本的来源：每次结构写入都要分配并复制数组。

### 4）`addIfAbsent`：去重追加

`addIfAbsent` 先基于快照做一次快速判断，不存在再进锁内二次确认，减少并发竞争下的误判插入。

```java
public boolean addIfAbsent(E e) {
    Object[] snapshot = getArray();
    return indexOf(e, snapshot, 0, snapshot.length) >= 0 ? false :
            addIfAbsent(e, snapshot);
}

private boolean addIfAbsent(E e, Object[] snapshot) {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        Object[] current = getArray();
        int len = current.length;
        if (snapshot != current) {
            int common = Math.min(snapshot.length, len);
            for (int i = 0; i < common; i++)
                if (current[i] != snapshot[i] && eq(e, current[i]))
                    return false;
            if (indexOf(e, current, common, len) >= 0)
                return false;
        }
        Object[] newElements = Arrays.copyOf(current, len + 1);
        newElements[len] = e;
        setArray(newElements);
        return true;
    } finally {
        lock.unlock();
    }
}
```

### 5）快照迭代器：读不加锁，但不感知新写入

迭代器创建时抓取数组快照，后续遍历只走这份快照，因此不会抛 `ConcurrentModificationException`，也不支持 `remove/set/add`。

```java
public Iterator<E> iterator() {
    return new COWIterator<>(getArray(), 0);
}

static final class COWIterator<E> implements ListIterator<E> {
    private final Object[] snapshot;
    private int cursor;

    private COWIterator(Object[] elements, int initialCursor) {
        cursor = initialCursor;
        snapshot = elements;
    }
}
```

这种语义非常适合“遍历期间允许并发写入，但遍历结果基于创建时视图”的业务场景。

## 常见问题

### 1. CopyOnWriteArrayList 线程安全吗？

是。写路径在锁内完成，读路径通过 `volatile` 引用获取可见快照，容器本身是并发安全的。

### 2. 为什么它读快写慢？

读操作不复制、不加锁；写操作需要复制整段数组并替换引用，写入频率越高，复制成本越明显。

### 3. 迭代时看不到新数据是 bug 吗？

不是。它是设计语义。迭代器使用固定快照，目标是保证遍历稳定且无并发修改异常。

### 4. 适合当通用 List 吗？

不建议。若写入比例不低，内存分配与数组复制代价会很高。通常用于监听器列表、配置快照、读多写少缓存视图。

-----------

## 总结

`CopyOnWriteArrayList` 的核心是“**写时复制 + 快照读取**”：把读写冲突转移为写入复制成本，换取并发读取的低阻塞和稳定遍历语义。

选型时先看读写比例。读多写少时它非常实用；写多读多或写突发明显时，优先考虑其他并发容器或读写锁方案。
