`ConcurrentHashMap` 在 JDK7 与 JDK8 之间经历了结构级重写。JDK7 以 `Segment` 为并发单元，JDK8 切换到 `Node` 数组为核心的数据布局，并在桶级更新里组合 `CAS`、`synchronized` 与协作扩容。两代实现都追求高并发读写，但路径和瓶颈位置发生了明显变化。

本文固定基于 **OpenJDK 7u / OpenJDK 8u** 对照，聚焦迁移时最影响理解的几条主线：数据结构、写入路径、扩容机制、计数策略与桶树化。

## 演进背景

JDK7 版本的 `ConcurrentHashMap` 在当时有效解决了全表大锁问题，但锁粒度仍受 `Segment` 数量限制。随着多核并发强度增长，热点 `Segment` 竞争会更早出现。

JDK8 调整思路后，目标转向两件事：

- 降低固定分段带来的并发上限约束
- 在保证安全性的前提下，让更多写路径先尝试无锁或低冲突路径

-----------

## JDK7：Segment 分段锁模型

### 结构布局

JDK7 的核心结构是 `Segment<K,V>[] segments`。每个 `Segment` 继承 `ReentrantLock`，内部维护自己的 `HashEntry<K,V>[] table` 与阈值、计数。

调用 `put` 时，先通过哈希高位定位到某个 `Segment`，再在该段上加锁并更新桶内链表。

### 写入主路径

`put` 的并发控制边界在 `Segment`。同一段内的写入互斥，不同段可并行。这个模型实现直接、易推理，但并发度受段数和访问分布影响明显。

### 扩容与计数

- 扩容以段为单位触发和执行
- 计数以段内统计为基础，再做聚合读取

这套方案在低到中等并发下表现稳定，但在热点集中场景里，锁争用仍会聚集在少数段上。

-----------

## JDK8：Node 数组 + 桶级并发模型

### 结构布局

JDK8 移除了 `Segment`，顶层直接维护 `Node<K,V>[] table`。桶中可能是：

- 普通链表结点 `Node`
- 红黑树封装 `TreeBin`
- 迁移标记结点 `ForwardingNode`

这让并发控制更贴近桶位而非固定分段。

### putVal 主路径

`putVal` 大致分为四段：

1. 空表时初始化
2. 目标桶为空时，先用 `CAS` 抢占安装头结点
3. 桶头为 `MOVED` 时进入 `helpTransfer` 协助扩容
4. 桶非空且非迁移状态时，锁定桶头对象，更新链表或树桶

这样的路径让低冲突写入可先走 `CAS` 快路，高冲突时再退化到桶级互斥。

### 扩容协作

JDK8 的扩容由多个线程协同完成。遇到迁移标记桶时，线程可参与 `transfer`。`sizeCtl` 与 `transferIndex` 共同控制扩容阶段与任务分配。

相较 JDK7 的段内扩容，JDK8 在高并发下更容易分摊迁移成本。

### 计数机制

JDK8 引入 `baseCount + CounterCell[]` 的分片计数模式。低竞争先尝试更新 `baseCount`，冲突升高时落到 `CounterCell` 分散写热点，最后通过 `sumCount()` 汇总。

该机制与 `LongAdder` 思路一致，降低高并发计数冲突。

-----------

## 关键差异总览

| 维度 | JDK7 | JDK8 |
|---|---|---|
| 顶层结构 | `Segment[]` + 段内 `HashEntry[]` | `Node[]` 直接承载桶 |
| 写入锁粒度 | 段级（`ReentrantLock`） | 桶级（桶头对象同步）+ 空桶 `CAS` |
| 扩容方式 | 段内扩容 | 全表迁移 + 多线程协作 `helpTransfer` |
| 计数方式 | 段计数聚合 | `baseCount + CounterCell[]` |
| 长链处理 | 以链表为主 | 链表与树桶并存（`TreeBin`） |

-----------

## 对阅读源码的建议

阅读 JDK7 版本时，建议先把 `Segment` 当作一张小型 `HashMap` 来看，优先跟 `put`、`rehash`、`scanAndLockForPut`。

阅读 JDK8 版本时，建议把主线固定在 `putVal`、`addCount`、`helpTransfer`，并结合 `MOVED`、`ForwardingNode`、`TreeBin` 这几个状态点理解分支切换。

在同一篇笔记中并排记录两代实现的“并发控制边界”，会更容易看清迁移动机和代价。

-----------

## 常见问题

### 1. JDK8 完全无锁了吗？

没有。JDK8 只是在部分路径上用 `CAS` 降低锁竞争，桶非空更新与树桶操作仍需要同步控制。

### 2. JDK8 一定比 JDK7 快吗？

并不绝对。收益取决于键分布、写入比例、线程数与硬件拓扑。JDK8 在高并发与热点冲突场景通常更有优势，但仍需基于实际负载评估。

### 3. 为什么还要专门看 JDK7？

线上系统可能仍运行旧版本，且很多并发容器设计讨论会引用 `Segment` 模型。理解 JDK7 有助于读懂历史实现与迁移风险。

-----------

## 总结

`ConcurrentHashMap` 从 JDK7 到 JDK8 的核心演进，是并发控制边界从**固定分段**转向**桶级动态路径**，并配套引入协作扩容与分片计数机制。

掌握两代实现的共同点（哈希分桶、冲突处理）和分歧点（锁粒度、扩容、计数），后续再读 `computeIfAbsent`、树桶细节或性能基准时会更容易落到具体代码路径。
