## HashMap简介

`HashMap` 是 Java 集合框架中最常用的键值对容器，实现了 `Map` 接口。它允许 `null` 键和 `null` 值（`null` 键只能有一个），底层是“数组 + 链表 + 红黑树”的混合结构。

`HashMap` 不是线程安全的，多线程场景下通常使用 `ConcurrentHashMap` 或外部加锁。

## 源码解析

### 底层数据结构

JDK8 中，`HashMap` 的核心字段如下：

```java
transient Node<K,V>[] table;
transient Set<Map.Entry<K,V>> entrySet;
transient int size;
transient int modCount;
int threshold;
final float loadFactor;
```

- `table`：真正存放桶（bucket）的数组。
- `size`：当前键值对数量。
- `modCount`：结构修改计数，用于 Fail-Fast。
- `threshold`：扩容阈值，通常等于 `capacity * loadFactor`。
- `loadFactor`：负载因子，默认 `0.75`。

节点结构：

```java
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next;
}
```

与树化、扩容相关的常量（JDK8，`java.util.HashMap`）：

```java
static final int DEFAULT_INITIAL_CAPACITY = 1 << 4; // 16
static final int MAXIMUM_CAPACITY = 1 << 30;
static final float DEFAULT_LOAD_FACTOR = 0.75f;
static final int TREEIFY_THRESHOLD = 8;
static final int UNTREEIFY_THRESHOLD = 6;
static final int MIN_TREEIFY_CAPACITY = 64;
```

### 构造函数

常用构造逻辑：

1. 无参构造：`loadFactor = 0.75`，此时并不会立刻分配 `table`（懒初始化）。
2. 指定初始容量构造：会把容量调整到“2 的幂”，这样索引计算可以用位运算。
3. 指定容量 + 负载因子构造：同时控制空间和扩容频率。

> 为什么容量要是 2 的幂？
>
> 因为索引计算使用 `(n - 1) & hash`。当 `n` 为 2 的幂时，位运算能更均匀地分布 hash，且性能高于取模运算。

### hash扰动函数

JDK8 的 hash 处理：

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

高位参与低位计算，是为了让 `(n - 1) & hash` 在只看低位时也能利用到高位信息，减少碰撞。

### put流程

`put` 入口直接委托 `putVal`，先算 `hash`：

```java
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}
```

`putVal` 开头完成「懒初始化 + 定位桶头」，空桶则直接 `newNode`：

```java
final V putVal(int hash, K key, V value, boolean onlyIfAbsent,
               boolean evict) {
    Node<K,V>[] tab; Node<K,V> p; int n, i;
    if ((tab = table) == null || (n = tab.length) == 0)
        n = (tab = resize()).length;
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);
    else {
        Node<K,V> e; K k;
        if (p.hash == hash &&
            ((k = p.key) == key || (key != null && key.equals(k))))
            e = p;
        else if (p instanceof TreeNode)
            e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
        else {
            for (int binCount = 0; ; ++binCount) {
                if ((e = p.next) == null) {
                    p.next = newNode(hash, key, value, null);
                    if (binCount >= TREEIFY_THRESHOLD - 1) // -1 for 1st
                        treeifyBin(tab, hash);
                    break;
                }
                if (e.hash == hash &&
                    ((k = e.key) == key || (key != null && key.equals(k))))
                    break;
                p = e;
            }
        }
        if (e != null) {
            V oldValue = e.value;
            if (!onlyIfAbsent || oldValue == null)
                e.value = value;
            afterNodeAccess(e);
            return oldValue;
        }
    }
    ++modCount;
    if (++size > threshold)
        resize();
    afterNodeInsertion(evict);
    return null;
}
```

阅读时可按分支拆：

1. `table` 未分配或长度为 0：先 `resize()`。
2. 桶位 `i = (n - 1) & hash` 上无节点：直接挂新节点。
3. 桶头 `p` 即目标 key：进入覆盖分支。
4. 桶头是 `TreeNode`：走 `putTreeVal`。
5. 否则沿链表向后找，末尾尾插新节点；链表长度达到树化阈值时调用 `treeifyBin`。
6. 若插入的是新键：`modCount`、`size` 自增，必要时 `resize()`。

### 扩容机制（resize）

扩容触发条件：`size > threshold`。

扩容行为：

1. 新容量通常是旧容量的 2 倍。
2. 新阈值同步扩大。
3. 迁移旧桶节点到新桶。

JDK8 优化点在于节点迁移不需要重新计算完整 hash，只需判断“旧容量位”是否为 1：

- 为 0：索引不变。
- 为 1：新索引 = 旧索引 + 旧容量。

这使得扩容迁移成本显著降低。

迁移单桶时典型写法如下（链表情形；同一思想也用于拆分树）：对每个节点看 `(hash & oldCap)`，为 0 的链到 `loHead`，为 1 的链到 `hiHead`，再挂回新表的两格下标 `j` 与 `j + oldCap`。

```java
Node<K,V> loHead = null, loTail = null;
Node<K,V> hiHead = null, hiTail = null;
Node<K,V> next;
do {
    next = e.next;
    if ((e.hash & oldCap) == 0) {
        if (loTail == null)
            loHead = e;
        else
            loTail.next = e;
        loTail = e;
    } else {
        if (hiTail == null)
            hiHead = e;
        else
            hiTail.next = e;
        hiTail = e;
    }
} while ((e = next) != null);
if (loTail != null) {
    loTail.next = null;
    newTab[j] = loHead;
}
if (hiTail != null) {
    hiTail.next = null;
    newTab[j + oldCap] = hiHead;
}
```

### 链表树化与退化

JDK8 引入红黑树是为了解决极端哈希冲突导致的长链表问题。

- `TREEIFY_THRESHOLD = 8`：链表长度达到 8，满足条件时尝试树化。
- `UNTREEIFY_THRESHOLD = 6`：树节点在 resize 或删除后可能退化回链表。
- `MIN_TREEIFY_CAPACITY = 64`：数组容量小于 64 时优先扩容，不急着树化。

`treeifyBin` 在桶上建链再转树；若当前表长不足 `MIN_TREEIFY_CAPACITY`，先扩容而不是树化（JDK8）：

```java
final void treeifyBin(Node<K,V>[] tab, int hash) {
    int n, index; Node<K,V> e;
    if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)
        resize();
    else if ((e = tab[index = (n - 1) & hash]) != null) {
        TreeNode<K,V> hd = null, tl = null;
        do {
            TreeNode<K,V> p = replacementTreeNode(e, null);
            if (tl == null)
                hd = p;
            else {
                p.prev = tl;
                tl.next = p;
            }
            tl = p;
        } while ((e = e.next) != null);
        if ((tab[index] = hd) != null)
            hd.treeify(tab);
    }
}
```

> 为什么不是一到 8 就树化？
>
> 小容量场景碰撞高可能只是暂时现象。优先扩容往往能快速打散冲突，避免过早引入红黑树带来的结构复杂度。

### get流程

`get` 委托 `getNode`：

```java
public V get(Object key) {
    Node<K,V> e;
    return (e = getNode(hash(key), key)) == null ? null : e.value;
}
```

`getNode` 先取桶头，再分链表与树查找（节选结构）：

```java
final Node<K,V> getNode(int hash, Object key) {
    Node<K,V>[] tab; Node<K,V> first, e; int n; K k;
    if ((tab = table) != null && (n = tab.length) > 0 &&
        (first = tab[(n - 1) & hash]) != null) {
        if (first.hash == hash &&
            ((k = first.key) == key || (key != null && key.equals(k))))
            return first;
        if ((e = first.next) != null) {
            if (first instanceof TreeNode)
                return ((TreeNode<K,V>)first).getTreeNode(hash, key);
            do {
                if (e.hash == hash &&
                    ((k = e.key) == key || (key != null && key.equals(k))))
                    return e;
            } while ((e = e.next) != null);
        }
    }
    return null;
}
```

1. 先比较桶头节点（最快路径）。
2. 若桶头不匹配：
   - 链表则顺序遍历。
   - 红黑树则走树查找。

时间复杂度平均是 `O(1)`，冲突严重时可能退化；JDK8 借助树化把最坏复杂度从 `O(n)` 改善到 `O(log n)`（树场景）。

### remove流程

删除本质是找到目标节点并断链（或树删除），然后：

- `size--`
- `modCount++`

`modCount` 变化意味着结构发生修改，迭代器可以据此检测并发修改。

### Fail-Fast机制

`HashMap` 的迭代器会保存创建时的 `expectedModCount`，遍历过程中若发现 `modCount != expectedModCount`，抛出 `ConcurrentModificationException`。

`HashIterator` 在 `nextNode()` 与 `remove()` 里对比 `modCount` 与 `expectedModCount`（JDK8 节选，泛型已补全）：

```java
abstract class HashIterator {
    Node<K,V> next;
    Node<K,V> current;
    int expectedModCount;
    int index;

    HashIterator() {
        expectedModCount = modCount;
        Node<K,V>[] t = table;
        current = next = null;
        index = 0;
        if (t != null && size > 0) {
            do {} while (index < t.length && (next = t[index++]) == null);
        }
    }

    final Node<K,V> nextNode() {
        Node<K,V>[] t;
        Node<K,V> e = next;
        if (modCount != expectedModCount)
            throw new ConcurrentModificationException();
        if (e == null)
            throw new NoSuchElementException();
        if ((next = (current = e).next) == null && (t = table) != null) {
            do {} while (index < t.length && (next = t[index++]) == null);
        }
        return e;
    }

    public final void remove() {
        Node<K,V> p = current;
        if (p == null)
            throw new IllegalStateException();
        if (modCount != expectedModCount)
            throw new ConcurrentModificationException();
        current = null;
        K key = p.key;
        removeNode(hash(key), key, null, false, false);
        expectedModCount = modCount;
    }
}
```

`KeyIterator.next()` 等入口通过 `nextNode()` 推进，从而在遍历时感知并发结构性修改。

## 常见问题

### 1. 为什么默认负载因子是0.75？

这是时间与空间的折中值：

- 太小：扩容频繁，浪费空间。
- 太大：冲突增加，查询变慢。

`0.75` 在工程实践里通常表现较稳。

### 2. JDK7和JDK8在HashMap上的关键差异？

- JDK7：头插法，冲突链表在高并发错误使用下可能形成环。
- JDK8：尾插法 + 红黑树，冲突场景更稳，最坏性能更可控。

### 3. HashMap线程安全吗？

不安全。

并发读写建议：

- 使用 `ConcurrentHashMap`
- 或在外层进行同步控制

## 小结

`HashMap` 的核心可以总结为四点：

1. 数组寻址 + 桶内结构（链表/红黑树）。
2. 懒初始化 + 2 倍扩容。
3. hash扰动提升分布质量。
4. `modCount` 支撑 Fail-Fast。

掌握 `put/get/resize/treeify` 四条主线，基本就能建立对 `HashMap` 源码的完整理解。
