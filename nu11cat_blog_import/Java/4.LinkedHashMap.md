`HashMap` 不保证遍历顺序，业务里若需要「插入顺序」或「访问顺序」可预测，需要额外结构。`LinkedHashMap` 在 `HashMap` 桶数组与哈希逻辑之上，维护一条双向链表，在常数级额外开销下提供有序迭代。

本文围绕节点结构、`newNode` 覆写、`afterNodeAccess` 与 `accessOrder`、以及基于 `removeEldestEntry` 的缓存淘汰用法，说明 `LinkedHashMap` 在 JDK8 中的实现要点。

## LinkedHashMap 简介

`LinkedHashMap` 继承 `HashMap`，实现 `Map` 接口。键值对仍存放在 `HashMap` 的桶里，通过哈希定位；同时每个节点在一条全局双向链表中持有前驱与后继指针，用于按链表顺序遍历 `entrySet()`。

`accessOrder` 为 `false`（默认）时，链表顺序与插入顺序一致。`accessOrder` 为 `true` 时，在调用 `get`、`put` 等访问到已有键后，会把对应节点移到链表尾部，便于实现近似 LRU 的访问语义。

`LinkedHashMap` 同样非线程安全，并发场景需使用 `ConcurrentHashMap` 或外部同步。

## 源码解析

以下方法体摘录自 **OpenJDK 8u** 的 `java.util.LinkedHashMap`。文中为阅读方便补全了泛型写法，逻辑与 JDK8 源码一致。

### 与 HashMap 的关系

`LinkedHashMap` 复用 `HashMap` 的容量、负载因子、`resize`、树化阈值等逻辑。差异集中在：

- 桶内节点类型为 `LinkedHashMap.Entry`（继承 `HashMap.Node`），多出 `before`、`after` 指针。
- 覆写 `newNode`、`replacementNode`、`newTreeNode`、`replacementTreeNode`，在 `HashMap` 插入或替换节点时挂入双向链表。
- 覆写 `HashMap` 的钩子：`afterNodeAccess`、`afterNodeInsertion`、`afterNodeRemoval`。

### 核心字段与 Entry 类型

```java
transient LinkedHashMap.Entry<K,V> head;
transient LinkedHashMap.Entry<K,V> tail;
final boolean accessOrder;

static class Entry<K,V> extends HashMap.Node<K,V> {
    Entry<K,V> before, after;
    Entry(int hash, K key, V value, Node<K,V> next) {
        super(hash, key, value, next);
    }
}
```

- `head` / `tail`：全局双向链表两端；`entrySet` 迭代沿 `head → after → …` 前进。
- `accessOrder`：`false` 为插入顺序，`true` 为访问顺序（构造器三参数可指定）。

### 新节点接入链表：`newNode` 与 `linkNodeLast`

`HashMap.putVal` 在空桶或链表尾部创建节点时调用 `newNode`。`LinkedHashMap` 覆写后先建 `Entry`，再挂到链表尾部：

```java
Node<K,V> newNode(int hash, K key, V value, Node<K,V> e) {
    LinkedHashMap.Entry<K,V> p =
        new LinkedHashMap.Entry<>(hash, key, value, e);
    linkNodeLast(p);
    return p;
}

private void linkNodeLast(LinkedHashMap.Entry<K,V> p) {
    LinkedHashMap.Entry<K,V> last = tail;
    tail = p;
    if (last == null)
        head = p;
    else {
        p.before = last;
        last.after = p;
    }
}
```

替换桶内节点时（例如 `resize` 里 `replacementNode`），用 `transferLinks` 把旧节点的 `before`/`after` 接到新节点上，避免链表断链：

```java
Node<K,V> replacementNode(Node<K,V> p, Node<K,V> next) {
    LinkedHashMap.Entry<K,V> q = (LinkedHashMap.Entry<K,V>) p;
    LinkedHashMap.Entry<K,V> t =
        new LinkedHashMap.Entry<>(q.hash, q.key, q.value, next);
    transferLinks(q, t);
    return t;
}
```

### 删除与 `afterNodeRemoval`

`HashMap.removeNode` 成功后调用 `afterNodeRemoval`，在双向链表中摘掉该节点：

```java
void afterNodeRemoval(Node<K,V> e) { // unlink
    LinkedHashMap.Entry<K,V> p =
        (LinkedHashMap.Entry<K,V>) e, b = p.before, a = p.after;
    p.before = p.after = null;
    if (b == null)
        head = a;
    else
        b.after = a;
    if (a == null)
        tail = b;
    else
        a.before = b;
}
```

### `get` 与 `accessOrder`：`afterNodeAccess`

`LinkedHashMap` 覆写 `get`，在命中且 `accessOrder` 为 `true` 时调整链表顺序：

```java
public V get(Object key) {
    Node<K,V> e;
    if ((e = getNode(hash(key), key)) == null)
        return null;
    if (accessOrder)
        afterNodeAccess(e);
    return e.value;
}
```

`afterNodeAccess` 把被访问节点从原位置摘下并接到 `tail`（已是尾则不动），并执行 `++modCount` 以反映「访问引起的顺序变化」：

```java
void afterNodeAccess(Node<K,V> e) { // move node to last
    LinkedHashMap.Entry<K,V> last;
    if (accessOrder && (last = tail) != e) {
        LinkedHashMap.Entry<K,V> p =
            (LinkedHashMap.Entry<K,V>) e, b = p.before, a = p.after;
        p.after = null;
        if (b == null)
            head = a;
        else
            b.after = a;
        if (a != null)
            a.before = b;
        else
            last = b;
        if (last == null)
            head = p;
        else {
            p.before = last;
            last.after = p;
        }
        tail = p;
        ++modCount;
    }
}
```

`accessOrder == false` 时，`get` 不调用 `afterNodeAccess`，链表只在插入新键时延伸。

### `afterNodeInsertion` 与 `removeEldestEntry`

`HashMap` 在 `putVal` 末尾调用 `afterNodeInsertion(evict)`。`LinkedHashMap` 用它检查是否淘汰最老节点：

```java
void afterNodeInsertion(boolean evict) { // possibly remove eldest
    LinkedHashMap.Entry<K,V> first;
    if (evict && (first = head) != null && removeEldestEntry(first)) {
        K key = first.key;
        removeNode(hash(key), key, null, false, true);
    }
}

protected boolean removeEldestEntry(Map.Entry<K,V> eldest) {
    return false;
}
```

子类覆写 `removeEldestEntry` 返回 `true` 时，会删除当前 `head` 指向的节点。典型用法是与 `accessOrder = true` 组合，按访问时间淘汰最少使用项。

### 按链表遍历：`containsValue`

与 `HashMap` 需要扫描所有桶不同，`LinkedHashMap` 可直接沿链表检查值：

```java
public boolean containsValue(Object value) {
    for (LinkedHashMap.Entry<K,V> e = head; e != null; e = e.after) {
        V v = e.value;
        if (v == value || (value != null && value.equals(v)))
            return true;
    }
    return false;
}
```

### 简易 LRU 缓存示例（可运行思路）

```java
Map<Integer, String> cache = new LinkedHashMap<Integer, String>(16, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<Integer, String> eldest) {
        return size() > 3;
    }
};
cache.put(1, "a");
cache.put(2, "b");
cache.put(3, "c");
cache.get(1);
cache.put(4, "d");
```

`size() > 3` 为真时移除 `head` 所代表的条目。`accessOrder = true` 时，`get` 会调用 `afterNodeAccess`，链表头尾随之变化；可在 `removeEldestEntry` 内打印 `eldest.getKey()` 对照预期。

树化路径下仍通过 `newTreeNode` / `replacementTreeNode` 与 `transferLinks` 维护链表顺序，阅读时可对照 `HashMap` 的 `treeifyBin` 与 `split`。

## 常见问题

### 1. LinkedHashMap 和 HashMap 遍历顺序为何不同？

`HashMap` 按桶下标与桶内链表或树的物理布局遍历，顺序随容量与插入历史变化。`LinkedHashMap` 额外维护插入或访问顺序链表，迭代器沿该链表行走，顺序稳定可预期。

### 2. 为何有了 LinkedHashMap 还要 TreeMap？

`LinkedHashMap` 提供的是插入或访问顺序，键之间没有大小排序。需要按键排序或自定义 `Comparator` 时，应使用 `TreeMap`（基于红黑树）。

### 3. 多线程能否直接使用 LinkedHashMap？

不能。与 `HashMap` 相同，需使用并发容器或外部同步。

-----------

## 总结

`LinkedHashMap` 在 `HashMap` 的哈希表之上增加双向链表，用固定套路的钩子方法在增删查时维护顺序。

需要按插入顺序输出或实现简单 LRU 缓存时，优先在需求明确的前提下使用 `LinkedHashMap`，并正确配置 `accessOrder` 与可选的淘汰策略。
