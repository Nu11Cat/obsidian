`ConcurrentHashMap` 在 JDK7 与 JDK8 之间发生了结构级重写。JDK7 以 `Segment` 分段锁为主，JDK8 改为 `Node[]` 桶数组，并把 `CAS`、桶头同步、协作扩容组合到同一条写入路径里。

这篇把原本拆开的几个主题合并到一篇：**演进背景、`putVal` 主链路、`transfer` 扩容、树化退化、`compute*` 复合原子操作**。源码摘录固定锚定 **OpenJDK 7u / OpenJDK 8u**，泛型按阅读习惯补全。

## 简介

`ConcurrentHashMap` 的目标是让高并发读写在安全前提下尽量减少阻塞。它和 `HashMap` 的共同点是哈希分桶；差异在于并发控制策略与扩容时机处理。

JDK8 类注释明确给出两个关键行为：

- 单 key 更新与读取之间有 happens-before 关系
- `size`、`isEmpty` 这类聚合状态在并发更新下更适合作为估算值

## 源码分节

### 1）JDK7 与 JDK8 的结构演进

JDK7 的核心是 `Segment[]`。每个 `Segment` 继承 `ReentrantLock`，内部再维护一张小哈希表。

```java
// OpenJDK 7u
final Segment<K, V>[] segments;

static final class Segment<K, V> extends ReentrantLock implements Serializable {
    transient volatile HashEntry<K, V>[] table;
    transient int count;
    transient int modCount;
    transient int threshold;
}
```

JDK8 则移除 `Segment`，直接把桶放到 `Node[] table`，并在桶位上分流并发路径：空桶先 `CAS`，非空桶进入桶头同步，迁移中桶走 `MOVED` 分支协助扩容。

这种变化让并发控制边界从“固定分段”转向“按桶状态动态切换”。

### 2）JDK8 的基础节点与哈希扰动

`Node` 中 `val` 与 `next` 是 `volatile`，满足并发读路径可见性；`setValue` 禁止外部修改，避免绕过 map 原子流程。

```java
static class Node<K, V> implements Map.Entry<K, V> {
    final int hash;
    final K key;
    volatile V val;
    volatile Node<K, V> next;

    Node(int hash, K key, V val, Node<K, V> next) {
        this.hash = hash;
        this.key = key;
        this.val = val;
        this.next = next;
    }
}
```

哈希扰动入口是 `spread`：

```java
static final int spread(int h) {
    return (h ^ (h >>> 16)) & HASH_BITS;
}
```

其目的不是改变哈希语义，而是让高位信息参与桶索引，减少局部聚集。

### 3）`putVal`：空桶 CAS，非空桶同步，迁移桶协助

JDK8 `put` 主路径如下：

```java
public V put(K key, V value) {
    return putVal(key, value, false);
}
```

`putVal` 关键分支：

- `table` 未初始化：`initTable`
- 桶为空：`casTabAt` 直接安装新节点
- 桶头 `hash == MOVED`：进入 `helpTransfer`
- 桶非空：`synchronized (f)` 修改链表或 `TreeBin`
- 末尾 `addCount(1L, binCount)` 更新计数并检查扩容

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    if (key == null || value == null) throw new NullPointerException();
    int hash = spread(key.hashCode());
    int binCount = 0;
    for (Node<K, V>[] tab = table;;) {
        Node<K, V> f; int n, i, fh;
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            if (casTabAt(tab, i, null, new Node<>(hash, key, value, null)))
                break;
        } else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        else {
            // 链表桶与树桶更新，省略其余细节
            synchronized (f) { /* ... */ }
            if (binCount != 0) { /* ... */ break; }
        }
    }
    addCount(1L, binCount);
    return null;
}
```

这条路径把写入冲突分级处理：低冲突优先走 `CAS`，冲突后进入桶级互斥。

### 4）扩容迁移：`addCount`、`helpTransfer`、`transfer`

`addCount` 除了更新规模，还负责触发或参与扩容。`sizeCtl` 是扩容状态核心变量，`transferIndex` 用于分配迁移区间。

```java
private final void addCount(long x, int check) {
    // baseCount / CounterCell 更新省略
    if (check >= 0) {
        Node<K, V>[] tab, nt; int n, sc;
        while (/* 达到扩容条件 */) {
            int rs = resizeStamp(n) << RESIZE_STAMP_SHIFT;
            if (sc < 0) {
                if (/* 仍可协助 */)
                    transfer(tab, nt);
            } else if (U.compareAndSwapInt(this, SIZECTL, sc, rs + 2))
                transfer(tab, null);
        }
    }
}
```

遇到 `MOVED` 桶时，线程通过 `helpTransfer` 加入迁移：

```java
final Node<K, V>[] helpTransfer(Node<K, V>[] tab, Node<K, V> f) {
    Node<K, V>[] nextTab; int sc;
    if (tab != null && (f instanceof ForwardingNode) &&
            (nextTab = ((ForwardingNode<K, V>) f).nextTable) != null) {
        int rs = resizeStamp(tab.length) << RESIZE_STAMP_SHIFT;
        while (nextTab == nextTable && table == tab && (sc = sizeCtl) < 0) {
            if (sc == rs + MAX_RESIZERS || sc == rs + 1 || transferIndex <= 0)
                break;
            if (U.compareAndSwapInt(this, SIZECTL, sc, sc + 1)) {
                transfer(tab, nextTab);
                break;
            }
        }
        return nextTab;
    }
    return table;
}
```

`transfer` 内部按步长拆分迁移任务，多个线程可以并行搬桶，避免单线程扩容拖慢写入。

### 5）树化与退化：先看容量，再看链长

相关阈值在 JDK8 是固定常量：

```java
static final int TREEIFY_THRESHOLD = 8;
static final int UNTREEIFY_THRESHOLD = 6;
static final int MIN_TREEIFY_CAPACITY = 64;
```

`treeifyBin` 的第一判断不是马上转树，而是先检查容量：

```java
private final void treeifyBin(Node<K, V>[] tab, int index) {
    Node<K, V> b; int n, sc;
    if (tab != null) {
        if ((n = tab.length) < MIN_TREEIFY_CAPACITY)
            tryPresize(n << 1);
        else if ((b = tabAt(tab, index)) != null && b.hash >= 0) {
            synchronized (b) {
                if (tabAt(tab, index) == b) {
                    // 链表节点转换为 TreeNode，最终包成 TreeBin
                    setTabAt(tab, index, new TreeBin<>(hd));
                }
            }
        }
    }
}
```

含义很直接：小表高冲突先扩容，容量足够后再树化。退化时由 `untreeify` 把树桶转回普通链表。

### 6）`compute*` / `merge`：原子复合操作

这组 API 解决“先读再写”的竞态问题。以 `computeIfAbsent` 为例，空桶路径会先放 `ReservationNode` 占位，再执行映射函数并回填结果。

```java
public V computeIfAbsent(K key, Function<? super K, ? extends V> mappingFunction) {
    if (key == null || mappingFunction == null)
        throw new NullPointerException();
    int h = spread(key.hashCode());
    V val = null;
    for (Node<K, V>[] tab = table;;) {
        Node<K, V> f; int n, i, fh;
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        else if ((f = tabAt(tab, i = (n - 1) & h)) == null) {
            Node<K, V> r = new ReservationNode<>();
            synchronized (r) {
                if (casTabAt(tab, i, null, r)) {
                    Node<K, V> node = null;
                    try {
                        if ((val = mappingFunction.apply(key)) != null)
                            node = new Node<>(h, key, val, null);
                    } finally {
                        setTabAt(tab, i, node);
                    }
                }
            }
            if (val != null) break;
        } else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        else {
            // 非空桶路径：锁桶后查找、插入或直接返回
            synchronized (f) { /* ... */ }
            break;
        }
    }
    if (val != null) addCount(1L, 1);
    return val;
}
```

`merge` 与 `compute` 同样在桶级原子区间内执行函数，能减少业务侧显式加锁需求，但函数本身应保持短小，避免把锁占用时间拉长。

## 常见问题

### 1. JDK8 的 ConcurrentHashMap 是无锁实现吗？

不是。它是“多策略并发控制”：空桶可 `CAS`，非空桶更新仍有同步区间，扩容时还有协作迁移控制。

### 2. 为何禁止 `null` key / value？

并发读取下，`null` 返回值需要明确表示“键不存在”。禁止 `null value` 能减少语义歧义与竞态判断复杂度。

### 3. 树化阈值到了就一定会转红黑树吗？

不一定。若表容量小于 `MIN_TREEIFY_CAPACITY`，实现会优先扩容，而不是立即树化。

### 4. `computeIfAbsent` 里能做重逻辑吗？

不建议。映射函数执行期间可能阻塞同桶更新，复杂逻辑会放大锁竞争并增加尾延迟。

-----------

## 总结

把 JDK7 和 JDK8 放在一张图里看，`ConcurrentHashMap` 的主变化是：并发边界从 `Segment` 固定分段，演进到按桶状态动态切换的写入与迁移路径。

在 JDK8 里，理解 `putVal`、`addCount`、`helpTransfer`、`treeifyBin` 与 `compute*` 这五个入口，基本就抓住了日常使用和问题排查最常遇到的源码主线。
