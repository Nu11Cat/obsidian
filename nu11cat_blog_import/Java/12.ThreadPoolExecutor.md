`ThreadPoolExecutor` 是 JUC 线程池的核心实现。面试高频问题（核心线程数、队列、拒绝策略、关闭流程）都能在 `execute -> addWorker -> runWorker -> getTask` 这条主链路里找到直接证据。

本文锚定 **OpenJDK 8u**，重点拆解任务提交流、工作线程创建、任务消费循环、空闲线程回收和关闭状态迁移。阅读时建议把“线程池状态”和“worker 数量”当成同一个原子控制单元看待。



<!-- more -->



## 简介

`ThreadPoolExecutor` 通过一个整型原子变量 `ctl` 同时编码两类信息：

- 高位：线程池运行状态（`RUNNING`、`SHUTDOWN`、`STOP`、`TIDYING`、`TERMINATED`）
- 低位：当前工作线程数量（`workerCount`）

这套编码使“状态检查 + 线程计数修改”可以在同一 CAS 路径上完成，避免状态字段和计数字段分离带来的并发竞争窗口。

## 源码分节

### 1）`ctl` 位域设计与状态常量

```java
private final AtomicInteger ctl = new AtomicInteger(ctlOf(RUNNING, 0));
private static final int COUNT_BITS = Integer.SIZE - 3;
private static final int CAPACITY   = (1 << COUNT_BITS) - 1;

private static final int RUNNING    = -1 << COUNT_BITS;
private static final int SHUTDOWN   =  0 << COUNT_BITS;
private static final int STOP       =  1 << COUNT_BITS;
private static final int TIDYING    =  2 << COUNT_BITS;
private static final int TERMINATED =  3 << COUNT_BITS;

private static int runStateOf(int c)    { return c & ~CAPACITY; }
private static int workerCountOf(int c) { return c & CAPACITY; }
private static int ctlOf(int rs, int wc){ return rs | wc; }
```

配套辅助方法如 `isRunning`、`runStateAtLeast` 都是围绕上述位布局展开。

### 2）`execute`：提交任务的三段逻辑

`execute` 注释直接给出三步策略：

1. 当前 worker 小于核心线程数，优先建新线程执行 `firstTask`
2. 否则尝试入队；入队后要二次校验线程池状态
3. 队列无法接收时尝试扩容到 `maximumPoolSize`；仍失败则拒绝

```java
public void execute(Runnable command) {
    if (command == null)
        throw new NullPointerException();

    int c = ctl.get();
    if (workerCountOf(c) < corePoolSize) {
        if (addWorker(command, true))
            return;
        c = ctl.get();
    }
    if (isRunning(c) && workQueue.offer(command)) {
        int recheck = ctl.get();
        if (!isRunning(recheck) && remove(command))
            reject(command);
        else if (workerCountOf(recheck) == 0)
            addWorker(null, false);
    } else if (!addWorker(command, false))
        reject(command);
}
```

二次校验 `recheck` 的目的：任务入队后池状态可能已变化，必须兜底处理“队列里有任务但池不再可执行”的竞态。

### 3）`addWorker`：创建工作线程的核心入口

`addWorker(firstTask, core)` 有两层循环：

- 外层按运行状态决定是否允许新增 worker
- 内层按 `corePoolSize` 或 `maximumPoolSize` 边界做 CAS 增量

```java
private boolean addWorker(Runnable firstTask, boolean core) {
    retry:
    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);

        if (rs >= SHUTDOWN &&
            !(rs == SHUTDOWN && firstTask == null && !workQueue.isEmpty()))
            return false;

        for (;;) {
            int wc = workerCountOf(c);
            if (wc >= CAPACITY || wc >= (core ? corePoolSize : maximumPoolSize))
                return false;
            if (compareAndIncrementWorkerCount(c))
                break retry;
            c = ctl.get();
            if (runStateOf(c) != rs)
                continue retry;
        }
    }

    // 省略 Worker 构造、加入 workers 集合、t.start 与失败回滚
}
```

进入启动阶段后，会构造 `Worker`、加 `mainLock` 把它放入 `workers` 集合，再 `t.start()`。若中途失败走 `addWorkerFailed` 回滚并 `decrementWorkerCount()`。

### 4）`Worker`：线程池内部的工作单元

`Worker` 继承 AQS 并实现 `Runnable`。这里的 AQS 锁用于保护任务执行期的中断语义，不用于任务队列同步。

```java
private final class Worker
        extends AbstractQueuedSynchronizer
        implements Runnable {

    final Thread thread;
    Runnable firstTask;
    volatile long completedTasks;

    Worker(Runnable firstTask) {
        setState(-1); // inhibit interrupts until runWorker
        this.firstTask = firstTask;
        this.thread = getThreadFactory().newThread(this);
    }

    public void run() { runWorker(this); }

    protected boolean tryAcquire(int unused) {
        if (compareAndSetState(0, 1)) {
            setExclusiveOwnerThread(Thread.currentThread());
            return true;
        }
        return false;
    }
}
```

`setState(-1)` 的初始化非常关键：线程刚创建时先抑制中断，直到 `runWorker` 开始再放开。

### 5）`runWorker`：任务执行主循环

`runWorker` 是工作线程核心循环：

- 首次优先执行 `firstTask`
- 后续通过 `getTask()` 从队列拉取
- 执行前后回调 `beforeExecute` / `afterExecute`
- 记录完成任务数并在退出时调用 `processWorkerExit`

```java
final void runWorker(Worker w) {
    Thread wt = Thread.currentThread();
    Runnable task = w.firstTask;
    w.firstTask = null;
    w.unlock(); // allow interrupts
    boolean completedAbruptly = true;
    try {
        while (task != null || (task = getTask()) != null) {
            w.lock();
            if ((runStateAtLeast(ctl.get(), STOP) ||
                (Thread.interrupted() && runStateAtLeast(ctl.get(), STOP))) &&
                !wt.isInterrupted())
                wt.interrupt();
            try {
                beforeExecute(wt, task);
                Throwable thrown = null;
                try {
                    task.run();
                } catch (RuntimeException x) {
                    thrown = x; throw x;
                } catch (Error x) {
                    thrown = x; throw x;
                } catch (Throwable x) {
                    thrown = x; throw new Error(x);
                } finally {
                    afterExecute(task, thrown);
                }
            } finally {
                task = null;
                w.completedTasks++;
                w.unlock();
            }
        }
        completedAbruptly = false;
    } finally {
        processWorkerExit(w, completedAbruptly);
    }
}
```

这里能直接回答面试中的两个问题：

- 任务异常会不会吞？不会，异常会进入 `afterExecute`，并影响 `completedAbruptly` 路径。
- 为什么线程池有时会补线程？`processWorkerExit` 在异常退出或 worker 不足时会尝试 `addWorker(null, false)`。

### 6）`getTask`：空闲回收与阻塞获取

`getTask` 决定 worker 何时退出、何时等待任务：

```java
private Runnable getTask() {
    boolean timedOut = false;
    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);

        if (rs >= SHUTDOWN && (rs >= STOP || workQueue.isEmpty())) {
            decrementWorkerCount();
            return null;
        }

        int wc = workerCountOf(c);
        boolean timed = allowCoreThreadTimeOut || wc > corePoolSize;

        if ((wc > maximumPoolSize || (timed && timedOut))
            && (wc > 1 || workQueue.isEmpty())) {
            if (compareAndDecrementWorkerCount(c))
                return null;
            continue;
        }

        try {
            Runnable r = timed
                    ? workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS)
                    : workQueue.take();
            if (r != null)
                return r;
            timedOut = true;
        } catch (InterruptedException retry) {
            timedOut = false;
        }
    }
}
```

核心点：

- 是否允许超时回收由 `allowCoreThreadTimeOut || wc > corePoolSize` 决定
- `poll` 超时后 `timedOut=true`，下一轮可能触发减员
- `take` 会无限阻塞等待任务，适用于核心线程常驻策略

### 7）关闭路径：`shutdown` / `shutdownNow` 与状态迁移

`shutdown` 进入 `SHUTDOWN`，中断空闲线程并允许队列中任务继续消费；`shutdownNow` 进入 `STOP`，尝试中断所有 worker 并返回队列剩余任务。

状态推进由 `tryTerminate` 统一收口：当队列空且 workerCount 为 0 时进入 `TIDYING -> TERMINATED`，并 `termination.signalAll()`。

## 可运行最小示例

```java
import java.util.concurrent.*;

public class TpeDemo {
    public static void main(String[] args) throws InterruptedException {
        ThreadPoolExecutor pool = new ThreadPoolExecutor(
                2, 4,
                30, TimeUnit.SECONDS,
                new LinkedBlockingQueue<>(2),
                new ThreadPoolExecutor.AbortPolicy()
        );

        for (int i = 0; i < 6; i++) {
            int id = i;
            pool.execute(() -> {
                System.out.println(Thread.currentThread().getName() + " run " + id);
                try { Thread.sleep(300); } catch (InterruptedException ignored) {}
            });
        }

        pool.shutdown();
        pool.awaitTermination(5, TimeUnit.SECONDS);
    }
}
```

这段代码可以观察到：核心线程优先创建、队列填充后继续扩容到最大线程、最终执行关闭流程。

## 常见问题

| 问题 | 说明 |
|------|------|
| `execute` 为什么要三段判断 | 要同时覆盖“核心扩容、入队、非核心扩容/拒绝”三种路径，并处理入队后状态变化竞态。 |
| 核心线程一定常驻吗 | 默认常驻；开启 `allowCoreThreadTimeOut` 后核心线程也会在空闲超时后回收。 |
| 拒绝策略何时触发 | 任务无法入队且无法新增 worker（达到上限或状态不允许）时触发 `reject`。 |
| 任务异常会影响线程池吗 | 任务异常会沿 `runWorker` 路径进入退出处理，可能触发替补 worker。 |
| `shutdown` 与 `shutdownNow` 差别 | 前者保留队列任务执行机会；后者尝试立即停止并返回未执行队列任务。 |

-----------

## 总结

`ThreadPoolExecutor` 主链路可以归纳为：`execute` 决策入口，`addWorker` 控制增员，`runWorker` 驱动执行循环，`getTask` 决定阻塞与减员，`tryTerminate` 负责生命周期收口。

理解这条链路后，再看参数调优（核心线程、队列容量、拒绝策略）就能直接映射到源码分支，并把配置选择与实际执行路径对应起来。
