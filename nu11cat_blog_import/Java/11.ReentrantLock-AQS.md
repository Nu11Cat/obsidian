`ReentrantLock` 是 `java.util.concurrent.locks.Lock` 的典型实现，内部通过 `Sync` 继承 `AbstractQueuedSynchronizer`（抽象队列同步器，常缩写为 **AQS**）完成独占加锁、可重入计数、公平策略与条件队列等机制。面试中常把它与内置监视器锁对照讨论；从源码阅读角度，应先把 **AQS 的 `acquire` / `release` 模板** 与 **`ReentrantLock` 对 `state` 的语义** 对齐，再展开公平与非公平差异。

本文摘录与叙述均锚定 **OpenJDK 8u**，泛型按阅读习惯补全。

## 简介

`AbstractQueuedSynchronizer` 提供一个基于 **volatile 状态变量 `state`** 与 **FIFO 双向等待队列** 的同步框架。具体同步器（如 `ReentrantLock` 的内部 `Sync`）只需实现资源获取与释放的判定逻辑：`tryAcquire` / `tryRelease`（独占模式）等；排队、阻塞、唤醒由 AQS 统一处理。

`ReentrantLock` 将 **`state` 解释为当前线程对锁的持有次数**：为 `0` 表示无持有者；大于 `0` 表示已被某线程持有，数值即重入深度。独占所有者由 `AbstractOwnableSynchronizer` 提供的 `exclusiveOwnerThread` 记录。

## 源码分节

### 1）`ReentrantLock` 与内部 `Sync`

对外 API 全部委托给 `final Sync sync`：

```java
public class ReentrantLock implements Lock, java.io.Serializable {

    private final Sync sync;

    public void lock() {
        sync.lock();
    }

    public void unlock() {
        sync.release(1);
    }

    public void lockInterruptibly() throws InterruptedException {
        sync.acquireInterruptibly(1);
    }

    public boolean tryLock(long timeout, TimeUnit unit)
            throws InterruptedException {
        return sync.tryAcquireNanos(1, unit.toNanos(timeout));
    }

    public Condition newCondition() {
        return sync.newCondition();
    }
}
```

`Sync` 抽象类继承 AQS，并实现可重入的获取与释放：

```java
abstract static class Sync extends AbstractQueuedSynchronizer {

    abstract void lock();

    final boolean nonfairTryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) {
            if (compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        } else if (current == getExclusiveOwnerThread()) {
            int nextc = c + acquires;
            if (nextc < 0)
                throw new Error("Maximum lock count exceeded");
            setState(nextc);
            return true;
        }
        return false;
    }

    protected final boolean tryRelease(int releases) {
        int c = getState() - releases;
        if (Thread.currentThread() != getExclusiveOwnerThread())
            throw new IllegalMonitorStateException();
        boolean free = false;
        if (c == 0) {
            free = true;
            setExclusiveOwnerThread(null);
        }
        setState(c);
        return free;
    }

    protected final boolean isHeldExclusively() {
        return getExclusiveOwnerThread() == Thread.currentThread();
    }

    final ConditionObject newCondition() {
        return new ConditionObject();
    }
}
```

要点：

- **首次获取**：`state` 为 `0` 时通过 `compareAndSetState` 竞争，成功则登记独占线程。
- **重入**：若当前线程已是持有者，仅在原 `state` 上累加，不再与队列中其他线程公平竞争同一次 `tryAcquire` 的“空锁槽”语义已由代码路径保证。
- **释放**：`tryRelease` 每次减去 `releases`；仅当计数回到 `0` 才清空独占线程并向 AQS 返回 `free == true`，从而触发后继唤醒。

### 2）非公平与公平：`NonfairSync` 与 `FairSync`

默认构造使用 **非公平** 同步器：

```java
public ReentrantLock() {
    sync = new NonfairSync();
}

public ReentrantLock(boolean fair) {
    sync = fair ? new FairSync() : new NonfairSync();
}
```

非公平锁在 `lock` 时先尝试一次“插队”式 CAS，失败再走标准 `acquire`：

```java
static final class NonfairSync extends Sync {

    final void lock() {
        if (compareAndSetState(0, 1))
            setExclusiveOwnerThread(Thread.currentThread());
        else
            acquire(1);
    }

    protected final boolean tryAcquire(int acquires) {
        return nonfairTryAcquire(acquires);
    }
}
```

公平锁不执行上述快速 CAS，直接进入 `acquire`，且 `tryAcquire` 在空锁时必须满足 **队列中无更早等待者**（`hasQueuedPredecessors` 为假）才允许获取：

```java
static final class FairSync extends Sync {

    final void lock() {
        acquire(1);
    }

    protected final boolean tryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) {
            if (!hasQueuedPredecessors() &&
                    compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        } else if (current == getExclusiveOwnerThread()) {
            int nextc = c + acquires;
            if (nextc < 0)
                throw new Error("Maximum lock count exceeded");
            setState(nextc);
            return true;
        }
        return false;
    }
}
```

`hasQueuedPredecessors` 用于判断是否存在已在队列中排队的线程：

```java
public final boolean hasQueuedPredecessors() {
    Node t = tail;
    Node h = head;
    Node s;
    return h != t &&
            ((s = h.next) == null || s.thread != Thread.currentThread());
}
```

### 3）AQS 独占获取模板：`acquire` 与 `release`

`acquire` 是独占、忽略中断的获取入口：先 `tryAcquire`，失败则入队并在队列中自旋或阻塞，直到成功；若在排队期间收到中断，则在返回前补一次 `selfInterrupt`：

```java
public final void acquire(int arg) {
    if (!tryAcquire(arg) &&
            acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
        selfInterrupt();
}
```

`release` 在 `tryRelease` 成功时唤醒头结点后继：

```java
public final boolean release(int arg) {
    if (tryRelease(arg)) {
        Node h = head;
        if (h != null && h.waitStatus != 0)
            unparkSuccessor(h);
        return true;
    }
    return false;
}
```

### 4）等待队列结点 `Node` 与入队

队列结点保存线程引用、前驱后继指针以及 `waitStatus`。独占模式下 `mode` 参数为 `Node.EXCLUSIVE`（实现里为 `null` 标记）。

```java
static final class Node {
    static final Node SHARED = new Node();
    static final Node EXCLUSIVE = null;

    static final int CANCELLED = 1;
    static final int SIGNAL = -1;
    static final int CONDITION = -2;
    static final int PROPAGATE = -3;

    volatile int waitStatus;
    volatile Node prev;
    volatile Node next;
    volatile Thread thread;
    Node nextWaiter;

    Node(Thread thread, Node mode) {
        this.nextWaiter = mode;
        this.thread = thread;
    }
}
```

`addWaiter` 将当前线程封装为结点并挂到队尾，必要时走完整 `enq`：

```java
private Node addWaiter(Node mode) {
    Node node = new Node(Thread.currentThread(), mode);
    Node pred = tail;
    if (pred != null) {
        node.prev = pred;
        if (compareAndSetTail(pred, node)) {
            pred.next = node;
            return node;
        }
    }
    enq(node);
    return node;
}
```

### 5）`acquireQueued`：自旋、前驱与 `park`

已在队列中的线程在 `acquireQueued` 中循环：若前驱为 `head` 且再次 `tryAcquire` 成功，则把自己设为新的 `head` 并返回；否则根据前驱 `waitStatus` 决定是否应进入 `park`。

```java
final boolean acquireQueued(final Node node, int arg) {
    boolean failed = true;
    try {
        boolean interrupted = false;
        for (;;) {
            final Node p = node.predecessor();
            if (p == head && tryAcquire(arg)) {
                setHead(node);
                p.next = null;
                failed = false;
                return interrupted;
            }
            if (shouldParkAfterFailedAcquire(p, node) &&
                    parkAndCheckInterrupt())
                interrupted = true;
        }
    } finally {
        if (failed)
            cancelAcquire(node);
    }
}
```

`shouldParkAfterFailedAcquire` 负责把前驱标记为需要唤醒后继（`SIGNAL`），或在遇到已取消前驱时跳过并压缩链接；返回 `true` 表示调用方可以安全 `park`。

```java
private static boolean shouldParkAfterFailedAcquire(Node pred, Node node) {
    int ws = pred.waitStatus;
    if (ws == Node.SIGNAL)
        return true;
    if (ws > 0) {
        do {
            node.prev = pred = pred.prev;
        } while (pred.waitStatus > 0);
        pred.next = node;
    } else {
        compareAndSetWaitStatus(pred, ws, Node.SIGNAL);
    }
    return false;
}
```

真正阻塞发生在 `parkAndCheckInterrupt`：

```java
private final boolean parkAndCheckInterrupt() {
    LockSupport.park(this);
    return Thread.interrupted();
}
```

### 6）`unparkSuccessor`：释放后的唤醒

`release` 在 `tryRelease` 成功后，若 `head.waitStatus` 非零，则唤醒第一个需要运行的后继线程。

```java
private void unparkSuccessor(Node node) {
    int ws = node.waitStatus;
    if (ws < 0)
        compareAndSetWaitStatus(node, ws, 0);

    Node s = node.next;
    if (s == null || s.waitStatus > 0) {
        s = null;
        for (Node t = tail; t != null && t != node; t = t.prev)
            if (t.waitStatus <= 0)
                s = t;
    }
    if (s != null)
        LockSupport.unpark(s.thread);
}
```

### 7）可中断与超时：`acquireInterruptibly` 与 `tryAcquireNanos`

`lockInterruptibly` 走 `acquireInterruptibly`：若排队过程中检测到中断，抛出 `InterruptedException` 并取消获取。

```java
public final void acquireInterruptibly(int arg)
        throws InterruptedException {
    if (Thread.interrupted())
        throw new InterruptedException();
    if (!tryAcquire(arg))
        doAcquireInterruptibly(arg);
}
```

`tryLock(timeout, unit)` 走 `tryAcquireNanos`，在 `doAcquireNanos` 中结合 `parkNanos` 与截止时间。

### 8）与 `tryLock()` 相关的公平语义

无参 `tryLock` 直接调用 `nonfairTryAcquire`，即使锁被构造为公平锁也是如此；类注释说明这是刻意的“插队尝试”，与带超时的 `tryLock` 在公平策略上的差异需要按文档区分使用。

## 可运行最小示例

下面示例展示两个线程竞争同一把 `ReentrantLock`；读源码时可对照调用栈：`lock` → `acquire` → `tryAcquire` / `acquireQueued` → `unlock` → `release`。

```java
import java.util.concurrent.locks.ReentrantLock;

public class ReentrantLockDemo {
    private static final ReentrantLock lock = new ReentrantLock();

    public static void main(String[] args) {
        Runnable task = () -> {
            lock.lock();
            try {
                System.out.println(Thread.currentThread().getName() + " in critical");
            } finally {
                lock.unlock();
            }
        };
        new Thread(task, "A").start();
        new Thread(task, "B").start();
    }
}
```

## 常见问题

| 问题 | 说明 |
|------|------|
| `state` 与重入的关系 | `state` 表示持有次数；同一线程再次 `lock` 会使数值递增，`unlock` 对称递减。 |
| 非公平为何默认 | 非公平路径在竞争低时减少不必要的挂起，吞吐通常更好；是否接受插队由业务决定。 |
| `lock` 与 `lockInterruptibly` | `lock` 在 AQS 路径上忽略中断标志，仅在获取成功后通过 `selfInterrupt` 传递中断；`lockInterruptibly` 在排队等待中响应中断并抛异常。 |
| `Condition` 与 `wait` | `newCondition()` 返回 AQS 的 `ConditionObject`，须在持有锁时 `await` / `signal`，语义与监视器条件等待类似，但 API 与实现路径不同。 |
| `IllegalMonitorStateException` | `unlock` 最终调用 `tryRelease`，当前线程不是持有者时会抛出该异常。 |

## 总结

`ReentrantLock` 的可重入语义由 **`Sync` 对 `state` 与 `exclusiveOwnerThread` 的组合更新** 实现；阻塞与唤醒由 **AQS 的 `acquire` / `release` 模板** 与 **`Node` 队列 + `LockSupport` 停调度** 完成。
