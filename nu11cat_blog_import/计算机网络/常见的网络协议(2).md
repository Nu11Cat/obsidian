## **TCP**

是面向连接的(必须一对一)、可靠的(保证一个报文一定到达接收端)、基于字节流(粘/拆包的“罪魁祸首”)的传输层通信协议

**为什么要有TCP**：IP层是不可靠的，不保证网络包的交付、按序交付、也不保证网络包的数据的完整性。那么就需要上层TCP协议来负责。TCP是一个工作在传输层的可靠数据传输的服务。

---

### **TCP头格式**

TCP 报文头用于描述连接状态和数据传输控制信息，是 TCP 实现可靠、有序传输的基础。

一个 TCP 头部主要包含以下关键字段：

- **源端口 / 目的端口**
   用于标识通信双方的应用进程。IP 地址确定主机，端口号确定主机上的具体应用。
- **序列号**
   表示当前报文段中数据的第一个字节在整个 TCP 字节流中的位置。序列号在连接建立时由一方随机生成初始值，之后随着数据发送按字节递增。
   序列号用于保证数据的有序性，并为确认、重传等机制提供依据。
- **确认应答号**
   表示接收方已经成功接收的数据范围，即下一次期望收到的字节序列号。确认应答号采用**累计确认**的方式，隐式地告诉发送方哪些数据已经被正确接收。
- **首部长度**
   表示 TCP 头部的长度，用于区分头部和数据部分。该字段的存在是因为 TCP 头部长度可变。
- **控制位**
   用于描述当前报文的控制含义和连接状态，常见控制位包括：
  - **SYN**：发起连接请求，用于同步双方的初始序列号
  - **ACK**：确认应答号字段有效
  - **FIN**：表示发送方已完成数据发送，请求关闭连接
  - **RST**：表示连接出现异常，强制中断连接
- **窗口大小**
   表示接收方当前还能接收的数据量，用于流量控制，防止发送方发送数据过快导致接收方处理不过来。
- **校验和**
   用于检测 TCP 报文在传输过程中是否发生差错，覆盖 TCP 头部和数据部分。
- **紧急指针**
   当 URG 控制位被置位时有效，用于指示紧急数据的位置，该机制在实际应用中较少使用。
- **选项**
   用于支持扩展功能，例如最大报文段长度（MSS）、窗口扩大因子、时间戳等。

### **三次握手**

三次握手是为了确保双方接收和发送能力都正常，对连接建立达成一致，并同步初始序列号，建立一条可靠的全双工通信通道。

1. 一开始客户端和服务端都处于 CLOSE 状态。先是服务端主动监听某个端口，处于 LISTEN 状态，同时在内部维护两个队列：半连接队列和全连接队列
2. 客户端随机初始化序号（client_isn），将此序号置于 TCP 首部的「序号」字段中，同时把 SYN 标志位置为 1，表示 SYN 报文。然后把SYN报文发送给服务端表示向服务端发起连接，之后客户端处于 SYN-SENT 状态
3. 服务端收到客户端的 SYN 报文后，随机初始化自己的序号（server_isn）并填入 TCP 首部的「序号」字段中，其次把 TCP 首部的「确认应答号」字段填入 client_isn + 1，接着把 SYN 和 ACK 标志位置为 1。然后发送给客户端，之后服务端处于 SYN-RCVD 状态
4. 客户端收到服务端报文后回应最后一个应答报文，应答报文 TCP 首部 ACK 标志位置为 1 ，其次「确认应答号」字段填入 server_isn + 1 ，最后把报文发送给服务端，这次报文可以携带客户到服务端的数据(但实际使用几乎不会)，之后客户端处于 ESTABLISHED 状态
5. 服务端收到客户端的应答报文后，也进入 ESTABLISHED 状态，此时连接就已建立完成

---

**为什么要三次握手，不能两次？四次？**

1. 三次握手是为了确保双方接收和发送能力都正常，建立一条可靠的全双工通信通道。
2. 如果两次，可能有已失效的连接请求报文突然又传到服务端，从而造成错误的连接建立。比如客户端第一次发送的 SYN 因网络延迟滞留很久，客户端以为超时了就重新发了一个 SYN 并完成连接，这个时候滞留的旧 SYN 报文又到达服务器，服务器误以为客户端又要建新连接，于是回 SYN+ACK，并进入半连接状态，造成资源浪费
3. 如果四次，其实是把第二次握手的SYN和ACK分开发送给客户端，但是可以合并成一次

---

**握手丢失了会发生什么**

1. 当第一次握手丢失：当客户端一直收不到第二次握手的回应，会触发超时重传机制，重新发送报文，并且与之前的序列号一样。（每次重传的超时时间是上次的2倍），如果还是没能收到服务端的第二次握手（SYN-ACK 报文），那么客户端就会断开连接。
2. 当第二次握手丢失：因为第二次丢失对客户端的影响一样，所以也会触发上面的操作。同时对于服务器自然也收不到第三次握手，那么服务端会重传 SYN-ACK 报文，也就是第二次握手的超时重传。所以，当第二次握手丢失了，客户端和服务端都会重传。
3. 当第三次握手丢失：服务端会重传 SYN-ACK 报文，当服务器重传到最大重传次数就断开连接。ACK 报文本身是不会有重传的，当 ACK 丢失了，就由对方重传对应的报文，最后服务器关闭连接。

---

**服务端收到客户端发起的SYN后发生了什么**

服务端收到客户端发起的 SYN 请求后，内核会把该连接存储到半连接队列，并向客户端响应 SYN+ACK，接着客户端会返回 ACK，服务端收到第三次握手的 ACK 后，内核会把连接从半连接队列移除，然后创建新的完全的连接，并将其添加到全连接队列，等待进程调用 accept() 函数时从全连接队列中取出该连接进行通信。

---

**为什么初始化的序列号要求不一样，如何随机产生的**

为了防止旧数据混入新连接、避免TCP 重放攻击，并确保每个连接的独立性和安全性。

1. 若新连接使用与已关闭连接相同的序号，网络中滞留的旧数据包可能被误认为属于新连接，造成数据混淆或逻辑错误  
2. 攻击者可以猜测和伪造数据包，进行TCP 重放攻击

TCP 的初始序列号 ISN 现代做法多用内核保密的种子/密钥结合连接信息和时间通过哈希/PRF 生成，使序号既唯一又难以预测，从而提升可靠性与安全性。

#### **半连接队列和全连接队列**

在 TCP 三次握手过程中，Linux 内核会维护两个队列来管理连接请求：

**半连接队列**（也称 SYN Queue）：当服务端收到客户端的 SYN 请求时，此时双方还没有完全建立连接，它会把半连接状态的连接放在半连接队列。

**全连接队列**（也称 Accept Queue）：当服务端收到客户端对 ACK 响应时，意味着三次握手成功完成，服务端会将该连接从半连接队列移动到全连接队列。如果未收到客户端的 ACK 响应，会进行重传，重传的等待时间通常是指数增长的。如果重传次数超过系统规定的最大重传次数，系统将从半连接队列中删除该连接信息。

这两个队列的存在是为了处理并发连接请求，确保服务端能够有效地管理新的连接请求。

半连接队列和全连接队列都存在长度上限。当队列已满时，内核可能直接丢弃新的连接请求，或向客户端返回 RST 报文，具体行为由内核实现和相关配置参数决定。

### **粘包和拆包**

TCP 是**面向字节流**的协议，本身不保留应用层消息的边界。因此，应用层发送的数据在 TCP 传输过程中，可能会被**合并发送**（粘包）或**拆分发送**（拆包）。

1. **粘包**
    指多个应用层消息在发送或接收时，被合并在同一个 TCP 报文段中传输。
    常见原因包括：
   - TCP 为提高传输效率，将多次小数据写操作合并发送
   - Nagle 算法启用时，会延迟发送小数据以合并成更大的报文
2. **拆包**
    指一个应用层消息被拆分成多个 TCP 报文段分别发送。
    常见原因包括：
   - 应用层消息长度超过 MSS（最大报文段长度）
   - 发送缓冲区空间不足
   - TCP 根据拥塞控制、流量控制策略主动拆分发送
3. **解决方案（在应用层处理）**
   - 固定长度消息：每条消息长度固定，接收方按固定长度读取
   - 分隔符协议：在消息末尾添加特殊分隔符，由接收方按分隔符解析
   - 长度字段协议：在消息头中携带消息长度，接收方按长度精确读取（最常见、最通用）

### **传输可靠性**

具体来说，TCP 保证可靠性的方式包括：

1. 数据分段与重组：大数据会被分割成小数据包传输，每个数据包都有序列号，接收方可以按序号重新组合数据，确保数据的顺序性。
2. 确认机制：接收方在成功接收数据后，会通过 ACK 报文以累计确认的方式通知发送方当前已接收的数据范围。
3. 重传机制：如果发送方没有在规定时间内收到确认消息，或者通过 ACK 反馈判断数据可能已丢失，TCP 会重新发送未确认的数据包，确保数据不丢失。
4. 流量控制：通过滑动窗口机制，TCP 控制发送方的数据发送速度，避免接收方处理不过来导致丢包。
5. 拥塞控制：通过算法如慢开始、拥塞避免、快速重传等，TCP 在网络出现拥塞时，动态调整发送速率，避免网络负载过重导致的数据丢失。

----

#### **重传机制**

重传机制是当报文丢失或延迟时，必须能检测并重新发送，保证可靠。

主要重传机制有：

1. 超时重传：发送方在发送每个报文后启动定时器，超时时间内没有收到ACK就重传。简单但是延迟大
2. 快速重传：当接收方收到失序报文时，会重复发送上一个 ACK；发送方若连续收到 3 个相同 ACK，就认为该段丢失，立即重传，无需等超时。速度更快
3. SACK 选择性确认： 普通 ACK 只能告诉“已收到第几字节前的数据”，而 SACK 允许接收方告诉发送方“哪些片段已收到”，这样发送方只需重传丢失的部分，而不是整段。 重传效率高。
4. 重复 SACK： DSACK 是 SACK 的扩展，用来告诉发送方“你重传的数据其实我之前已经收到了”。  这样就能判断是真正的丢包还是 ACK 丢失引起的误判。

#### **流量控制**

**TCP 如何实现流量控制**

流量控制是让发送方根据接收方的处理能力调整发送速度，防止接收方被数据“淹没”的一种机制。

TCP 实现流量控制的主要机制是 滑动窗口。流量控制的目的是防止发送方发送的数据超过接收方的处理能力，避免因接收缓冲区溢出导致的数据丢失。

具体来说，TCP 通过以下方式实现流量控制：

1. 接收窗口：每个 TCP 连接都有一个接收窗口，表示接收方的缓存空间大小。接收方通过 窗口大小 告诉发送方自己当前能接受的最大数据量。窗口的大小会动态变化，反映接收方的处理能力和缓冲区空间。
2. 滑动窗口机制：发送方根据接收方反馈的窗口大小来控制数据的发送量。接收方窗口的大小决定了发送方能在没有确认的情况下发送的数据量。窗口的滑动意味着随着接收方处理完部分数据，接收窗口逐渐“打开”，发送方可以继续发送更多数据。
3. 调整发送速率：如果接收方的窗口大小变小（比如缓冲区已满），发送方会减少数据发送的速率，避免接收方缓冲区溢出。

通过滑动窗口机制，TCP 能够根据接收方的处理能力动态调整发送的数据量，确保数据传输不会因为接收方处理能力不足而丢失或造成过载。

#### **拥塞控制**

**TCP 如何实现拥塞控制**

拥塞控制是 发送端根据网络（路径）当前能承受的容量动态调整可发送的数据量，防止把过多数据同时注入网络导致丢包和性能崩溃  

 TCP 的拥塞控制是通过在发送端维护一个“拥塞窗口”变量，并根据网络反馈（ACK、丢包、超时等）动态调整 cwnd 大小来实现的。  

TCP 的拥塞控制通过动态调整数据发送速率来避免网络拥塞，确保网络的稳定性。TCP 使用了四种主要的拥塞控制算法：

1. 慢启动：当连接开始时，TCP 发送方会将拥塞窗口设置为一个较小的值（通常是 1 或 2 个最大段大小 MSS）。在慢启动阶段，拥塞窗口会在每个 RTT 内呈指数增长，直到达到一个阈值，进入后续阶段。慢启动的目的是迅速探测网络的带宽，避免初期发送过多数据导致拥塞。
2. 拥塞避免：当拥塞窗口大小超过慢启动阈值时，进入拥塞避免阶段。此时，TCP 不再以指数增长的方式增加窗口，而是采用线性增长。在拥塞避免阶段，拥塞窗口以每个 RTT 大约增加 1 个 MSS的速度线性增长，窗口增大速度减缓，以避免过快增加负载导致拥塞。
3. 快速重传：当发送方连续收到 3 个重复的 ACK 时，表示某个数据包丢失了。此时，TCP 立即重传丢失的数据包，而无需等待超时，从而减少了重传的延迟。
4. 快速恢复：在快速重传后，TCP 进入快速恢复阶段。此时，拥塞窗口大小被设置为丢失数据包时窗口大小的一半，继续通过线性增长的方式进行拥塞避免。这使得 TCP 可以迅速恢复并继续传输数据，而不是重新进入慢启动阶段。

### **四次挥手**

四次挥手是 TCP 双方各自独立关闭发送通道的过程，确保数据传输完整、有序地关闭连接。其中 FIN 表示一方不再发送数据，ACK 表示对对方关闭行为的确认。

1. 客户端主动调用关闭连接的函数，向服务器发送FIN报文段，表示没有数据要发送，但是还能接收数据，进入FIN_WAIT_1状态
2. 服务端收到FIN后，TCP 栈会在接收缓冲区中插入一个 EOF，服务端应用程序 `read()` 到它后，发现对方已关闭发送，回复ACK，进入CLOSE_WAIT状态。客户端收到这个 ACK 后进入 FIN_WAIT_2 状态。
3. 服务器处理完剩余数据并调用 `close()`，发送FIN报文，进入 LAST_ACK 状态表明不再发送数据
4. 客户端接收到服务端的 FIN 包，并发送 ACK 确认包给服务端，此时客户端将进入 TIME_WAIT 状态。服务器收到后进入CLOSE 状态
5. 最后客户端经过 2MSL(2*报文最大生存时间) 时间之后，也进入 CLOSE 状态

双方都可以主动断开连接，但是主动关闭连接的，才有 TIME_WAIT 状态

**为什么要四次挥手，FIN和ACK不能一起发？**

TCP是双工的，连接两端的关闭是独立的。四次挥手的本质，是分别关闭两个方向的字节流。服务端在收到 FIN 后可能还有数据要发送，因此不能立即发送 FIN，必须分两次完成 ACK 和 FIN。

 如果服务端在收到客户端的 FIN 时，确实已经没有要发送的数据，就可以立即同时发送 ACK + FIN 报文，即 第二次和第三次合并成一次。  

**握手丢失了会发生什么**

在 TCP 四次挥手过程中，任意一个报文段都可能因网络原因丢失。TCP 通过超时重传和状态机保证连接最终能够被正确关闭。

1. 第一次挥手（FIN）丢失
    客户端发送 FIN 后进入 FIN_WAIT_1 状态，若在超时时间内未收到服务端的 ACK，会触发 FIN 的超时重传。
    若多次重传仍未收到确认，客户端最终放弃连接并进入 CLOSE 状态。
2. 第二次挥手（ACK）丢失
    服务端已收到客户端的 FIN 并进入 CLOSE_WAIT 状态，但客户端未收到 ACK，仍停留在 FIN_WAIT_1。
    客户端会重传 FIN，服务端在收到重复的 FIN 后，会再次发送 ACK，客户端随后进入 FIN_WAIT_2 状态。
3. 第三次挥手（FIN）丢失
    客户端处于 FIN_WAIT_2 状态等待服务端的 FIN。若服务端发送的 FIN 丢失，服务端会对 FIN 进行超时重传。
    客户端在收到重传的 FIN 后，发送 ACK 并进入 TIME_WAIT 状态，挥手流程继续完成。
4. 第四次挥手（ACK）丢失
    客户端发送 ACK 后进入 TIME_WAIT 状态。若该 ACK 丢失，服务端未收到确认，会重传 FIN。
    客户端在 TIME_WAIT 期间收到重传的 FIN 后，会再次发送 ACK，并重新开始等待 2MSL，最终双方正常关闭连接。

**为什么要等2MSL**

1. 防止最后一个 ACK丢失：如果第四次挥手的ACK丢失服务端重传FIN， 客户端等待 2MSL 的时间内仍然能收到这个重传的 FIN，如果不等直接关闭就会导致 服务端就会一直重传 FIN，浪费资源。
2.  等待 2×MSL 可以确保所有旧连接中的延迟或乱序报文都从网络中消失，下次连接的时候不会收到上次连接遗留的数据

#### **Nagle 算法和延迟确认**

TCP 是面向字节流的协议，允许应用层频繁调用 `send()`，即使每次只发送很少的数据。如果每次调用都立即发送一个 TCP 报文段，就会产生大量“小包”，带来较大的协议头开销，降低网络利用率。

**Nagle 算法**的核心思想是：
 **在前一个已发送的数据还未被确认（ACK）之前，不立即发送新的小数据，而是将其暂存在发送缓冲区中。**
 只有在满足以下条件之一时，才会立即发送数据：

- 前一个数据段已经收到 ACK
- 缓冲区中积累的数据大小达到一个 MSS

这样可以有效减少小报文段数量，提高带宽利用率，但代价是**增加了数据发送延迟**，对实时性要求高的场景不友好。

------

TCP 在正常情况下，接收方在收到数据后会发送 ACK 进行确认。但如果每收到一个很小的数据段就立即发送 ACK，也会产生大量小 ACK 报文。

**延迟确认（Delayed ACK）**的策略是：
 **接收方在收到数据后，并不立刻发送 ACK，而是等待一小段时间（通常几十到几百毫秒），以期能够将 ACK 与即将发送的数据一起捎带，或合并多个 ACK。**
 在以下情况下，接收方会提前发送 ACK：

- 有数据需要反向发送（可捎带 ACK）
- 在等待期间又收到了第二个数据段

延迟确认可以减少 ACK 数量、节省带宽，但同样会**增加交互时延**。

------

当 **Nagle 算法和延迟确认同时启用** 时，可能出现“发送方在等 ACK，接收方在等数据”的情况，从而放大延迟，造成明显的性能问题。因此，在低延迟、强交互的应用（如实时通信）中，通常会关闭 Nagle 算法。

------

#### **服务端出现大量 TIME_WAIT 的原因**

首先需要明确，**TIME_WAIT** 是 TCP 四次挥手后，由**主动关闭连接的一方**进入的正常状态，其持续时间通常为 **2MSL**（报文最大生存时间，在 Linux 中一般约为 60 秒）。

因此，当服务端出现大量 `TIME_WAIT`，**本质原因是服务端主动关闭了大量 TCP 连接**。常见场景包括：

1. **服务端主动发起关闭（最常见原因）**

- **HTTP 服务场景**：在 HTTP/1.1 中虽然默认使用长连接，但服务器可能在连接空闲超时、达到请求上限、或返回错误响应（如 4xx、5xx）后，由服务端主动关闭连接。每一次这样的关闭，都会在服务端留下一个 `TIME_WAIT`。
- **短连接模型**：如果服务使用短连接（例如未启用 Keep-Alive，或部分 RPC 场景），每次请求处理完成后由服务端关闭连接，就会频繁产生 `TIME_WAIT`。

2. **反向代理或负载均衡架构**

- 在 Nginx 等反向代理架构中，Nginx 作为客户端与后端应用服务器建立 TCP 连接。如果连接的关闭是由后端应用（如 Tomcat）主动发起的，那么 `TIME_WAIT` 会堆积在后端服务器上，而不是代理节点上。

3. **连接池配置不合理**

- 服务端在访问下游服务（如数据库、Redis）时通常会使用连接池。如果连接池频繁销毁和重建连接（例如空闲超时过短或异常处理不当），这些主动关闭操作同样会在本地产生大量 `TIME_WAIT`。

------

**大量 `TIME_WAIT` 的潜在影响：**

单个 `TIME_WAIT` 状态连接占用的资源并不多，但当数量非常大时，可能导致：

- **端口耗尽**：新建连接需要使用本地临时端口，而大量 `TIME_WAIT` 会占用这些端口，导致无法建立新连接
- **内核资源消耗**：每个 socket 都需要维护状态结构，占用一定内存

------

**常见的解决思路：**

1. **内核参数调整（缓解手段）**

- **端口复用**：开启 `net.ipv4.tcp_tw_reuse`，允许在特定条件下复用 `TIME_WAIT` 状态的端口
- **增大端口范围**：调整 `net.ipv4.ip_local_port_range`，扩大可用的本地端口数量
- **避免使用 tcp_tw_recycle**：该选项在复杂网络环境下存在严重问题，已在新内核中移除

1. **优化应用和架构设计（根本手段）**

- **尽量使用长连接**：正确配置并使用 Keep-Alive，减少频繁的连接建立和关闭
- **调整关闭策略**：在架构允许的情况下，**让客户端主动关闭连接**，将 `TIME_WAIT` 分散到大量客户端，从而避免集中堆积在服务端

## **UDP**

UDP 是一个简单的、无连接的、不提供可靠性保证的传输层协议。它只提供最基本的数据传输功能，不保证数据包的顺序、交付或避免重复，因此开销极小、延迟很低，有明确的消息边界，不会出现 TCP 意义上的粘包问题。非常适合那些对实时性要求高于可靠性的应用。

---

### **QUIC**

QUIC —— 基于 UDP 的可靠传输协议

UDP 本身不提供可靠性保证，但 QUIC 在 UDP 之上、由应用层在用户态实现了一整套可靠传输机制，从而在保持 UDP 低延迟特性的同时，提供了接近 TCP 的可靠性。HTTP/3 正是构建在 QUIC 之上的。

QUIC 的主要特性包括：

- **可靠传输与重传机制**：
   QUIC 为每个数据流维护序号和确认机制，当检测到数据丢失时，只重传丢失的数据，而不是像 TCP 那样可能阻塞整个连接，从而减少延迟。
- **多路复用且无队头阻塞**：
   QUIC 在一个连接中支持多个独立的数据流。某个流发生丢包或重传时，不会阻塞其他流的数据传输，解决了 TCP 在多路复用场景下的队头阻塞问题。
- **连接迁移**：
   QUIC 使用连接 ID 标识连接，而不是仅依赖四元组（源 IP、源端口、目的 IP、目的端口）。当客户端网络发生变化（如从 Wi-Fi 切换到移动网络）时，连接仍可继续使用，无需重新建立连接。
- **内置拥塞控制**：
   QUIC 内置拥塞控制算法，根据丢包、RTT 等网络反馈动态调整发送速率，避免向网络注入过多数据导致拥塞，其基本思想与 TCP 拥塞控制一致，但实现更加灵活。
- **更快的连接建立**：
   QUIC 将传输层与加密层深度结合，默认使用 TLS 1.3，可以在更少的 RTT 内完成连接建立，显著降低建连延迟。

## **TCP vs UDP**

1. 连接方式，TCP传输数据需要先建立连接；UDP不需要连接，即刻传输数据
2. 可靠性：TCP在正常网络条件下尽力保证数据不丢失、不重复、按序到达；UDP不保证到达或顺序
3. 传输方式，TCP是流式传输，没有边界，但保证顺序；UDP是一个包一个包发送，有边界但是可能丢包和乱序
4. 传输效率，TCP有 握手、校验、流控，慢；UDP开销极小、延迟很低，速度快
5. TCP 有拥塞控制和流量控制机制，保证数据传输的可靠性和网络稳定性；UDP 本身不具备拥塞控制机制，发送速率不受网络状态反馈约束。
6. 适用场景：TCP适合数据准确性和完整性要求高的场景，如文件传输、HTTP、邮件等；UDP适合对实时性和速度要求高于可靠性的场景，如实时通信、音视频、游戏、DNS 等
