## **网络模型**

>**分层是为了**标准化交互的过程，具体讲，分层一方面使各层相互独立，不需要考虑其他层的实现，只需要调用下层功能；另一方面提高了灵活性和可替换性，类似高内聚、低耦合的原则；并且使得复杂的计算机网络系统变得易于设计，实现和标准化。

### **`OSI`模型**

OSI七层模型是理论上的网络分层模型，把通信过程分为七层，主要用于教学和标准化

1. 应用层：为用户/应用提供网络服务，如HTTP、FTP、DNS、SMTP。
2. 表示层：负责数据的格式化、加密和解密，如JSON/XML 编码、TLS 加密等。
3. 会话层：负责会话的建立、维护和终止，如会话恢复、检查点等。
4. 传输层：提供端到端的数据传输，如TCP、UDP。
5. 网络层：负责跨网络寻址与路由转发，如IP 协议。
6. 数据链路层：在同一局域网内可靠传输帧，如MAC 地址、以太网协议、ARP。
7. 物理层：负责比特的传输，如网线、光纤、集线器。

---

### **`TCP/IP`四层模型**

TCP/IP 四层模型是实际网络中常用的参考模型

1. 应用层：为用户提供网络服务，如HTTP、FTP、DNS、SMTP、RPC 等，对应七层模型的会话层、表示层、应用层合并。
2. 传输层：提供端到端的数据传输，如TCP、UDP，对应七层模型的传输层。
3. 网络层：负责跨网络寻址与路由转发，如IP 协议，对应七层模型的网络层。
4. 网络接口层：负责在物理网络上传输数据帧，如网卡驱动、以太网，对应七层模型的物理层 + 数据链路层。

---

在 TCP/IP 四层模型 中，各层对应的**数据传输单位**是：

应用层：消息 / 数据
 传输层：报文段（TCP） / 数据报（UDP）
 网络层：数据包
 网络接口层：帧

## **网络问题**

#### **从输入`URL`到界面展示的过程**

1）**URL 解析**
 浏览器解析 URL，得到协议、主机名、端口、路径等信息，为后续生成 HTTP 请求做准备。

2）**DNS 解析：域名 → IP 地址**
 浏览器先查本地缓存 → 操作系统缓存 → 本地 DNS 服务器；
 若本地 DNS 没有命中，会进行递归查询，通过根 DNS → 顶级域 DNS → 权威 DNS，最终返回目标主机的 IP 地址。

3）**建立连接**
 若协议为 HTTP/1.1 或 HTTP/2 over TCP，浏览器通过三次握手建立 TCP 连接；
 （HTTP/3 则基于 QUIC/UDP，不经过 TCP 三次握手）。

4）**生成 HTTP 请求报文**
 包含请求行、请求头、可选请求体等内容。

5）**传输层封装（TCP）**
 HTTP 报文被封装进 TCP 段，添加源端口、目的端口、序号、ACK 等信息；TCP 负责可靠传输、重传、流量控制与拥塞控制。

6）**网络层处理（IP）与路由转发**
 TCP 段被封装成 IP 包，加上源 IP 和目的 IP。路由器依据路由表逐跳转发 IP 包，直至到达目标网络。

7）**数据链路层与 MAC 地址解析**
 在每一跳链路中，设备通过 ARP 获取下一跳或目标主机的 MAC 地址，将 IP 包封装为以太网帧，由交换机依据 MAC 地址表完成二层转发。

8）**目标服务器接收与解封装**
 服务器按层依次移除 MAC 头、IP 头、TCP 头，通过端口号交给对应的应用程序（如 Web 服务器），最终得到 HTTP 请求报文。

9）**服务器生成 HTTP 响应报文**
 包括状态行、响应头、响应体（HTML/CSS/JS/JSON等）。

10）**服务器经原连接返回响应数据**
 通过已有 TCP/QUIC 连接把响应发回客户端。

11）**浏览器解析与渲染页面**
 解析 HTML 构建 DOM，解析 CSS 构建 CSSOM，合成 Render Tree，布局、绘制并最终展示到屏幕。

12）**连接关闭**
 HTTP/1.1 支持长连接 keep-alive，可能复用；闲置后通过 TCP 四次挥手关闭；HTTP/3 使用 QUIC 关闭连接。

#### **网页慢转圈圈排查问题**

最直接的办法就是抓包，排查的思路大概有：

1，先确定是服务端的问题，还是客户端的问题。先确认浏览器是否可以访问其他网站，如果不可以，说明客户端网络自身的问题，然后检查客户端网络配置（连接wifi正不正常，有没有插网线）；如果可以正常其他网页，说明客户端网络是可以正常上网的。

2，如果客户端网络没问题，就抓包确认 DNS 是否解析出了 IP 地址，如果没有解析出来，说明域名写错了，如果解析出了 IP 地址，抓包确认有没有和服务端建立三次握手，如果能成功建立三次握手，并且发出了 HTTP 请求，但是就是没有显示页面，可以查看服务端返回的响应码：

- 如果是404错误码，检查输入的url是否正确；
- 如果是500，说明服务器此时有问题；
- 如果是200，F12看看前端代码有问题导致浏览器没有渲染出页面。

3，如果客户端网络是正常的，但是访问速度很慢，导致很久才显示出来。这时候要看客户端的网口流量是否太大的了，导致tcp发生丢包之类的问题。

总之就是一层一层有没有插网线，网络配置是否正确、DNS有没有解析出 IP地址、TCP有没有三次握手、HTTP返回的响应码是什么。

---

#### **服务端正常启动但是客户端请求不到有哪些原因**

如果客户端请求的接口没有响应，排查的方式：

- 检查接口IP地址是否正确，ping一下接口地址。
- 检查被测接口端口号是否正确，可以在本机Telnet接口的IP和端口号，检查端口号能否连通
- 检查服务器的防火墙是否关闭，如果是以为安全或者权限问题不能关闭，需要找运维进行策略配置，开放对应的IP和端口。
- 检查你的客户端（浏览器、[测试工具 (opens new window)](https://so.csdn.net/so/search?q=测试工具&spm=1001.2101.3001.7020)），是否设置了网络代理，网络代理可以造成请求失败。

如果客户端的请求有响应，但是返回了错误状态码，那么根据错误码做对应的排查：

- 400：客户端请求错误，比如请求参数格式错误
- 401：未授权，比如请求header里，缺乏必要的信息头。（token，auth等）
- 403：禁止，常见原因是因为用户的账号没有对应的URL权限，还有就是项目中所用的中间件，不允许远程连接（Tomcat）
- 404：资源未找到，导致这种情况的原因很多，比如URL地址不正确
- 500：服务器内部错误，出现这种情况，说明服务器内部报错了 ，需要登录服务器，检查错误日志，根具体的提示信息在进行排查
- 502/503/504（错误的网关、服务器无法获得、网关超时）：如果单次调用接口就报该错误，说明后端服务器配置有问题或者服务不可用，挂掉了；如果是并发压测时出现的，说明后端压力太大，出现异常，此问题一般是后端出现了响应时间过长或者是无响应造成的

## **HTTP**

`标题划分思路：请求报文的请求行有请求方法和版本，请求头部有HTTP字段，在响应报文有状态码`

HTTP，即超文本传输协议。

>超文本：文字、图片、视频等的混合体，有超链接功能，最常见的：HTML

用于Web浏览器和服务器之间传输的协议。定义了浏览器如何请求网页，以及服务器如何响应这些请求。

默认端口：80

----

### **请求/响应报文**

**1. 请求报文：**

- 请求行：包含请求方法、请求目标（URL或URI）和HTTP协议版本。

- 请求头部：包含关于请求的附加信息，就是HTTP字段。

- 空行：请求头部和请求体之间用空行分隔。

- 请求体：可选，包含请求的数据，通常用于POST请求等需要传输数据的情况。

**2. 响应报文**：

- 状态行：包含HTTP协议版本、状态码和状态信息。

- 响应头部：包含关于响应的附加信息，就是HTTP字段。

- 空行：响应头部和响应体之间用空行分隔。

- 响应体：包含响应的数据，通常是服务器返回的HTML、JSON等内容。

------

### **请求类型**

| 方法        | 说明                                                         |
| ----------- | ------------------------------------------------------------ |
| **GET**     | 用于获取资源。是幂等的。不应修改服务器资源。                 |
| **POST**    | 用于提交数据、创建资源、触发操作。非幂等。                   |
| **PUT**     | 用于替换（更新）指定资源，通常要求带完整资源内容。是幂等的。 |
| **PATCH**   | 用于对资源进行部分更新。非幂等，但是HTTP规范允许设计成幂等的。 |
| **DELETE**  | 用于删除指定资源。理论上幂等（删一次和删多次结果一致）。     |
| **HEAD**    | 与 GET 相同，但只返回响应头，不返回主体。常用于检查资源是否存在、获取更新时间、判断文件大小。 |
| **OPTIONS** | 查询服务器支持哪些 HTTP 方法。用于浏览器跨域 CORS 中的预检请求。 |
| **TRACE**   | 回显服务器收到的请求，用于测试/诊断。但是已被广泛禁用（存在安全隐患）。 |
| **CONNECT** | 建立隧道（通常是 HTTPS 的 TLS 隧道）。典型场景：浏览器通过代理访问 HTTPS。 |

---

**GET 和 POST 有什么区别**

1. GET是从服务器获取指定的资源，请求的参数位置一般写在URL中，参数只允许ASCII，并且浏览器(不是HTTP)对URL长度有限制。GET是安全且幂等的，是“只读”操作，可被浏览器缓存。
2. POST是根据请求符合(报文body)对指定数据的资源做出处理，POST 请求携带数据的位置一般是写在报文 body 中，body 中的数据可以是任意格式的数据，只要客户端与服务端协商好即可，而且浏览器不会对 body 大小做限制。POST是不安全不幂等的，是“新增/提交数据”操作，会修改服务器资源，浏览器一般不会缓存POST请求。

注：理论上，任何请求都可以带 body 的，只是规范上GET不带body；并且URL 中的查询参数也不是 GET 所独有的，POST 请求的 URL 中也可以有参数的。

### **版本**

**HTTP/0.9**

只有 GET 方法，无头部，响应仅为 纯文本（通常是HTML）。行为上等同于短链接，但是那个时候还没有没有长连接作为对比对象，协议中也没有“连接管理”的概念。

**HTTP/1.0**

1. 引入了头部，也就是请求头/响应头，相应的也引入了状态码
2. 响应不再只支持HTML，可以传输任意类型的数据
3. 新增多种请求方法
4. 基础的缓存机制：服务器在返回响应时，告诉客户端这个资源的“过期时间”或“最后修改时间”，客户端在下次需要这个资源时，可以据此判断是直接使用本地副本，还是需要向服务器重新请求。存在的问题：依赖绝对时间，控制手段单一，不可靠不灵活。
5. 默认短连接， 每次请求都会重新建立并关闭一次 TCP 连接，频繁的三次握手与四次挥手造成明显的带宽和性能浪费。

**HTTP/1.1**

1. 默认长连接， 保持 TCP 连接在多个请求间复用，显著减少连接开销，提高传输效率。 
2. 管道化，同一个 TCP 连接上，客户端不用等上一个响应返回，就连续发送多个请求，服务器按顺序依次返回响应。但是因为队头阻塞严重等问题实际被弃用。
3. 缓存机制：引入了Cache-Control头部和ETag机制，  提供了丰富的缓存指令，支持更精确控制，使用内容哈希标识资源版本而不是时间，缓存校验更加精准高效。
4. 新增并强制使用Host头部：1.0中服务器无法区分同一 IP 下的多个网站，强制Host头的引入实现了虚拟主机支持，使一台服务器能同时服务多个域名  
5. 范围请求机制：允许客户端只获取资源的部分字节（或多段），支持断点续传与按需加载。通过Range等头部配合多种状态码协调实现
6. 分块传输，服务器在无法预先确定响应体长度时仍然可以向客户端持续发送数据，通过将数据拆成多个 chunk 逐步发送。
7. 新增了响应状态码，如206等。

**HTTP/2**

1. 连接方式：多路复用，在同一连接上可以同时传输多个请求和响应，互不干扰。多路复用技术允许多个请求和响应在单个 TCP 连接上并行交错传输，也就是流/帧机制，解决了 HTTP/1.1 应用层的队头阻塞问题，但 HTTP/2 依然受到TCP 层队头阻塞的影响。（长连接强调时间复用，多路复用强调空间/并发复用）
2. 传输格式：二进制分帧，更高效、更易解析  （之前都是ASCII文本格式）
3. 头部压缩： 使用 HPACK 压缩头部，大幅减少冗余 
4. 服务器推送：可以在客户端请求一个资源时，将其他相关资源一并推送，减少客户端的请求次数和延迟
5. 安全性：实际使用中，浏览器强制要求TLS，HTTP/2 实际等同于 HTTPS，总是基于TLS加密

**HTTP/3**

[QUIC协议](https://nu11cat.cn/2025/12/30/%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BD%91%E7%BB%9C/%E5%B8%B8%E8%A7%81%E7%9A%84%E7%BD%91%E7%BB%9C%E5%8D%8F%E8%AE%AE(2)/#QUIC)

1. 使用QUIC协议，在UDP基础上实现了可靠传输、多路复用、加密和连接迁移
2. 彻底解决跨流的队头阻塞问题，把“队头阻塞”从传输层拆开到每个流里，从根上消除了跨流阻塞。
3. 加密机制：TLS内嵌于QUIC，对整个数据包加密，更加安全
4. 更快地连接建立，首次连接1次RTT，复用则0次
5. 丢包处理：QUIC 局部重传，仅影响丢失的数据流 
6. QUIC使用64位标识连接，网络环境改变也能保持连接

### **常见字段**

HTTP字段是出现在HTTP请求或响应头部的一组键: 值结构，用于携带额外的指令或说明，帮助服务器和客户端正确处理HTTP消息。

- HOST：可以将请求发往「同一台」服务器上的不同网站
- Content-Length：表明本次回应的数据长度
- Connection：最常用于客户端要求服务器使用「HTTP 长连接」机制，以便其他请求复用
- Content-Type：用于服务器回应时，告诉客户端，本次数据是什么格式
- Content-Encoding：说明数据的压缩方法。表示服务器返回的数据使用了什么压缩格式

---

### **状态码**

用于表示服务器对客户端请求的处理结果，按首位数字分为五类。

**1xx 信息性——请求已接收，继续处理**

- **100 Continue**：客户端应继续发送请求（常用于分块大请求）。
- **101 Switching Protocols**：切换协议（如升级到 WebSocket）。
- **102 Processing**：服务器正在处理，适用于长耗时请求。

**2xx 成功——请求已成功处理**

- **200 OK**：请求成功，返回正常响应内容。
- **201 Created**：资源创建成功（常见于 POST）。响应头通常包含 `Location` 指向新资源。
- **202 Accepted**：请求已接收但尚未处理（异步任务）。
- **204 No Content**：成功但无返回内容（例如 DELETE）。
- **206 Partial Content**：返回部分内容（支持断点续传）。

**3xx 重定向——需进一步操作**

- **301 Moved Permanently**：永久重定向，资源已迁移到新 URL。
- **302 Found**：临时重定向，资源暂时转移。
- **303 See Other**：建议客户端使用 GET 请求新的 URL。
- **304 Not Modified**：资源未修改，客户端可使用缓存。
- **307 Temporary Redirect**：与 302 类似，但要求保持请求方法不变。
- **308 Permanent Redirect**：与 301 类似，且要求保持请求方法不变。

**4xx 客户端错误——请求存在问题，服务器无法处理**

- **400 Bad Request**：客户端请求格式错误，如参数不合法。
- **401 Unauthorized**：未认证，需要提供有效凭证（如 Token）。
- **403 Forbidden**：已认证，但权限不足或被服务器拒绝访问。
- **404 Not Found**：资源不存在或路径错误。
- **405 Method Not Allowed**：HTTP 方法不被允许（如对资源使用 PUT）。
- **409 Conflict**：资源冲突（如重复提交导致版本冲突）。
- **413 Payload Too Large**：请求体过大。
- **429 Too Many Requests**：请求频率超限（常见于限流）。

**5xx 服务器错误——服务器处理失败**

- **500 Internal Server Error**：服务器内部异常。
- **501 Not Implemented**：服务器不支持请求的方法。
- **502 Bad Gateway**：作为网关/代理的服务器，从上游获得无效响应。
- **503 Service Unavailable**：服务不可用，通常因为过载或维护。
- **504 Gateway Timeout**：作为网关/代理的服务器，等待上游超时。
- **505 HTTP Version Not Supported**：不支持的 HTTP 协议版本。

### **其他**

其他一些经常讨论到的问题：

####  **URI 和 URL 的区别是什么**

URL是URI的一种

URI是统一资源标识符，唯一标识一个资源

URL是统一资源定位符，指出资源访问位置和方式

---

#### **HTTP 是不保存状态的协议, 如何保存用户状态?**  

HTTP是无状态协议，每次请求都是独立的，必须依靠额外机制显式传递状态信息

1. Cookie，最基础的状态存储机制， 由服务器通过响应头 `Set-Cookie` 发送给客户端，浏览器自动在后续请求中携带  
2. Session，基于Cookie的服务器状态保存方案， 服务器在内存/数据库中保存用户状态，客户端仅保存一个 `SessionID`（通常放在 Cookie 中），用来关联状态
3. Token/JWT：无状态认证方案，适合分布式系统， 服务器签发加密 Token（如 JWT），客户端每次请求携带（通常放在 Header 的 `Authorization` 中），服务器验证后识别用户

---

#### **HTTP的缓存技术**

浏览器/中间服务器在本地保存资源副本，减少重复请求、提示加载速度、节省带宽

有两种实现方式：强制缓存和协商缓存

1. 强制缓存即HTTP/1.0的Expires绝对过期时间判断和HTTP/1.1的Cache-Control 相对过期时间和策略判断
2. 协商缓存即HTTP/1.0的 通过资源修改时间判断是否更新  和HTTP/1.1的 内容哈希标识判断是否更新  

协商缓存是强制缓存的“备胎”，只有强制缓存失效时才会触发协商缓存流程。

## **HTTPS**

HTTPS，即超文本传输安全协议。 是在 HTTP 协议基础上通过SSL/TLS协议加密，加密层在HTTP和TCP之间。如果是 HTTP/3，则 HTTP 直接运行在 QUIC 协议之上。

默认端口：443

---

**HTTPS如何解决了HTTP的安全问题?**

首先明白HTTP的安全问题：

1. 明文存储，存在窃听风险
2. 可以截获数据包并修改内容，存在篡改风险
3. 无法确认与你通信的服务器的身份，存在冒充风险

对应的，HTTPS：

1. 混合加密，即对称加密和非对称加密，握手阶段通过非对称加密交换会话密钥，通信过程全部使用对称加密加密明文数据
2. HMAC 基于哈希函数，将数据和通信双方共享的对称密钥一起计算消息认证码。由于攻击者无法获得会话密钥，即使能够截获并修改数据，也无法生成合法的认证结果，从而被接收方识别并丢弃，避免数据被悄悄篡改。

   补充：在现代 TLS 中，通常使用 AEAD 加密算法（如 AES-GCM），将数据加密和完整性校验合并在一次操作中完成。发送方在加密数据的同时生成认证标签，接收方在解密时会同步校验该标签。如果数据在传输过程中被篡改，校验将无法通过，数据会被直接丢弃，从而防止攻击者对通信内容进行隐蔽修改。本质目标与 HMAC 相同，都是在加密通信的基础上保证数据的完整性和不可篡改性。
3. 数字证书的数字签名，通过校验收到的服务器公钥是否具有权威机构CA的证明来判断是不是伪造的

---

### **SSL/TLS 协议**

TLS 是 SSL 的升级版，同一个东西的不同版本，只是SSL叫习惯了所以保留了

**握手过程（以 TLS 1.2 为例）**

1. ClientHello：客户端向服务器发起 ClientHello 请求，发送客户端支持的 TLS 协议版本 + 客户端生成的随机数（Client Random，用来生成会话密钥） + 支持的密码套件列表（密钥交换、加密、摘要算法）
2. ServerHello：服务器做出响应，发送确认使用的 TLS 协议版本（不支持则关闭连接） + 服务器生成的随机数（Server Random，用来生成会话密钥） + 确认使用的密码套件 + 服务器的数字证书
3. 客户端回应：客户端收到响应后，先通过浏览器 / 操作系统内置的 CA 公钥校验证书真实性，然后从数字证书中取出服务器公钥，生成一个随机数（Pre-Master Secret），并使用服务器公钥加密后发送给服务器。
    随后客户端与服务器分别使用 Client Random、Server Random 和 Pre-Master Secret，通过协商好的算法各自生成本次通信的会话密钥。客户端发送加密的 Finished 消息，用于校验握手过程的完整性。
4. 服务端计算会话密钥，发送加密的 Finished 消息。

至此握手阶段结束，进入加密通信阶段。之后的通信仍然是 HTTP 协议，只是使用会话密钥对内容进行加密（对称加密 + 消息认证机制，如 HMAC 或 AEAD）。

补充：在 TLS 1.3 中，不再使用 Pre-Master Secret，而是通过 (EC)DHE 密钥交换直接协商共享密钥，默认具备前向保密，握手过程也更加简化。

**握手为什么要使用随机数，以及为什么要三个随机数**

为什么使用随机数：

1. 保证每次会话密钥的唯一性，避免不同连接生成相同密钥，降低被推测或复用的风险
2. 防止重放攻击，即攻击者重放历史握手数据来诱导使用旧密钥
3. 增加密钥熵，提高不可预测性，增强整体安全性

为什么要三个，而不是只用一个？

1. 确保通信双方都参与密钥生成，避免单方决定，即使某一方的随机数可预测也难以推导最终会话
2. 提高协议安全性与实现健壮性，将随机性来源分散，有助于抵御实现缺陷或弱随机数问题

### **对称加密和非对称加密**

对称加密和非对称加密是现代密码学的两种核心机制，它们的根本区别在于加密和解密所使用的密钥是否相同。

对称加密采用单密钥体系，加密和解密使用同一把密钥。它的优点是计算速度非常快，效率很高，非常适合用来加密大量的实际业务数据。但它的核心缺陷在于密钥分发非常困难，通信双方必须通过一个安全的渠道预先共享同一把密钥，这在实际的互联网环境中很难保证安全，而且密钥的管理和维护成本也会随着通信方数量的增加而急剧上升。

非对称加密则采用了公钥和私钥配对的双密钥体系。用公钥加密的数据只能用对应的私钥解密，反之亦然。它的最大优势是完美解决了密钥分发难题，公钥可以完全公开，发送方直接用接收方的公钥加密数据即可，私钥则始终由接收方自己安全保管。但它的缺点是计算速度非常慢，比对称加密慢好几个数量级，不适合直接加密大量数据。

正因为它们各有优劣，在实际应用中总是相辅相成的。最典型的例子就是HTTPS/TLS协议。在连接建立阶段，使用非对称加密（如RSA）来安全地交换一个临时的会话密钥，这个过程解决了最关键的密钥分发问题。随后，双方就使用这个会话密钥切换到对称加密算法（如AES）来加密所有后续的通信数据，从而兼顾了安全性和高性能。

---

### **数字证书**

**签发和校验过程**

签发：1，CA把持有者公钥、用途、颁发者、有效时间等信息打包成一个包，对这些信息进行HASH计算，2，CA使用私钥将该hash值加密，生成Certificate Signature，也就是数字签名，3，将 Certificate Signature 添加在文件证书上，形成数字证书。

校验：1，客户端使用同样的Hash算法获取证书Hash值H1，2，通常浏览器和操作系统集成了CA公钥信息，浏览器收到证书后可以使用CA公钥解密Certificate Signature，得到哈希值H2，3，对比H1和H2，如果值相同则可信。

## **DNS**

DNS，域名系统，它是互联网中用于将域名转换为对应 IP 地址的分布式数据库系统。

默认端口：53

**DNS基于UDP**，但也可以使用TCP：使用UDP主要是轻量，简单，快，低延迟。也可以用TCP，比如响应过大（UDP 截断）或区域传送时使用 TCP。

---

### **DNS服务器**

DNS服务器是把域名转化成IP的域名解析服务器，还可以反向解析，缓存结果和递归查询。

- 根 DNS 服务器：返回各个顶级域（如 `.com`、`.org`）对应的 TLD 服务器信息
- 顶级域（TLD）服务器：管理具体顶级域
- 权威 DNS 服务器：存储具体域名的 IP 信息，是最终解析来源
- 递归 DNS 服务器：客户端请求先到这里，它负责逐级查询直到拿到 IP

---

### **原理**

1. 查询浏览器缓存和本地缓存，没命中则请求网络配置指定的本地DNS服务器
2. 由本地 DNS 服务器发起迭代查询，依次向根 DNS、TLD DNS 和权威 DNS 查询，最终获得 IP并缓存

---

### **其他**

**DNS 报文格式**

DNS 报文分为查询报文和响应报文，两者的整体结构相同，均由固定长度的头部和若干可变长度的数据区域组成。

- 标识符（ID）。16 比特，由客户端生成，用于唯一标识一次查询。服务器在响应报文中会原样返回该标识符，客户端据此将响应与对应的请求进行匹配。
- 标志（Flags）。16 比特的控制字段，用于描述报文类型和处理方式，其中常见的标志位包括：
  - 查询 / 响应标志位（QR），`0` 表示查询报文，`1` 表示响应报文；
  - 权威应答标志位（AA），表示响应是否来自该域名的权威 DNS 服务器；
  - 期望递归标志位（RD），由客户端设置，表示希望服务器执行递归查询；
  - 递归可用标志位（RA），由服务器设置，表示其支持递归查询。
     其余标志位用于差错处理和协议扩展。
- 记录数量字段。包括问题记录数、回答资源记录数、权威资源记录数和附加资源记录数，用于指示后续各数据区域中资源记录的数量。
- 问题区域（Question）。包含客户端查询的域名以及查询类型和查询类，用于明确“查什么”。
- 回答区域（Answer）。包含对查询结果的资源记录，用于返回域名对应的数据。**一个域名在回答区域中可以包含多条资源记录，因此同一主机名可能对应多个 IP 地址。**
- 权威区域（Authority）。包含相关的权威 DNS 服务器信息，通常用于指示下一步应向哪些权威服务器继续查询。
- 附加区域（Additional）。包含对解析过程有帮助的补充信息，例如权威服务器的 IP 地址，以减少额外的 DNS 查询。

## **WebSocket**

WebSocket 是一种基于 TCP 连接的全双工通信协议，即客户端和服务器可以同时发送和接收数据。用于弥补 HTTP 协议在持久通信能力上的不足。客户端和服务器仅需一次握手，两者之间就直接可以创建持久性的连接，并进行双向数据传输。

WebSocket 在应用层定义了明确的消息帧结构，为每条消息提供边界信息，使接收端能够准确解析完整消息，避免了基于 TCP 字节流通信中需要手动处理消息边界的问题。

使用场景：适用于需要客户端与服务器之间进行高频、低延迟、实时双向通信的场景

---

### **原理/握手过程**

1. 浏览器/客户端向服务器发送一个 HTTP 请求，其中包含特殊头部：请求升级协议为WebSocket、用于服务器生成响应校验的随机字符串、协议版本
2.  服务器接收到请求后，如果支持 WebSocket 回复一个 HTTP 101 状态码，确认升级协议为WebSocket和版本，验证客户端的随机字符串
3. 握手成功，可以双向发送消息

---

### **消息格式**

数据包在WebSocket叫做帧，主要关注以下字段：

**opcode字段**：标志数据帧的类型，比如1是text(String)，2是二进制数据类型([]byte)，8关闭连接

**payload length 字段**：用于描述后续 payload data 的长度

**payload data字段**：真正要传输的数据

---

### **其他**

**WebSocket vs HTTP**

1. WebSocket 是基于持久连接的全双工通信；HTTP 是无状态的请求-响应模型。
2. WebSocket长连接；HTTP/1.0默认短链接，HTTP/1.1默认长连接但仍然是客户端请求服务器响应
3. WebSocket初始为HTTP发起握手，确认后升级为 WebSocket 长连接；HTTP每次请求独立。
4. WebSocket 双方都可主动发送数据，适合高频、实时交互场景如实时聊天、在线游戏、推送通知等；HTTP  客户端必须先发起请求，适合传统网页通信

---

**WebSocket vs 长、短轮询**

1. 短轮询，客户端定期询问服务器是否有新数据，多次短连接，只能客户端到服务器，服务器资源浪费，实时性差。
2. 长轮询，客户端发起请求后，若服务器暂时没有新数据，则保持连接不立即返回；一旦有数据再返回响应，客户端收到响应后立即发起下一次请求。
3. WebSocket，持久双向连接，全双工，一次连接长期复用，实时性好服务器压力小

---

**WebSocket vs SSE**

SSE也是实现服务器向浏览器实时推送消息的技术。单向通信，只能服务器到客户端，是基于HTTP的扩展，天然支持断线重连和文本事件流，但不支持双向通信。SSE的连接方式是客户端发起一次HTTP请求然后服务端不断推送事件流，优点是更加轻量和简单

---

**WebSocket vs Socket**

Socket 是操作系统提供的一组 API，是介于应用层和传输层之间的一层抽象，用来建立和管理 TCP/UDP 通信。

 WebSocket 是一种基于 TCP 的应用层协议，通常通过 Socket API 实现。
